# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`eos` is an agent-based macroeconomic simulation written in plain Java. It models a colony (closed by default, but optionally **opened** to a stream of external money that bankrolls immigration) of laborers, firms (capital, enjoyment, necessity), a bank (one by default, though the design now allows several), and markets that clear by supply/demand each time step. Each settlement also occupies a disc of a given **size** with a fixed number of build **slots** that hold its firms (and, later, housing and other buildings). Laborer households age and die on a real mortality schedule. Running a simulation produces CSV time-series files for analysis (prices, volumes, firm finances, bank rates, laborer state).

The root package is `eos`, under the standard Maven layout at `src/main/java/eos/` (e.g. `src/main/java/eos/agent/firm/CFirm.java` is `eos.agent.firm.CFirm`).

## Build & run

Maven project, Java 21 (`pom.xml`). Tests use JUnit 5 (`mvn test`).

```powershell
mvn clean compile          # compile to target/classes
mvn exec:exec              # run HomogeneousEconomy (forks a JVM with -ea)
mvn exec:exec -Dsim.main=eos.simulation.HeterogeneousEconomy   # run another entry point
mvn package                # build the jar in target/
mvn test                   # run the JUnit 5 suite (each Simulation runs full as a smoke test)
```

`exec:exec` runs in a **forked JVM with assertions enabled** (`-ea`) because the code uses `assert` as real invariant checks; the main class is the `sim.main` property (default `eos.simulation.HomogeneousEconomy`). **Every standard settlement carries a default _export sector_ and a default _ruler_.** The export sector is a single `StrategicFirm` staffed by nobles (`SimulationHarness.createDefaultStrategicSector`); its export earnings accrue to the equity of the bank holding its account, so that bank is not a zero-profit intermediary. The ruler (`SimulationHarness.createDefaultRuler()`) banks in **gold**, spends a little on enjoyment, and **taxes** the colony's banks and nobles into its treasury (so it accumulates rather than spending down); its gold transactions fire the gold bank's FX fee. Both are skipped only by the analytical sweeps (`ScaleSweep`/`LatitudeSweep`, which live apart in the **`eos.simulation.tools`** sub-package) and `SmallOpenEconomy`, left bare so they keep measuring what they were built for. The entry points (README carries the summary table; the mechanics each exercises are detailed in *Architecture* below):

- `HomogeneousEconomy` — homogeneous agents, single copper bank; the default run.
- `HeterogeneousEconomy` — heterogeneous agents, randomized init.
- `TwoBankEconomy` — two copper banks, agents split across them (the multi-bank example); the export firm and its nobles bank at A.
- `SmallOpenEconomy` — opened to external inflow + immigration, growing past its starting 90 laborers (2E + 2N + 1C, two copper banks); bare, with no export sector or ruler.
- `AristocraticEconomy` — the default colony plus **five** `Noble` owner households (the senior noble owns the **silver** bank); the full three-currency hierarchy (copper/silver/gold) with FX friction on dividends. See *Agents → Nobles*.
- `StrategicEconomy` — nobles bank in silver while the export firm banks in copper, so their export _wages_ cross currencies (FX friction on primary income).
- `HanseaticEconomy` — two colonies near Lübeck in one `GameSession`: independent economies but a shared name pool (surnames unique across both); 20 laborers, 1E + 2N, 5 nobles. Both colonies (like every ruler-bearing colony) found and replace their labor force from a peasant pool, so each **collapses** once the reserve drains — both within months at this minimum scale (the two differ by their independent RNG streams). `run()` returns the first.
- `PeasantEconomy` — the default colony with a **smaller `promotionRatio`** (so a larger standing reserve), charting relief spending (`Peasants.csv`) before the colony collapses.
- `MeritocraticEconomy` — the three-currency colony (nobles in silver) foregrounding **promotion**: the ruler elevates the ablest peasants into laborer households (a **high `promotionRatio`**, so most of the pool becomes laborers). Collapses like the rest once the reserve drains.

**The peasant pool is the universal labor model.** Every ruler-bearing colony seeds a `PeasantPool` of `cfg.peasantPoolSize()` peasants (default 900; and, so it can grow, a default `BuilderFirm` staffed from that pool); on day 0 the ruler promotes the ablest `round(promotionRatio · poolSize)` of them into laborer households (each endowed with the **sum of its 12 skill levels** in copper, funded from the ruler's treasury, and carrying its larder ration out of the pool), the rest staying as the standing reserve. Dead laborers are replaced the same way from the reserve. The reserve peasants live on a **low relief ration** the ruler funds, bought price-sensitively (so the extra mouths don't starve the workforce out of the food market — see *Agents → Peasant pool*); the reserve persists a couple of years as a genuine replacement pool, then — with no inflow yet — drains and the colony **spirals to collapse** (the default scale lasts ~5–7 years; tiny colonies collapse within months). Only the bare analytical sims — the sweeps and `SmallOpenEconomy`, which have no ruler/pool — keep direct founding + same-dynasty succession and stay stable. People generated into the pool also roll a random **gender** (see *Agents → Laborers/skills*). See *Agents → Peasant pool*.
- `ScaleSweep` / `LatitudeSweep` — analytical sweeps in `eos.simulation.tools` (see below).

Each scenario is a `static run()` that builds and runs the colony and returns a `SimulationHarness`, plus a `main()` that calls it — **except the sweeps `ScaleSweep` and `LatitudeSweep`**, whose `main()` runs a multi-colony sweep printing a table to stdout (no CSVs): `ScaleSweep.sweep()` the minimum stable scale, `LatitudeSweep.sweep()` the maximum survivable latitude (bisecting the survives/starves boundary, or reporting that the limit lies beyond the swept range when no swept latitude starves) — while their `run()` is just the convention hook returning one default harness. A scale `k` there means `k` enjoyment + `k` necessity firms (+1 capital firm) and `k·45` laborers; "stable" generalizes the test invariants — population sustained, consumer prices finite/positive **and below 1000× their initial ceiling** (a runaway stays a finite double, so mere finiteness isn't enough), bank pools/rates finite, no thrown `-ea` invariant. Each `run()` creates a `GameSession` (which holds the seed and its own `Rng`) and from it a `Settlement` (`session.newSettlement(cfg.settlementName(), cfg.startDate(), cfg.meanInitAgeYears(), …)` — colonies are **named**, exposed via `Settlement.getName()`), then hands that colony to the harness. **`SimulationHarness` consolidates the shared construction** (markets, banks, firm/laborer loops, printers, run/cleanup) into a single `Settlement` instance; a `Simulation` supplies only what differs — which bank each agent uses (`IntFunction<Bank>`), and how each agent's initial state is drawn (`IntToDoubleFunction`). The call order in the harness mirrors the original sims and matters (deferred settlement + reproducible RNG consumption). A run is configured through `SimulationConfig.DEFAULT`.

Tests (`src/test/java`, JUnit 5) treat each simulation as a smoke test: `Simulation*Test` calls `Simulation*.run()` for a full ~10k-step run and asserts invariants on the returned harness. Because every ruler-bearing colony now founds and replaces its labor force from a finite peasant pool, those runs **collapse** (`SimulationAssertions.assertCollapsed`: the colony ran to completion without tripping an `-ea` invariant and ended with its labor force gone); `ClosedColonySmokeTest` checks each closed colony reaches that collapse with the expected bank count (the banks persist past the colony's death). The bare analytical colonies (`ScaleSweep`, `SmallOpenEconomy`) keep direct founding and stay stable. `LaborTrainsSkillsTest` (which needs living laborers to inspect their trained `PLANTS` skill) builds its own `HomogeneousEconomy`-style **pool** colony over a **short, pre-collapse horizon** and made deliberately **necessity-heavy** (few enjoyment firms, many necessity firms) — because workers now eat only the `RationSize.FINE` ration, necessity is otherwise too small a sector to lift `PLANTS` above the noise of an untrained skill; concentrating labor on farming makes the training signal clear (the production firm mix is unchanged — the test only configures its own colony to exercise the mechanism); it also **disables weddings** (`setWeddingConfig(... capacity 0)`), which are orthogonal to skill training and otherwise add noise (female ex-spouses becoming heads via widowhood with short training histories). `WeddingMarketTest` covers the wedding feature directly: the bride-price curve is super-linear in skill, and a short pool-colony run weds opposite-gender spouses out of the pool (the ruler, wedding first and free, ends married). `ScaleSweepTest` runs only the default scale to completion (the full sweep is too heavy for CI) and checks the minimum-selection logic as a pure function over synthetic results. `SmallOpenEconomyTest` checks its own invariants directly (population *grows* past its starting 90, both banks active, prices finite) rather than via `assertHealthy`, which assumes the large closed-colony population. `SettlementSolarTest` checks the daily solar times directly (no economy): London gets ~16.5 h of daylight at the June solstice vs ~7.8 h in December, dawn < sunrise < sunset < dusk on a normal day, and astronomical dawn/dusk are `null` at London midsummer. (`LatitudeSweep` is a `main`-only sweep with no test, per design.) `SlotTableTest` checks the precalculated slot table matches the spreadsheet (total/road/wall/effective/special-site spot-checks across sizes 0–95, plus the derived columns self-consistent) and that a colony founds at the floor size (3 → 15 slots) and grows one size at a time to fit the firms placed on its slots (a 16th firm forces size 4). `AristocraticEconomyTest` checks its invariants directly rather than via `assertHealthy`, since its banks are deliberately profitable (non-zero equity) and the colony ultimately **collapses** (`assertCollapsed`); the rentier aristocracy and banks outlive the laborer collapse, so it still asserts the firms *and* the silver bank paid dividends (`bank.getDistributedProfit() > 0`) and the nobles keep positive, finite wealth through the collapse. (It no longer asserts the nobles *grew past* their seed fortunes, nor that a noble succession fired — the colony now collapses within a few years, so the declining economy's dividends needn't outrun consumption and the working-age nobles needn't live long enough to die and be succeeded.) It additionally carries **ruler / three-currency** coverage: three banks (copper, silver, gold), the gold bank profiting from the ruler's gold→copper purchases, and exactly one gold-banking ruler whose wealth stays finite — it both taxes and bankrolls the pool's relief (overdrawing into a loan), so over this colony's life it may run a bounded deficit rather than accumulating; detailed taxation-accumulation coverage lives in `RulerTaxationTest` (a pool-less colony, so no relief drain). Settlement state (including the agent-ID counter and the `Rng`, now owned by the `GameSession`) is per-instance, but the `SimLog` logging handler is still a process-global static, so Surefire keeps `reuseForks=false` — a fresh JVM per test class isolates that static state (and keeps `-ea` on).

Output CSVs are written to an `output/` directory created relative to the **current working directory** at runtime (i.e. the project root when launched via Maven; see `io/printer/CSVPrintWriter.java`). Runs are reproducible: a `GameSession` derives each colony's `Rng` deterministically from its seed, so the same seed yields identical output.

## Architecture

### The step loop is the spine

`Settlement` is an **instance** (not a static singleton): a `GameSession` owns the seed and creates colonies from it (`settlement/GameSession.java`), so several independent colonies could coexist in one JVM (tests still use one). Each colony gets its **own economic `Rng`** (seeded from the session seed and the colony's index, so colonies don't interleave their random streams), but **shares** the session's `NameRegistry` and `Demography` — the shared name pool is what keeps dynasty surnames unique across every colony in the session. Colony 0 uses the bare seed, so a lone colony's economic stream is unchanged by the per-colony scheme. Agents, banks and markets each hold a reference to the `Settlement` they belong to (injected through their constructors); printers receive it in `print(Settlement)`. Agent IDs are per-colony too — the `nextAvailableID` counter lives on `Settlement` (`agent.getID()` is drawn from `colony.nextAgentID()`), so two colonies have fully independent ID/account spaces. Collaborators register with the colony via `colony.addBank(...)` / `addMarket(...)` / `addAgent(...)` / `addPrinter(...)`. `colony.run(steps)` calls `colony.newDay()` repeatedly. `newDay()` first **recomputes the day's solar times** for the colony's location and current date (dawn/sunrise/sunset/dusk and the precalculated daylight duration — see *Daylight and the solar package* below), then defines the fixed phase order every tick (`settlement/Settlement.java`):

1. `agent.act()` for every alive agent — agents read last step's prices/income and **post buy/sell offers** to markets (they do not transact yet); a death is flagged via `die()`.
2. `bank.act()` for every registered bank — recomputes total loan/deposit, sets loan & deposit interest rates, pays/charges interest.
3. Remove the dead and **spawn replacements** — one successor household per dead agent, via the registered **replacement policies** (`Settlement.addReplacementPolicy`, a list tried in order until one returns non-null). The colony registers a **built-in household-succession policy** in its constructor that asks each dead `Household` for its own heir (`Household.successor(colony)`): a `Noble` returns a same-dynasty heir inheriting its firms and banks (reusing the noble's own `NobleConfig`), a `Ruler` an heir that also takes over the colony's ruler reference — so noble and ruler succession needs **no per-simulation wiring** (a new self-replacing population type is likewise succeeded automatically). A `Laborer`'s `successor` returns null (its heir needs colony-seeding constants the colony doesn't hold), so the harness registers the laborer replacement policy: for a ruler-bearing colony it **promotes the ablest peasant** out of the pool into a new laborer household (see *Peasant pool*), and for a bare pool-less colony it makes a same-dynasty heir. A dead agent that is a `Household` and gets **no** successor is an extinct dynasty, so its surname is **recycled** back into the session name pool (`names.releaseDynastyName(...)`). This **does** fire in the ruler colonies: once the pool reserve drains, promotion returns no replacement, dead laborers go unreplaced, their surnames recycle, and the colony collapses — the bare colonies' same-dynasty replacement always succeeds, so there it never fires. Then run the registered **step actions** (`Settlement.addStepAction`, e.g. injecting external money into a bank's equity) and admit any **immigrants** (`Settlement.setImmigrationPolicy`, new households funded out of that equity). All default to no-ops.
4. `market.clear()` for every market — **this is where transactions actually settle**.
5. `printer.print()` for every printer.
6. `updateInflation()` (CPI = mean of consumer-good market prices over `INFLATION_TIME_WIN`).

The key mental model: **`act()` posts intentions against last step's state; `clear()` executes them.** Order of construction in a simulation matters because of this deferred settlement (the labor market is cleared once manually before `run()` so firms have workers in step 0).

### Settlement size and build slots

A `Settlement` is a **disc of radius `size`** with a fixed number of **build slots** — the spatial capacity that holds firms (and, later, housing and other buildings). The geometry is a precalculated lookup, `SlotTable` (`settlement/SlotTable.java`), loaded once from `/slots.json` (the exported design spreadsheet, sizes 0–95) and held on the `GameSession`: like `NameRegistry`/`Demography` it is **shared** by every colony in the session — it is pure geometry, independent of seed and location — and threaded into the `Settlement` constructor alongside them (the sim never passes it; `GameSession.newSettlement` supplies it). Each row is a `SlotInfo` record (`settlement/SlotInfo.java`): `total` build slots = `floor(π·size²)` (size 0 hand-set to 1, the lone slot the export firm takes); of those `road = floor((size/100)·total)` are consumed by roads — a **linearly-growing congestion share** — and `wall` (≈ circumference; the exact column is transcribed from the sheet, the one column no clean formula reproduces) by walls; what's left is `effective = total − road − wall`, the usable capacity. `maxSpecialSites` (1→6, unlocked at sizes `{0,4,10,19,31,57}`) counts **special sites** — out-of-band slots for enormous buildings/projects *not* subject to the normal limit (tracked as a number only; nothing occupies them yet). `SlotInfo.wallBuildTimePercent()` (= `Wall%²`) is a wall build-speed multiplier the `BuilderFirm` uses (inverted to a work multiplier) when costing a ring's wall work — walls go up fast while the colony is small, slower as its circumference grows. Because roads eat an ever-larger share, **effective capacity peaks at size 66 (4244 slots) and then declines**; the table caps at 95 (a colony would take far more than a normal run to approach it).

Each `Settlement` carries a **mutable** `size` and a `List<Slot>` — one `Slot` (`settlement/Slot.java`: a nullable `Agent` occupant with `occupy`/`vacate`/`isVacant`, typed `Agent` because only firms occupy slots today, to widen to an occupant interface when housing arrives) per effective slot. A colony is **founded at `SlotTable.MIN_SIZE` (3 → 15 effective slots)** and **grows to fit its occupants**, but *how* it grows depends on its lifecycle. `claimSlot(occupant)` seats the occupant on the first vacant slot; when none is free it diverges: **at founding** (before `start()`) it lays out the initial footprint by calling `setSize(size+1)` (which extends the slot list — growth-only; shrinking below the occupied count throws) one step at a time until the occupant fits — a one-time genesis sizing — whereas **once the colony is live** (after `start()`) it does *not* grow itself: the only way a running colony gets bigger is through its `BuilderFirm` (see *Agents* → *Builder*). `SimulationHarness` claims a slot for **every firm** — capital, enjoyment, necessity and the `StrategicFirm` — so the firm-heavy default colonies (10E+12N+1C+1 strategic+1 builder = 25 firms, or `ScaleSweep`'s 2k+1) found themselves at size 4 (29 slots), while the small-open (6-firm) colony stays at the floor 3. This realizes "smallest size that fits the firms, floored at 3" with no per-sim configuration: `size` is **not** a `SimulationConfig` field — it is always founded at 3 and emerges from what's placed. Slot placement moves no money and consumes no randomness, so it does not perturb the economic stream.

A `Settlement` grows during the run only if it has a **builder**, a `BuilderFirm` registered via `setBuilder` (a colony without one simply never grows once live — `claimSlot` throws if a live builderless colony runs out of room; this is the case for the bare analytical sims, which have no pool and so no builder). Every **pool-bearing** colony now gets a default builder (created in `createDefaultPeasantPool`), so the standard sims all carry one — though it stays effectively **dormant** in practice: they found at a size that fits all their firms and add none during the run, so no growth ring is ever queued, and the colony collapses before it would expand. When a live colony's `claimSlot` finds no free slot, it **queues the next ring** for the builder (`requestGrowth`) and holds the occupant **pending** (returning `null` — slots are pure bookkeeping, so the firm is economically active immediately regardless), seating it once the ring is built. A ring (size `n → n+1`) splits into `BuildProject`s (`settlement/BuildProject.java`) by funder: a `LAND` task per pending occupant (the slot it will stand on, funded by **that firm**) plus the ring's `ROAD` and `WALL` public works (funded by the **ruler**). The builder bills each sponsor at cost as it works; `completeFinishedRings()` calls `setSize` and seats the pending firms once a ring's land, road and wall tasks are all done. The **scaffold cap** (`BuilderConfig.scaffoldCap`, the limit on build-units per step) and the wall build-speed factor bound how fast this goes. Special sites (`maxSpecialSites`) remain a count only — nothing occupies them yet.

### Markets clear by finding a price

Each `Market` (subclass of `eos/market/Market.java`) collects offers during the act phase and settles in `clear()`:
- `ConsumerGoodMarket` binary-searches for a price where demand ≈ supply (bounded to ±`zeta` of last price), then fills all buyers/sellers pro-rata and moves money via `Bank`. Buyers express willingness via a `Demand` strategy object (`market/Demand.java`), not a fixed quantity.
- `LaborMarket` shuffles employers/employees and allocates workers proportionally to each firm's wage budget, paying each worker an equal per-head wage (`wageBudget/headcount`) via `withdraw`+`credit`. The **labor each worker delivers is skill-scaled**, not 1: the firm's `Labor` good is increased by the worker's `productivity` (a skill-10 worker contributes 1, the legacy homogeneous case), so two workers at the same firm earn the same wage but supply different amounts of labor. That `productivity` is **firm-specific and computed when the market clears**: a worker delivers `productivityOf(relevantLevel)`, where `relevantLevel` is its (rounded mean) level in *that employer's* skills. Each firm declares the skills its work reads and trains (`Firm.laborSkills()` — necessity→`PLANTS` (subsistence agriculture), enjoyment→`{ARTISTIC, CRAFTING, SOCIAL}`, capital→`CRAFTING`, builder→`CONSTRUCTION`, the strategic export firm→`INTELLECTUAL`; a firm with none falls back to the worker's overall level). So **necessity output tracks a laborer's PLANTS skill, the export firm a noble's INTELLECTUAL, and so on** — each worker is as productive as it is skilled *at that firm's particular work*. Performing the labor also **trains** the worker: each step employed it gains `XP_PER_LABOR` (1, passion-scaled) experience in each of its employer's skills (which then decays daily above level 10 — see `skill/`), so workers **specialize** (better at the work they do often) and each worked skill — hence its output — settles at a train-vs-decay equilibrium. Productivity is read **live** from current skills, computed per worker per firm in `clear()` (the `relevantLevel` helper). A worker's output is then **scaled by the length of the day**: a laborer delivers 100% of its skill-scaled output at `FULL_OUTPUT_DAYLIGHT_HOURS` (8) hours of daylight, proportionally more on longer (summer) days and less on shorter (winter) ones — `productivity * (1 + sensitivity·(daylight/8 − 1))`. The `sensitivity` is itself penalized at high latitudes (`daylightSensitivity()`): full up to 45°, fading linearly to 0 at the poles, so the extreme polar daylight swings don't blow up the economy. **Only laborers** get this daylight scaling (it lives in `addEmployee(Laborer)`); nobles supplying labor to the strategic sector use the `addEmployee(int, Bank, double, SkillTracker)` primitive with a multiplier of 1 and are unaffected. The coupling is calibration-sensitive: deep-winter labor *scarcity* (factor ≪ 1) is the binding stability concern, so the 8h reference — which keeps winter output near 100% — matters. The rest-day calendar's wage-smoothing (see *The liturgical calendar*) absorbs the deep-winter shortfall, so the colony survives the **whole swept range** — `LatitudeSweep` finds no starvation up to 70°; the reference shapes winter output but no longer sets a latitude ceiling within the tested range.
- `CapitalMarket` matches firms buying machines against `CFirm` sellers.
- `WeddingMarket` (`market/WeddingMarket.java`) clears **on the weekly day of rest only** (`DayType.WEEKEND`): each step an unmarried household (one with just its head — see *Marriage* below) posts itself as a seeker, and on a weekend the market marries up to `WeddingConfig.capacity()` (default **4**) of them in **priority order** — the **ruler first, then nobles, then laborers**, and within a rank the richer household first (`Household.weddingPriority()`, ruler 2 / noble 1 / laborer 0). Each seeker takes the **best available peasant of the opposite gender** to its head out of the `PeasantPool` (highest `overallLevel`, the younger breaking a tie) as a spouse, paying a **non-linear bride-price** (`WeddingConfig.costFor` = `baseCost·(1+skill)^skillExponent`, default `2·(1+skill)²` copper) that flows to the **ruler** (the peasants' patron); a household that cannot afford its best match waits for another weekend (the ruler weds **free**, from its own wards). The match is a deterministic skill/age sort (no RNG), so a colony with no pool or no living ruler — the bare analytical sims — clears it to nothing and is undisturbed. Reported by `WeddingPrinter` (`Weddings.csv`, one event-driven row per couple). When a weekend's wedding demand goes **unmet** — seekers were present but **no wedding happened** (the pool ran out of opposite-gender candidates, or none were affordable) — the colony **recruits one immigrant** into the pool for next time: a young, fresh adult of random gender (`PeasantPool.addImmigrant`), paid for in **gold that leaves the economy** (an uncredited withdrawal from the ruler's treasury, `Bank.extractExternalFunds`, the documented inverse of `injectExternalFunds`) at `WeddingConfig.immigrantCost()` = `immigrantCostFactor · baseCost` (default `100·2` = 200 copper ≈ 0.17 gold) — but **only if the ruler can afford it** from a positive treasury, else the recruitment waits. This keeps weddings from stalling permanently once the founding reserve drains (the `HanseaticEconomy` example more than doubles its weddings with it), at the cost of a slow gold outflow that the `Peasants.csv` `Imported` column tracks. Parameters live in `WeddingConfig` (override per-run via `SimulationHarness.setWeddingConfig` before `createMarkets`; `capacity 0` disables weddings — and with them the recruitment).

### Bank is the payment + credit system

`Bank` (`bank/Bank.java`) is an **instance** class (constructed with a `BankConfig`); each agent holds its accounts at a specific `Bank` (`agent.getBank()`, injected through every constructor). A bank holds one `Account` per agent keyed by the agent's ID (which doubles as account number). Every account has a `CHECKING` and `SAVINGS` balance; a **negative savings balance is a loan**. Money only moves between checking accounts. Payments are **agent-routed**: a transfer is `payerBank.withdraw(payerID, amt)` + `payeeBank.credit(payeeID, amt, purpose)`, which is correct even when the two parties bank at different `Bank` instances (there is no combined `pay` — split it). The payment-purpose codes `Bank.PRIIC`/`SECIC`/`OTHER` stay `static` (shared across banks). Each bank also carries a `CurrencyType` (`bank/CurrencyType.java`: `COPPER`/`SILVER`/`GOLD`, set via `BankConfig.currency`, default `COPPER`) — its denomination, so a colony can run banks in different currencies — the default hierarchy banks commoners in copper, nobles in silver, and the ruler in gold. The colony has a **fixed exchange rate** with copper as the base unit (prices are quoted in copper): silver = 100 copper, gold = 1200 copper (historically plausible 15th-century ratios — gold:silver ≈ 12:1, silver:copper ≈ 100:1). Because the rate is fixed and freely convertible the currencies are **fungible**, so all internal accounting stays in copper and the rate is used only to convert non-copper amounts and to **display** balances in a bank's own currency — `CurrencyType.toCopper`/`fromCopper`/`convert` (exposed as `Settlement.convert`), with `BankPrinter`/`NoblesPrinter`/`PersonsOfInterestPrinter` each carrying a `Currency` column and reporting monetary values in the holder's currency. At the fixed rate alone, currency assignment does not change the dynamics (the copper-denominated figures are unchanged), so changing the *display* currency changes nothing. The dynamics only move once exchange carries **friction**: the `BankConfig.exchangeFeeRate` knob (default 0) makes a non-copper bank a **money-changer** that skims a fee whenever its client's money crosses the copper boundary. Because every price is quoted in copper, a non-copper account converts on *every* payment, so the fee is charged inside `Bank.withdraw`/`credit` exactly when `currency != COPPER` (the payer bears it on a withdrawal, the payee on a credit), retained as the bank's equity/profit; a copper bank is already in the base currency and never charges. The default silver and gold money-changers set it to `0.02` (`DEFAULT_EXCHANGE_FEE_RATE`), so a silver noble loses 2% converting each dividend in and each purchase out, and the gold-banking ruler loses 2% whenever it buys (copper-quoted) enjoyment — both banks profit from the fee they retain. This is the conversion friction a *fixed* rate previously could not provide (a genuinely *floating* rate would be a further, larger step). This tiered scheme is the **default banking system** every settlement uses, provided by `SimulationHarness`: `getCopperBank()` (the zero-profit base-currency bank for commoners — laborers and firms), `getSilverBank()` and `getGoldBank()` (the nobles' and ruler's money-changers, each charging `DEFAULT_EXCHANGE_FEE_RATE` = 2%). Each is created **lazily on first request** and shared thereafter, so a colony carries only the tiers its classes use. Because every standard settlement now has a default ruler, every standard run populates **gold** (the ruler), and the runs whose nobles bank in silver (`AristocraticEconomy`, `StrategicEconomy`, `HanseaticEconomy`) populate **silver** too; `HomogeneousEconomy`/`HeterogeneousEconomy` bank their default nobles at the single copper bank, so those run copper + gold. (Request copper before silver/gold so the banks are numbered/ordered copper, silver, gold.) The bare exceptions that skip the ruler (and the export sector) are the analytical sweeps `ScaleSweep`/`LatitudeSweep` (one copper bank) and `SmallOpenEconomy` (two *copper* banks, agents split across them as in `TwoBankEconomy` — which itself adds the ruler's gold bank on top of its two copper banks). Critically, agents read their last-step income from public `Account` fields — `priIC` (primary income: wage/sales), `secIC` (secondary: dividends), `interest` — and **reset these to 0 at the end of their own `act()`**. Loan interest rate responds to the loan/deposit gap; deposit rate redistributes collected interest to creditors. By default the bank is a **zero-profit pass-through** (`depositIR · totalDeposit == loanIR · totalLoan`). Three `BankConfig` knobs let it retain profit into a cumulative `equity` field (charted as the `Equity` column in `BankPrinter`): `spread` keeps a fraction of the interest margin instead of paying it to depositors, `feeRate` charges the payer a per-withdrawal transaction fee, and `exchangeFeeRate` charges the currency-exchange fee described above (only on a non-copper bank). All default to 0. That retained profit is **distributable to a noble owner**: the bank separately tracks `cumulativeProfit` (gross spread + fees ever retained) minus `distributedProfit` paid out, exposed as `getDistributableProfit()`, and `payDividend(amt)` skims that profit out of equity. This is deliberately *only* the profit slice of equity — not the estates-in-transit or injected external funds that also pass through equity — so a noble can draw a bank dividend without disturbing inheritance or the open-colony money buffer. Both fields are passive (and `payDividend` is only called by a noble owner), so a run with no bank-owning noble is unaffected. **Caveat:** the zero-profit pass-through is load-bearing for the model's calibrated stability — empirically `spread ≈ 0.005` is safe and profitable, but `spread ≈ 0.02` already starves the laborer population via a price spike; making the bank meaningfully profitable needs the firm/laborer parameters retuned.

The `equity` field is also the bank's link to **inheritance** and the **open colony**. A dead account holder's estate is folded into equity (`inheritAndClose`) and immediately handed to its successor household (`openInheritedAcct`, which draws the inherited balances back out of equity) — so money stays in circulation, and in a closed run these net to zero. `injectExternalFunds(amount)` adds genuinely new money from outside the colony into equity; it is the entry point for an open colony, where accumulated equity bankrolls new immigrant households (each opened via `openInheritedAcct`, conserving every external dollar as either circulating money or equity below the immigration threshold).

### Agents

`Agent` (`agent/Agent.java`) is the abstract base: unique ID, alive flag, `act()`, and `getGood(name)`. The three **household** types — `Laborer`, `Noble` and `Ruler` — additionally extend `AbstractHousehold` (`agent/AbstractHousehold.java`), the base that consolidates everything a named, aging dynasty shares (each previously carried it verbatim): the identity draw (head / `birthDate` / `foundingDate` / `skill`, skill drawn **before** the head so the given name's rarity tracks it), account opening (fresh endowment vs. equity-funded inheritance/immigration), `ageDays()` / `getAgeYears()` / `getWealth()`, and death settlement — `dieAndSettleEstate()` snapshots the estate and closes the account, `checkOldAgeDeath()` runs the per-step old-age check, `resetIncomeAccumulators(acct)` clears the income fields. A subclass supplies only how it earns and spends in `act()` and which goods it holds. The `eos.agent.Household` interface they implement carries the **polymorphic contract** the step loop and printers dispatch on — `isWorkforce()` (does this household count toward the colony's labor force, whose loss ends the colony — true only for `Laborer`), `role()` (its label in logs and the persons-of-interest roster), `successor(colony)` (its own heir, see the step loop), and the reporting accessors `getIncome()` / `getWealth()` / `getBank()` — so a new population type slots in without editing `Settlement` or the printers. Two families:

- **Laborers** (`agent/laborer/Laborer.java`) earn wages, decide consumption vs. savings (target-savings model sensitive to the real interest rate), buy Necessity and Enjoyment, eat the `RationSize.FINE` ration (half a unit) of Necessity per step (`LaborerConfig.eatAmt`), and `die()` if they can't eat. They steer their consumption budget toward Necessity in proportion to how far their stock sits below the **target necessity stock** (`colony.getTargetNStock()`, default 26 units ≈ a 26-day buffer), spending the surplus on Enjoyment; that target is a property of the **colony** (a colony-wide environmental constant), not a per-laborer `LaborerConfig` value. **Skill lives on the `Person`, not the household** (`eos.skill`): every person carries a `SkillTracker` of **12 RimWorld-style skills** (`Skill` enum: Construction, Plants, Intellectual, …), each a `SkillRecord` with a `level` in `[0, 20]` and a `Passion` (`NONE`/`MINOR`/`MAJOR`, scaling its learn rate); `SkillRecord.learn(xp)` advances the level along a piecewise-linear XP curve `(0→100, 9→1000, 19→3000)`. The **single scalar** `Household.getSkill()` is now **derived** as the head's `SkillTracker.overallLevel()` (the rounded mean of the 12 levels) and drives **notability and name rarity**. A worker's **labor productivity**, though, is now **firm-specific** — its level in the firm's own skills (not the overall scalar) mapped through the **piecewise** curve `Household.productivityOf` — `max(0.01, (skill/10)²)` up to skill 10, then the steeper `(skill/10)³` above it (skill 0 → 0.01; skill 5 → 0.25, unchanged; skill 10 → 1, the legacy homogeneous unit; skill 20 → 8, so mastery is disproportionately rewarded), computed when the labor market clears (see `LaborMarket`). A person's skills are drawn at birth by `Demography.newSkillTracker(meanSkill)` on the demographic **skill RNG** (a stream separate from both the economic and the mortality streams): each skill's level around a **gender-specific colony mean skill** (`colony.getMeanSkill(gender)` — `meanSkillMale` (default 5) or the lower `meanSkillFemale` (default 2), both colony-start properties; sd 3, clamped) plus a random passion (≈60/30/10 NONE/MINOR/MAJOR), a fresh draw for every head including each successor (so skill is not inherited). Averaging 12 draws around 5 makes a typical male worker's `overallLevel ≈ 5`, so productivity ≈ `productivityOf(5) ≈ 0.25` with low spread — aggregate effective labor sits below the old "everyone = 1" world; founding a colony with a higher mean raises it. Gender is rolled (50/50) only for people generated into the `PeasantPool`; directly-drawn heads — nobles, the ruler, and pool-less laborers — are `MALE`, so a female reaches the labor force only by promotion. **Skills grow through work and fade through disuse**: each step a worker is employed it gains XP in its employer's skills (see `LaborMarket`), while `Settlement.newDay` ticks every living person's `SkillTracker` once a day, decaying mastery above level 10 (passion-eased; see `skill/`). Labor productivity is read **live** from the worker's current level *in the firm it works* (so a farmer's necessity output tracks its `PLANTS`, a noble's export its `INTELLECTUAL`), and `Household.getSkill()` likewise tracks current skills — each worked skill settling at a training-vs-decay equilibrium, and workers specializing by what they do most. Each laborer is a **household** holding a `List<Person>` of members (head first, exposed via `getMembers()`/`getMemberCount()`; **size 1 — just the head — for now**, the seam for households to grow later). The head is a `Person` (a given name — male for a directly-founded laborer, or the promoted peasant's own, possibly female — plus a unique dynasty surname from `colony.getNames()`, see `name/` below, and its `SkillTracker` and `Gender`); the death log reports `head.fullName()`. The head carries a `birthDate` (the source of truth for its age — the span from `birthDate` to the in-game date) and may also `die()` of old age each step against the `mortality/` life table; founding heads draw a working-age birth date from a Gaussian centered on `colony.getMeanInitAgeYears()`. Distinct from that, the household records a `foundingDate` — the in-game date it came into being, the moment its already-grown head *arrived in the colony and founded it* (a working-age adult, born years earlier, sets up the household now). So `birthDate` is the head's biological birth; `foundingDate` is the household's creation. The shared constructor builds the head's `SkillTracker` **before** naming the head, so the given name is picked at a rarity that tracks the head's overall skill (abler households get rarer, more distinctive given names; the surname is still the plain unique-dynasty draw — see `name/`). Any death folds the estate into the bank's equity; the household is then replaced via `Settlement.addReplacementPolicy` — by **promotion from the peasant pool** in a ruler-bearing colony (a fresh, skill-endowed household, *not* an inheritor — so the dead estate stays in equity and the labor force only holds while the reserve lasts; see *Peasant pool*) or by a **same-dynasty heir** in a bare pool-less colony (inheriting the estate, keeping money and the labor force stable). A separate **equity-funded constructor** creates a fresh new-dynasty household for open-colony immigration; an **adopt-a-`Member` constructor** lets a promoted peasant *become* a laborer head (keeping its name, skills and age). Mortality is a core, always-on feature: every head ages and the demographic draw is taken on a separate salted RNG, so it never perturbs the economic random stream (see `mortality/` below).
- **Firms** (`agent/firm/`): `CFirm` produces Capital (machines); `ConsumerGoodFirm` (abstract) is subclassed by `EFirm` (Enjoyment) and `NFirm` (Necessity), which combine labor + capital via a **Cobb-Douglas production function** `A·L^β·K^(1-β)`. Consumer firms adjust output by marginal profit and decide whether to buy/replace capital based on return-on-capital vs. interest rate and capacity utilization. The **wage budget** follows one of two rules (`FirmConfig`): the legacy cash-flow-gap nudge (`laborShare == 0`), or — since the labor market pays a single wage `totalBudget/N` — a **labor-share** rule (`laborShare > 0`) that budgets a fixed fraction of revenue for wages, so total wage spending scales with the colony and wages aren't outrun by a growing labor pool. `SimulationConfig.laborShare` defaults to 0.5, so the default run uses the labor-share rule (a stable but nominally-inflating closed colony).

- **Nobles** (`agent/noble/Noble.java`) are the **owners** of firms **and banks**: rentier households that live off profit, not wages. A noble is a named household exactly like a laborer — its `Person head` is drawn with `colony.getNames().nextHead()` (unique dynasty surname) — but it never enters the labor market. Each step it *pulls* a dividend from every owned firm (`dividendRate · max(0, firm.getProfit())`) and every owned bank (`dividendRate · bank.getDistributableProfit()`), moving the surplus to the noble through the bank's **secondary-income channel** (`Bank.SECIC` / `Account.secIC`). It then spends a fraction of its liquid wealth back into the consumer-good markets (so dividends circulate) and saves the rest, and **eats the `RationSize.LAVISH` ration** (a full unit/day) from a household necessity larder it keeps stocked (`necessityReserveDays`, default **30** days of its own household; capped so a noble does not hoard food — it buys necessity but never ate it before, which let an uncapped stock grow without bound and starve the workforce out of the market). This is the "option A" ownership design: nobles influence the labor market only **indirectly**, via consumption → firm revenue → the labor-share wage budget. The dividend draws only on public firm getters and the bank's existing transfer primitives, so it touches no firm/bank code, and a run with no nobles is unaffected. Like laborers, nobles **age and die** (a `birthDate` sampled the same way, old-age death against the life table), and a **successor of the same dynasty inherits the estate and the holdings** (its `firms` and `banks` lists, via `Noble.successor` and the colony's built-in household-succession policy — no per-sim rule) — so the aristocracy persists across generations and keeps drawing dividends. A noble's head carries a full `SkillTracker` exactly like any person (drawn around the colony mean, re-rolled each generation); its `overallLevel()` drives the noble's notability and name rarity, while its **`INTELLECTUAL`** skill drives its **strategic-export productivity** — the export firm declares `{INTELLECTUAL}` as its labor skill, so the labor market derives the noble's output from its INTELLECTUAL level (the general per-firm rule, see `LaborMarket`; there is no longer a noble-specific productivity method). Since the noble *trains* `INTELLECTUAL` by working the sector, this is a feedback loop: it grows more productive at exporting the more it exports (up to the train-vs-decay equilibrium). Nobles have no starvation death (they are wealthy rentiers), only old age. Parameters live in `NobleConfig` (`dividendRate`, `consumptionRate`, `necessityShare`, `necessityReserveDays`). `AristocraticEconomy` is the worked example (five nobles split ownership of all firms); `NoblesPrinter` writes the `Nobles.csv` time-series (with an `AvgAge` column).

- **Ruler** (`agent/ruler/Ruler.java`) is the settlement's single sovereign: the owner of its **gold** bank and its sole client (the bank has no others). Unlike a `Noble` it draws no dividends — it is a **treasury that indulges**: a named `Household` that ages, dies and is succeeded by a same-dynasty heir (via `Ruler.successor`, the colony's built-in succession policy — which also hands the heir the colony's ruler reference, so a builder bills the heir, not the dead sovereign; and it is tracked as a person of interest), earns no wage or dividends, but it **levies taxes**: each step, before it spends, it skims a fraction of every bank's distributable profit (`SimulationConfig.bankProfitTaxRate`, default **0.05**) and of every living noble's per-step income (`nobleIncomeTaxRate`, default **0.02**) into its treasury — the first slice of the taxation feature (`Ruler.collectTaxes`, run in `act()` before spending; a no-op when both rates are 0, which leaves runs byte-identical). It then spends a small fraction of its fortune (`consumptionRate`, a constructor param; `SimulationHarness.DEFAULT_RULER_CONSUMPTION_RATE` = 0.0002) on **enjoyment**, posting a buy offer to the (copper-quoted) enjoyment market exactly as a noble does. It also keeps a **lavish table**: it eats the `RationSize.GOURMET` ration (2 units/day) and restocks necessity toward a reserve — also copper-quoted, so that buying likewise fires the gold bank's FX fee (see *Goods → RationSize*). Its opening fortune (`SimulationHarness.DEFAULT_RULER_GOLD` = **50** gold, held in copper internally) is the **founding cash for the whole labor force**: on day 0 it capitalizes each promoted laborer with that laborer's skill-sum endowment (gold→copper, firing the gold bank's FX fee), then taxes and feeds the pool out of the remainder. It never starves (and, funding the pool's relief by overdrawing, may run a bounded deficit over a colony's life). **Every standard settlement has a ruler**, created by `SimulationHarness.createDefaultRuler()` (which also lazily creates the gold bank) — like the export sector it is part of the default setup, skipped only by the sweeps and `SmallOpenEconomy`. Because enjoyment is priced in copper, the ruler's spending converts **gold → copper**, firing the gold bank's `exchangeFeeRate` (its tax revenue, landing as gold, fires the same fee), so the gold bank, too, turns an FX profit. It is deliberately a distinct type rather than a `Noble`: `NobleConfig`'s 5%/step consumption would drain the treasury in months, and the ruler is a tier above the nobles, so this also keeps it out of `Nobles.csv`. The two reporting touchpoints — `PersonsOfInterestPrinter` (a gold-denominated row, income always 0) and the per-POI death log in `Settlement.newDay` — need no `Ruler` case: both read its `role()` ("Ruler") and the `Household` reporting accessors polymorphically. In a sim the ruler is created **before the labor force** (it holds the founding cash and promotes the laborers out of the pool), after the firms and any nobles. The class hierarchy thus mirrors the currency hierarchy: commoners → copper, nobles → silver, ruler → gold.

- **Builder** (`agent/firm/BuilderFirm.java`) is the firm that **physically grows the colony** — the only thing that can enlarge a *live* settlement (see *Settlement size and build slots*). It is a labor-only firm modeled on `StrategicFirm`, and is **staffed exclusively by peasants**: it hires on a dedicated `PeasantLabor` market (`BuilderFirm.LABOR_MARKET`) whose only workers are the `PeasantPool`'s peasants (the corvée labor of the poor), just as the export firm draws on a noble-only market. It converts that labor into *build-units* (`A·L^β`, capped by the `BuilderConfig.scaffoldCap` **scaffold cap**; daylight scaling rides along for free, as the workers already deliver day-length-scaled labor), which it applies to the `BuildProject`s in the colony's build queue — land clearance, road building, wall raising. The work it delivers is **billed to each task's sponsor at cost**: the firm that requested a slot pays for the land it will occupy, and the **ruler** pays for the ring's roads and walls (a sponsor short on cash overdraws into a small loan via `Bank.withdraw`, so building is never stalled by tight cash; the public-works bill is charged to the colony's *current* ruler, so a succession mid-build bills the heir, not the dead sovereign's closed account). Because the peasants are the ruler's wards (it feeds the pool), **the wage the builder pays for their labor is reimbursed to the ruler**: the pool registers each peasant with the ruler's account as the wage target (`PeasantPool.act`), so the builder's wage budget flows to the ruler when the `PeasantLabor` market clears, and the peasants train `CONSTRUCTION` by building. The builder stays a near **zero-profit conduit** (like `StrategicFirm`): sponsors fund the wages, which the ruler — the peasants' patron — collects. It registers itself as the colony's sole builder in its constructor (`Settlement.setBuilder`), bids a wage budget only while it has work queued (and lays its workers off when idle), and is reported by `BuilderPrinter`. Parameters live in `BuilderConfig` (the build-curve `A`/`beta`, the scaffold cap, seed/working wage budgets, and per-slot work costs for land/road/wall) — all placeholders pending calibration. A colony with **no** builder simply never grows once live; a colony **with** a builder also needs a `PeasantPool` (its workforce). **Every pool-bearing colony now gets a default builder** — `createDefaultPeasantPool` creates one (at the pool's bank) before the pool, so the standard sims all carry it. In practice it stays **dormant**: those colonies found at a size that already fits their firms and add none during the run, so no growth ring is ever queued and the colony collapses before it would expand — but the wiring is live and ready for the pool's refill phase, when growth becomes possible.

- **Marriage** (`market/WeddingMarket.java`, *Marriage*). A household head may take a **spouse of the opposite gender** as a second `Member`, wed off the `WeddingMarket` (above) out of the `PeasantPool`. The spouse adopts the household's surname and is a **full member**: it **works** (every household member is posted to the labor market with its own skills — `LaborMarket.addEmployee(Laborer)` loops the members, and `Noble.postNobleLabor` does the same for the noble export sector — so head and spouse may end up at different firms and train different skills, but all wages credit the one household account; the ruler's spouse does not work, as the ruler sells no labor), it **eats** (every household scales its ration by `getMemberCount()` — a laborer eats `FINE`, a noble `LAVISH`, the ruler `GOURMET` *per member*; the laborer's necessity stock target and minimum-buy scale with the household too), and it **ages and dies** on the same per-member schedule. Member mechanics reuse the existing seam: `AbstractHousehold.addMember` adds the spouse, and `checkOldAgeDeath` already **promotes a surviving member to head** when the head dies — so a widowed spouse simply becomes the new head (no succession/inheritance event, and a female ex-spouse thus enters the workforce as head). If a laborer household cannot feed everyone, the **non-head members starve off first** (`removeNonHeadMember`) and the household lives on; only the head going unfed dissolves it. A household seeks a spouse (`AbstractHousehold.seekSpouseIfSingle`, called from each `act()`) only while it has just its head, so each household holds at most one spouse for now (households grow no further yet).
- **Peasant pool** (`agent/PeasantPool.java`) is the **universal labor model** for ruler-bearing colonies (see `docs/peasant-pool.md`). It is an `Agent` holding a `List<Member>` — people with skills, a **gender** and an age but no household, no wage, no enjoyment — that the ruler **feeds** and **promotes from**. The pool is seeded with `cfg.peasantPoolSize()` peasants (default 900; each rolling a 50/50 gender on the demographic RNG — males drawn around `meanSkillMale`, females around the lower `meanSkillFemale`, named from the matching given-name table); `SimulationHarness.foundLaborersFromPool` then has the **ruler promote the ablest `round(promotionRatio · poolSize)`** (highest `overallLevel`) into laborer households on day 0 — each adopting the peasant as its head (keeping its gender), endowed with the **sum of its 12 skill levels** in copper (funded from the ruler's treasury, gold→copper FX) and **carrying its larder ration out of the pool** (`drawPromotionStock`, so the food is conserved rather than created). The rest stay as the **standing reserve**. The same `promoteToLaborer` is the laborer replacement policy: a dead laborer is replaced by promoting one reserve peasant, until the reserve drains — after which deaths go unreplaced, the shrinking workforce produces less food, and the colony **collapses** (the default scale lasts ~5–7 years; the refill that would make this sustainable is a later phase). **Reserve peasants eat the `RationSize.SIMPLE` relief ration** (`RELIEF_RATION`, 0.25 units/day — the poor-relief tier; a worker eats `FINE` = 0.5, a noble `LAVISH` = 1) drawn from a **per-peasant larder** (`BUFFER_DAYS` = 15 units, the ration a promoted peasant carries out). The pool **refills that larder toward its full per-peasant size, but only up to a price-sensitive money budget** (`RELIEF_BUDGET_PER_PEASANT`, billed to the ruler who borrows): so as food grows scarce the pool buys less and **yields it to the working laborers** (who, unlike the pool, have a guaranteed minimum), rather than the extra mouths out-bidding the workforce into starvation — without this the doubled population collapses the colony in weeks. (The refill target is the *full* larder, not the ration-scaled one — refilling toward a ration-scaled target left the pool idling on its seed larder until it drained and then crashing, instead of buying daily.) The low ration plus the larder buffer let the reserve **persist a couple of years** as a working replacement pool before it drains. Pooled peasants age, die of old age, and have their skills decay daily, so the reserve also drains demographically. The one inflow today is **gold-funded immigration from the `WeddingMarket`** (see *Markets*): a weekend whose wedding demand goes unmet recruits one young immigrant into the pool, paid for in gold that leaves the colony — replenishing the marriageable reserve once founding has drained it. The whole feature is keyed on the pool's **presence** (no config flag): a colony with a pool founds/replaces from it (and gets a default builder — see *Builder*), a bare colony (the sweeps, `SmallOpenEconomy`) founds directly with same-dynasty succession. Reported by `PeasantPrinter` (`Peasants.csv`). The knobs are `peasantPoolSize` and `promotionRatio` (`SimulationConfig`): a high ratio promotes most of the pool (small reserve, the colony lives on its laborers), a low ratio holds a large reserve.

Note: `ConsumerGoodFirm.act()` contains explicitly-labeled **"hacks"** — capital-purchase nudges that only fire after a warm-up period (`timeStep > config.warmupDays()`, default 2000) once the rolling average profit is meaningful, spurring an extra machine on a profit spike (`profit > capitalNudgeProfitFactor · avgProfit`, with the utilization bar lowered to `capitalNudgeUtilThreshold`) or one fewer on a deep loss. These knobs live in `FirmConfig` (defaults `2000`/`5`/`0.8`), but the block is model **calibration tuning stability**, not incidental logic — changing the defaults needs re-validation against the test invariants and the `ScaleSweep`.

### Goods, printers, utils

- `good/` — `Good` is a mutable quantity holder; `Capital`, `Labor`, `Necessity`, `Enjoyment` are the concrete goods. `RationSize` is an enum of named daily necessity rations (`GOURMET` 2.0, `LAVISH` 1.0, `FINE` 0.5, `SIMPLE` 0.25, in units/person/day): the social hierarchy of the table — the **ruler** eats `GOURMET`, **nobles** `LAVISH`, **workers** (laborers) `FINE`, **pooled peasants** `SIMPLE` (poor relief). Workers eating the modest `FINE` ration (rather than a full unit) is what frees enough food for a colony to also feed its peasant pool.
- `io/printer/` — each `Printer` writes one CSV via `CSVPrintWriter`. Register printers with `Settlement.addPrinter(...)` (which writes the header row immediately) and finalize with `Settlement.cleanUpPrinters()` after the run. **Cadence is monthly**: the base `Printer.shouldPrint(colony)` gates every row on `getDate().getDayOfMonth() == 1` (within the printer's `[start, end]` step bounds) — there is no per-step `period`/`stepSize` knob. **Number formatting is centralized in `CSVPrintWriter`**: any `Double`/`Float` cell is rendered to two decimals (`%.2f`); ints, dates and already-formatted `String` cells pass through untouched. `BankPrinter` pre-formats its four interest-rate columns and the inflation column as truncated percents (e.g. `0.45%`) via its `formatPercent` helper, so those reach the writer as strings. `LaborersPrinter` aggregates over the **living** laborer population read from `colony.getAgents()` each step (so it tracks replacements and immigrants, not a fixed initial array), averaging over the live count, with `AvgAge` and `Count` columns. `NoblesPrinter` does the same for the noble population (dividends, income, consumption, wealth). `BuilderPrinter` charts the colony's construction — its `Size` and `EffectiveSlots` (which step up as a ring completes), the builder's `Delivered`/`TotalDelivered` build-units and finances, and the `ActiveTasks`/`RemainingWork` of the ring being built (0 when idle). `WeddingPrinter` is **event-driven, not monthly**: it writes one row per couple as weddings happen (the `WeddingMarket` records each step's weddings when it clears, just before printers run), with both parties' name/type/gender/skill/age and the bride-price — registered in `addCommonPrinters` (`Weddings.csv`), so quiet steps and pool-less colonies contribute no rows.
- `io/SimLog.java` — event logging via `java.util.logging`. `SimLog.init()` (called first thing in each `main`) installs a formatter that prefixes every record with the current **in-game date** (`Settlement.getDate()`, e.g. `1452-03-14: WARN …`), writing to **stderr** with per-record flush. Classes that emit events are annotated with Lombok `@Log` and call `log.info(...)` for events or `log.warning(...)` for anomalies (e.g. a price crossing `PRICE_SKYROCKET_FACTOR`×its initial level in `ConsumerGoodMarket`). Logged events include the **colony being founded** (`Settlement.start`) and **dying** (`updateLifecycle`, when its last laborer is gone), and a **notable person** — any household whose head's overall skill is above `Household.NOTABLE_SKILL` (15) — recorded by name at creation (rare at the default `meanSkill` 5; commoner in a higher-skill colony). The colony tracks a roster of **persons of interest** — every `Noble` and every notable household — to which households register themselves at creation (`Settlement.addPersonOfInterest`, exposed via `getPersonsOfInterest()`) and from which they are removed when they die. **Death logging is confined to that roster**: only a person of interest's death is logged, exactly once (in `Settlement.newDay`'s dead-agent sweep), so ordinary laborer deaths produce no log spam. The roster's yearly *listing* is a CSV, not a log: `PersonsOfInterestPrinter` writes one row per living person of interest on the first day of each year (columns `Date, Name, Type, Skill, Age, Income, Wealth, Currency`); it is registered by the noble-bearing sims (`AristocraticEconomy`, `StrategicEconomy`). It is **fully polymorphic** — `Type` is each household's `role()`, and income/wealth/currency come from `Household.getIncome()` / `getWealth()` / `getBank().getCurrency()` — so a new population type appears with no case added. This is for discrete events — bulk time-series still goes through printers to CSV. Note `Settlement.run`'s progress counter prints the in-game date to stdout every January 1st (once per in-game year), so progress and events are on separate streams.
- The in-game date is the canonical display unit: every place a step was shown as an integer now shows `Settlement.getDate()` — the SimLog prefix, the progress counter, and the first column of every printer CSV (header `Date`, ISO `yyyy-MM-dd`). `Settlement` holds the start date (set from `cfg.startDate()` by `SimulationHarness`); `getDate()` returns `startDate + timeStep` days. The integer `timeStep` is still used internally for control flow (`getTimeStep() == 0`, the `> 2000` calibration hacks, the printers' `[start, end]` step bounds). Printers emit one row on the **first day of each in-game month** (`Printer.shouldPrint` gates on `getDate().getDayOfMonth() == 1`); there is no longer a `period`/`stepSize` cadence knob.
- `util/` — `Rng` (minimal seedable RNG over `java.util.Random`; **`GameSession` mints one per colony** — derived from the session seed and the colony's index — and the colony exposes it via `colony.getRng()`, so colonies in one session have independent, reproducible economic streams; seed the session for reproducible runs) and `Averager` (rolling window average used for inflation and long-term rates).
- `name/` — `NameRegistry` holds the **complete name sets** for a session, loaded from the tiered JSON tables in `src/main/resources` (`dynasty-human.json`, `male-human.json`, `female-human.json`) via Jackson (`jackson-databind`, the one runtime dependency beyond Lombok). Each tier owns a cumulative-weight band `(prev, percent]`, so `NameTable.pick(rng)` draws weighted toward common names. The names are stored **common→rare**, and `total` ≈ 99 acts as a percentile scale, so `NameTable.pickAtRarity(rng, percentile)` biases the draw toward a chosen slice of the rarity range (0 = commonest, 1 = rarest, jittered ±0.125 for variety) — used to give **abler households more distinctive names**. The `GameSession` owns the registry and drives it from a **separate naming RNG** (a salted copy of the seed) so naming is deterministic but never perturbs the economic random stream. Dynasty surnames are drawn **without replacement** so every household *in use* across the session — in any colony — has a unique surname; given names are drawn **with replacement**. The pool is finite, so an extinct dynasty's surname is **recycled**: `releaseDynastyName(surname)` returns it (with its original weight) to the drawable pool when a household dies with no successor, so the uniqueness invariant is over *living* households, not permanent consumption. The registry tracks the `inUse` set for this and guards against releasing a surname that isn't out. `nextHead()` returns a `Person` (male given name + unique dynasty surname); `nextHeadInDynasty(surname)` draws a fresh head keeping an existing surname (a successor household, which therefore does **not** consume or release a surname). The **rarity-aware overloads** `nextHead(nameRarity)` / `nextHeadInDynasty(surname, nameRarity)` draw the given name via `pickAtRarity` instead — the household passes the head's `overallLevel() / Household.MAX_SKILL`, so a household's given-name rarity tracks its head's overall skill (surnames are unaffected, still the unique-dynasty draw). A `Person` returned by the registry is a **name only** (`skills == null`); the household attaches the head's `SkillTracker` via `Person.withSkills(...)`, and `Person` identity (`equals`/`hashCode`) is over the name, not the mutable skills. Households are tagged with the `eos.agent.Household` interface (identity members `getMembers()` with the head-first convenience `getHead()` and `getMemberCount()`, plus `getSkill()` — now derived from the head's `SkillTracker` — `getAgeYears()`, the `MIN_SKILL`/`MAX_SKILL`/`NOTABLE_SKILL` constants, the static `productivityOf(skill)` curve, and `isNotable()`; the polymorphic members `isWorkforce()`, `role()`, `successor(colony)`; and the reporting accessors `getIncome()`/`getWealth()`/`getBank()`) so the step loop can recycle the surname of one that dies unreplaced, succeed and report each household without switching on its type, and so all three household types share the skill and notability concepts. The shared machinery itself lives one level down in `AbstractHousehold` (see *Agents*), which all three extend.
- `skill/` — the **person skill system** (RimWorld-style, minus genetic aptitudes and traits). `Skill` is an enum of the 12 skills (Construction, Plants, Intellectual, Mining, Shooting, Melee, Social, Animals, Cooking, Medicine, Artistic, Crafting); `Passion` (`NONE`/`MINOR`/`MAJOR`) scales both the learn rate (0.35 / 1.0 / 1.5 via `learnRateFactor()`) and, inversely, the decay rate (1.0 / 0.5 / 0.0 via `decayRateFactor()` — a major-passion skill never fades). A `SkillRecord` holds a `level` (clamped `[0, 20]`) and a `Passion`; `learn(xp)` adds passion-scaled experience and advances the level (possibly several at once, capped at 20) along the XP curve — `xpRequiredForLevelUp(level)` reads the piecewise-linear curve through `(0→100, 9→1000, 19→3000)` via a tiny `linearInterp(points, x)` helper, clamped to the endpoints. `decay()` is the inverse: a skill **above the floor (level 10)** loses `level · 0.05 · passion.decayRateFactor()` XP/day (higher levels fade faster, carrying underflow down a level), but never below the floor — basic competence is permanent, only mastery erodes without practice. A `SkillTracker` (one per `Person`) holds one `SkillRecord` per skill, created up front, with `getSkill`/`learn`/`tick` (tick decays every record), `level(Skill)` (a single skill's level, which the labor market reads for firm-specific productivity), and `overallLevel()` (rounded mean of the 12 levels, read through `Household.getSkill()` for notability and name rarity). Skills are randomized at birth by `Demography.newSkillTracker` and then **grow through work and fade through disuse**: `LaborMarket.clear()` grants 1 XP (passion-scaled) per employed step in each skill the worker's firm trains (`Firm.laborSkills()`), while `Settlement.newDay` calls `SkillTracker.tick()` once a day for every living household member; labor productivity is read live from the worker's level *in its employer's skills* (`LaborMarket.relevantLevel`, not the overall level), so each worked skill settles at an equilibrium where training balances decay (low-passion skills plateau near competence, passionate ones can still master), keeping long-run productivity bounded.
- `mortality/` — `LifeTable` is an age-specific mortality schedule (the bundled `WEST_LEVEL_3` is Coale-Demeny West Level 3, e0 = 25; `nqx` per age group → a per-day death probability). `Demography` (owned by `GameSession`) is the per-session service for the random draws a household makes **at birth**: it draws founding household ages from a Gaussian truncated to working age and decides each step whether a head dies of old age (on its salted **mortality RNG**), and it builds a new person's skills (`newSkillTracker(meanSkill)`: one randomized `SkillRecord` per `Skill` — its level around the gender-specific colony mean skill via `sampleSkill`, clamped to `[0, 20]`, plus a random `Passion`) and rolls a person's gender (`sampleGender`, 50/50) on a **second, separate skill RNG** so these draws don't perturb the mortality stream either. Both RNGs are salted copies of the session seed, separate from the economic one, so all these draws are deterministic per seed yet never perturb the economic random stream. Mortality is a core always-on feature (no on/off toggle).

### Daylight and the solar package

- `solar/` — a vendored sunrise/sunset calculator (the Lucky Cat Labs / `ca.rmen` algorithm, ported from `BigDecimal` to `double`). `SolarEventCalculator.computeSunriseTime/computeSunsetTime` turn a `GeoLocation` (a `(latitude, longitude)` record — the project-local replacement for the original's external `Node`), a `Zenith` (the solar declination: `OFFICIAL` 90.83°, `CIVIL` 96°, `NAUTICAL` 102°, `ASTRONOMICAL` 108°) and a `java.util.Calendar` date into a local clock time. It is **legacy `Calendar`-based**, so `Settlement` bridges from its `java.time.LocalDate`. The calculator **throws** `UnsupportedOperationException` when an event does not occur (the sun never reaches the zenith — e.g. no astronomical twilight at high latitude in midsummer); callers must catch this. (Note: the port also fixed an inverted range-normalization bug in `getSunTrueLongitude`.)
- The colony has a fixed **geographic location** — `latitude`/`longitude` in decimal degrees (north/east positive), colony-start properties threaded from `SimulationConfig` through `GameSession.newSettlement` into `Settlement` exactly like `meanSkillMale`/`meanSkillFemale`/`targetNStock` (default London, 51.5074, −0.1278). From these, `Settlement` **computes and stores the day's solar times** at the top of every `newDay` (and seeds them in the constructor for step 0): `getDawn()`/`getSunrise()`/`getSunset()`/`getDusk()` (UTC `LocalTime`; **dawn/dusk are the astronomical** sunrise/sunset, sunrise/sunset the **official** ones) and `getDaylightHours()` (the precalculated sunrise→sunset span). An undefined event is stored as `null` (and `daylightHours` as `NaN`) rather than throwing — e.g. dawn/dusk are null at London around midsummer, when astronomical twilight lasts all night. These feed the daylight-scaled labor output (see `LaborMarket` above).

### The liturgical calendar

- `calendar/` — classifies every in-game date as a `DayType` (`WORKDAY`/`WEEKEND`/`HOLIDAY`) on the **15th-century Western (pre-Reformation) calendar**. `LiturgicalCalendar.dayType(LocalDate)` is a pure date lookup: a feast day is a `HOLIDAY` (it **outranks** the weekend — a feast on a Sunday reports as `HOLIDAY`); any other **Sunday** is a `WEEKEND` (in the period the *only* weekly rest day is Sunday — Saturday is a workday, so there is no modern Sat–Sun weekend); everything else is a `WORKDAY`. The feast set is a **curated list of fixed-date universal feasts** loaded once from `/feasts.json` (31 feasts: Christmas, Epiphany, Candlemas, the Apostles' days, Michaelmas, All Saints, etc.) — the **movable** Easter-derived feasts (Ascension, Pentecost, Corpus Christi…) are deliberately *not* modeled, and feasts do **not** vary by region. Like `SlotTable` it is pure (seed- and location-independent), so a single `LiturgicalCalendar` is **shared** by every colony in a `GameSession` (loaded at session start, threaded into the `Settlement` constructor) and exposed via `Settlement.getDayType()` / `getDayType(LocalDate)`.

- **`DayType` gates labor by firm.** Each firm declares which day types it operates on via `Firm.operatesOn(DayType)` (default: **workdays only**). On a day it does not operate a firm hires no one, so it produces nothing — its workers rest. The overrides realize the rest calendar: **enjoyment firms** also run on the **weekend** (`EFirm` → workday + Sunday: the leisure trade keeps going on the day off, but shuts on feast days); the **export firm** runs **every day** (`StrategicFirm` → the colony's strategic lifeline, worked by the dedicated noble class, so it ignores the rest calendar — on feast days the laborer firms are all shut, leaving the nobles the only ones working, as required); **necessity, capital and the builder** keep the default workday-only rule. So on a workday everything runs; on Sunday only enjoyment (laborers) and export (nobles); on a feast only export (nobles). The gate applies only once the colony is `started()` — the pre-run seeding clear hires every firm so step 0 has a workforce whatever day it falls on.
- **No labor-pool flooding.** Because labor is a single fungible pool, naively closing most firms on a rest day would dump the whole workforce into the few open ones. `LaborMarket` avoids this with a **fair-share cap**: a closed firm is still registered (so its wage budget still counts toward the total against which every firm's share of the workforce is sized) but is allocated *no one* — it reserves its budget-proportional slice and those workers simply rest. So an open firm gets only its own normal share; the rest of the workforce sits idle. On a normal day (all firms open) this is identical to the old allocation, so normal days are unchanged.
- **Revenue smoothing keeps closed firms solvent.** A `ConsumerGoodFirm`'s labor-share wage budget is sized off a **30-day average revenue** (`REVENUE_SMOOTH_WIN`), not the single day's revenue. This is load-bearing for the rest calendar: with the produce-from-yesterday's-labor lag, a firm that rests has a zero-supply day on reopening, and budgeting off that single zero day would collapse the wage budget to zero, starve the firm of labor, and oscillate into a production collapse. Smoothing over the rest-day cycle keeps the budget positive through closed-day runs.
- **Food gets extra capacity.** Necessity firms run a higher Cobb-Douglas `A` than the other consumer firms — `SimulationHarness.NECESSITY_TECH_FACTOR` (2.0) — because food production stops on the ~80 rest days a year while the population eats every day, so output on the working days must cover the whole year (and the extra hiring it drives also keeps necessity workers training `PLANTS`). **Caveat:** like the bank zero-profit pass-through, this whole rest-day wiring is calibration-sensitive — the smoothing window, the necessity factor and the productivity curve were tuned together against the test invariants and the sweeps; changing one needs re-validation.

## Conventions

- Reproducibility hinges on the single seed set at the top of `main`; the same seed yields identical runs.
- Display names & `toString()`: `Agent.getName()` returns a display name that defaults to the class simple name; subclasses override it via `setName(...)` (firms set "Capital Firm" / "<Product> Firm"). Markets and banks (not agents) carry their own default `name` — `good + " Market"` and `"Bank " + colony.nextBankNumber()` (numbered per colony). Laborers, firms, banks and markets override `toString()` with a concise, side-effect-free debug summary (no bank lookups, so they're safe to print on dead agents).
- When adding a new market or good, register the market with `Settlement.addMarket(...)` **before** constructing agents that reference it — agent constructors look markets up by good name (`Settlement.getMarket(name)`).
- Behavioral/model parameters of an agent or bank live in an immutable `*Config` record (`LaborerConfig`, `FirmConfig`, `BankConfig`, `WeddingConfig`) with a `DEFAULT` constant holding the canonical values; the owner stores it as a `final` field and the constructor takes it as a required param. This makes parameters per-instance (enabling heterogeneous populations / multiple banks) and tunable per run from the simulation's `main`. Structural/non-tunable constants — capital lifetime (`CFirm.CAPITAL_LIFE`), market price-band width (`ConsumerGoodMarket.zeta`), the inflation window on `Settlement` (`INFLATION_TIME_WIN`), payment-purpose index constants (`Bank.PRIIC`/`SECIC`/`OTHER`) — stay as `static final` on the relevant class. `Settlement` keeps its constants inline (`INFLATION_TIME_WIN` stays `static final`; the per-run start date, founding-age mean, target necessity stock and the **gendered mean skills** (`meanSkillMale`/`meanSkillFemale`) are constructor args supplied by `GameSession` via `newSettlement`). The target necessity stock and the mean skills live on `Settlement` rather than `LaborerConfig` precisely because they are **colony-start properties**, fixed for the colony's life and uniform across its households (they cannot vary across laborers within one colony). The colony also has a clear **lifecycle**: `start()` (called at the top of `run()`) marks it founded, and it **dies** terminally the step its last laborer is gone — `newDay` detects this (`updateLifecycle`), recording `getDeathDate()` and logging it; query via `isAlive()` / `isDead()`. `run()` **stops early** once the colony is dead (a dead colony produces nothing more, so finishing the horizon only burns compute). Mortality is always on — there is no toggle. The `ConsumerGoodFirm.act()` "hacks" calibration numbers now live in `FirmConfig` too (`warmupDays`, `capitalNudgeProfitFactor`, `capitalNudgeUtilThreshold`), so they are named and tunable per run rather than inline magic numbers, with defaults matching the original behaviour.
- Run-level parameters live in `SimulationConfig` — an immutable record of scalars plus nested sub-records (`PriceRange`, `FirmInit`, `CFirmInit`, `LaborerInit`) with a `DEFAULT`. Besides the calendar, population counts, market price bounds and per-agent-type initial state, it carries the demographic and macro knobs: `meanInitAgeYears` (founding-age mean), `targetNStock` (the necessity stock every laborer targets, default 26), `meanSkillMale`/`meanSkillFemale` (the centers of a person's per-skill starting-level draw by gender, default 5/2) and the colony's `latitude`/`longitude` (default London 51.5074, −0.1278, driving the daylight-scaled labor output) — all passed into the `Settlement` constructor — `externalInflowPerStep` + `immigrationThreshold` (the open-colony money inflow and the equity level that funds each immigrant; inflow 0 = closed), `laborShare` (the firms' wage-budget rule), `peasantPoolSize` (the peasant pool the colony is seeded with, default 900), and `promotionRatio` (the fraction of that pool promoted into laborer households on day 0, default 0.45). **There is no `numLaborers` field** — the labor-force size is `round(promotionRatio · peasantPoolSize)` for a pool colony, or the explicit count argument to `SimulationHarness.createLaborers(numLaborers, …)` for the bare pool-less ones (the sweeps, `SmallOpenEconomy`). The harness reads these and wires them: `enableExternalInflow(gatewayBank)` registers the inflow step-action + immigration policy (no-op when inflow is 0), and the consumer firms' `FirmConfig` is seeded with `laborShare` (overridable via `setFirmConfig`). **Each step is one in-game day**: the config holds a `startDate` (`LocalDate`, default 1444-12-11) and `durationYears` (default 25), and derives `numStep()` = days from `startDate` to `endDate()` (9131 for the default 25-year run). `dateAt(step)` maps a step back to its in-game date (step 0 == `startDate`). Each simulation's `main` binds `SimulationConfig cfg = SimulationConfig.DEFAULT` and reads values via accessors (`cfg.eFirm().checking()`); only the per-run differences stay in `main` — the **seed** and the **init logic** (`HomogeneousEconomy` fixed vs. `HeterogeneousEconomy` randomizing around the config values). The simulations are entry points with no constructor, so the config is a `main`-local bound to `DEFAULT` rather than a constructor param.
- The config records carry Lombok `@Builder(toBuilder = true)`. The hand-written `DEFAULT` constants remain the source of truth (no `@Builder.Default`); the builder exists so a run can derive a variant cheaply — `SimulationConfig.DEFAULT.toBuilder().durationYears(10).build()` — and so multi-field value records can be constructed by name instead of positionally. Builders are additive; nothing in the default run path calls them. Lombok is already a `provided` dependency (`@Getter` is used on agents); `@Builder` on records needs Lombok ≥ 1.18.20 (project is on 1.18.34).
