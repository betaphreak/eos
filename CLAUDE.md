# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`eos` is an agent-based macroeconomic simulation written in plain Java. It models a colony (closed by default, but optionally **opened** to a stream of external money that bankrolls immigration) of laborers, firms (capital, enjoyment, necessity), a bank (one by default, though the design now allows several), and markets that clear by supply/demand each time step. Laborer households age and die on a real mortality schedule. Running a simulation produces CSV time-series files for analysis (prices, volumes, firm finances, bank rates, laborer state).

The root package is `eos`, under the standard Maven layout at `src/main/java/eos/` (e.g. `src/main/java/eos/agent/firm/CFirm.java` is `eos.agent.firm.CFirm`).

## Build & run

Maven project, Java 21 (`pom.xml`). Tests use JUnit 5 (`mvn test`).

```powershell
mvn clean compile          # compile to target/classes
mvn exec:exec              # run HomogeneousEconomy (forks a JVM with -ea)
mvn exec:exec -Dsim.main=eos.simulation.HeterogeneousEconomy   # run another entry point
mvn package                # build the jar in target/
mvn test                   # run the JUnit 5 suite (each Simulation runs full as a smoke test)
```

`exec:exec` runs in a **forked JVM with assertions enabled** (`-ea`) because the code uses `assert` as real invariant checks; the main class is the `sim.main` property (default `eos.simulation.HomogeneousEconomy`). **Every settlement now carries a default _export sector_ and a default _ruler_.** The export sector is a single `StrategicFirm` staffed by a small class of nobles, bundled by `SimulationHarness.createDefaultStrategicSector(bank)`, so the strategic good and a noble class are part of the standard setup, not just the strategic/aristocratic runs; the export firm injects its export earnings into the equity of the bank holding its account, so **that bank is no longer a zero-profit intermediary** (its equity grows with cumulative net exports). The ruler is the settlement's sovereign (`SimulationHarness.createDefaultRuler()`), banking in **gold** — so the colony also carries a gold bank — and spending its treasury down on enjoyment, which fires the gold bank's FX fee. Both are created in every standard run; the only deliberate exceptions are the two analytical sweeps (`ScaleSweep`/`LatitudeSweep`) and `SmallOpenEconomy`, left as bare closed/open colonies so they keep measuring/illustrating what they were built for. The entry points are `HomogeneousEconomy` (homogeneous agents, plus the default export sector with its nobles banking at the single bank), `HeterogeneousEconomy` (heterogeneous, randomized init), `TwoBankEconomy` (two banks, agents split across them — a worked example of the multi-bank setup; the export firm and its nobles bank at A so the colony keeps its two banks), `ScaleSweep` (a **scale sweep** that scales the consumer-firm and laborer counts down together at the default firms:laborers ratio to find the smallest colony that stays stable), `LatitudeSweep` (a **latitude sweep** that places the standard closed colony at a range of latitudes and finds the highest one where it still feeds itself — because labor output is daylight-scaled, deep-winter shortfall starves the colony past a limit, ~58° at the default 8h reference), and `SmallOpenEconomy` (the **small open colony**: two banks, a small firm count — 2 enjoyment + 2 necessity + 1 capital firm — 90 laborers (this was `ScaleSweep`'s minimum stable *closed* scale before the daylight-scaled labor raised it to ~180; it stays viable here only because the colony is **open** and grows), and external inflow + immigration growing the population past its starting size; bank A is the gateway and so carries equity rather than being a zero-profit intermediary), and `AristocraticEconomy` (the default colony **plus an aristocracy**: **five** `Noble` owner households split ownership of all firms, and the senior noble also owns the **silver** bank — the nobles' money-changer, which profits from its FX fee; they draw firm and bank dividends through the secondary-income channel and spend it back into the consumer markets — a worked example of the owner/ownership design, including noble mortality + inheritance of holdings. These nobles also **staff the default export sector** — a `Noble` automatically works any `NobleLabor` market present — so they earn export wages on top of dividends; five of them keeps the fixed-wage export firm earning more than its wage bill, so the copper bank's equity grows rather than draining. The senior noble (n=0) owns the silver bank. Together with the default gold-banking **ruler** this is the full **three-currency hierarchy** — laborers and firms in `COPPER`, nobles in `SILVER`, the ruler in `GOLD`, **with conversion friction** (the silver and gold money-changers charge an `exchangeFeeRate` on every cross-currency payment) — the worked example of the `CurrencyType` split that the now-removed `BimetallicEconomy` used to carry on its own), `StrategicEconomy` (a colony whose nobles staff the default export sector but, unlike the others, **bank in silver** while the export firm banks in copper — so their export _wages_ cross copper → silver and the silver money-changer skims its FX fee on that primary income (`AristocraticEconomy` exercises the same FX on dividends, i.e. secondary income); the export earnings net of wages accrue to the copper bank's equity, the defining observable), and `HanseaticEconomy` (the worked example of **two settlements in one `GameSession`**: two neighbouring Hanseatic colonies near Lübeck — Lübeck Altstadt (53.867°N) and Bad Schwartau (53.920°N) — built from one session, so they have **independent economic streams** but a **shared** name pool and demography, hence dynasty surnames unique across *both*; they do not trade. Each is a closed colony at the minimum stable closed scale (180 laborers; k=4, see `ScaleSweep` — the daylight-scaled labor raised that floor from the old k=2 / 90, which is why even the open `SmallOpenEconomy` only survives 90 by *growing*) with **4 enjoyment + 6 necessity + 1 capital firm** and a strategic export sector worked by 5 nobles, all on the default tiered banking; its `run()` builds and runs *both* and returns the first. The nobles each open with 10 silver and **stockpile a necessity reserve** — `NobleConfig.necessityReserveDays` (7) × (laborers/nobles) units each, so the nobles collectively hold ~7 days of the population's necessity, a reserve for a later feature to draw on. Building that reserve in a subsistence colony would spike the necessity price (and if the nobles ever *fill* it they redirect to enjoyment and spike that instead), so the consumer firms are split **fewer enjoyment than necessity** (4 &lt; 6) — giving the necessity sector the headroom to absorb the reserve buying and letting the colony hold at the minimum scale). Each consists of a `static run()` that builds and runs the colony and returns a `SimulationHarness`, plus a `main()` that just calls `run()` — **except the sweeps `ScaleSweep` and `LatitudeSweep`**, whose `main()` runs a multi-colony sweep that prints a table to stdout (no CSVs) — `ScaleSweep.sweep()` the minimum stable scale, `LatitudeSweep.sweep()` the maximum survivable latitude (bisecting the survives/starves boundary) — while their `run()` is just the convention hook returning one default harness. A scale `k` there means `k` enjoyment + `k` necessity firms (+1 capital firm) and `k·45` laborers; "stable" generalizes the test invariants — population sustained, consumer prices finite/positive **and below 1000× their initial ceiling** (a runaway stays a finite double, so mere finiteness isn't enough), bank pools/rates finite, no thrown `-ea` invariant. Each `run()` creates a `GameSession` (which holds the seed and its own `Rng`) and from it a `Settlement` (`session.newSettlement(cfg.settlementName(), cfg.startDate(), cfg.meanInitAgeYears(), …)` — colonies are **named**, exposed via `Settlement.getName()`), then hands that colony to the harness. **`SimulationHarness` consolidates the shared construction** (markets, banks, firm/laborer loops, printers, run/cleanup) into a single `Settlement` instance; a `Simulation` supplies only what differs — which bank each agent uses (`IntFunction<Bank>`), and how each agent's initial state is drawn (`IntToDoubleFunction`). The call order in the harness mirrors the original sims and matters (deferred settlement + reproducible RNG consumption). A run is configured through `SimulationConfig.DEFAULT`.

Tests (`src/test/java`, JUnit 5) treat each simulation as a smoke test: `Simulation*Test` calls `Simulation*.run()` for a full ~10k-step run and asserts invariants on the returned harness (most laborers survive, prices finite/positive, default banks finite — no longer *zero-profit*, since every settlement now runs an export sector whose earnings accumulate in the holding bank's equity; the `TwoBankEconomy` case also checks both *copper* banks carry loan+deposit pools, the ruler's gold bank aside). `ScaleSweepTest` runs only the default scale to completion (the full sweep is too heavy for CI) and checks the minimum-selection logic as a pure function over synthetic results. `SmallOpenEconomyTest` checks its own invariants directly (population *grows* past its starting 90, both banks active, prices finite) rather than via `assertHealthy`, which assumes the large closed-colony population. `SettlementSolarTest` checks the daily solar times directly (no economy): London gets ~16.5 h of daylight at the June solstice vs ~7.8 h in December, dawn < sunrise < sunset < dusk on a normal day, and astronomical dawn/dusk are `null` at London midsummer. (`LatitudeSweep` is a `main`-only sweep with no test, per design.) `AristocraticEconomyTest` checks its invariants directly (population sustained, prices finite) rather than via `assertHealthy`, since its banks are deliberately profitable (non-zero equity); it also asserts the firms *and* the silver bank paid dividends (`bank.getDistributedProfit() > 0`), the nobles grew past their seed fortunes, and a noble succession occurred. It additionally carries the **ruler / three-currency** coverage formerly in `BimetallicEconomyTest` (now removed): three banks (copper, silver, gold), the gold bank profiting from the ruler's gold→copper purchases, and exactly one gold-banking ruler that has drawn its 10-gold treasury down. Settlement state (including the agent-ID counter and the `Rng`, now owned by the `GameSession`) is per-instance, but the `SimLog` logging handler is still a process-global static, so Surefire keeps `reuseForks=false` — a fresh JVM per test class isolates that static state (and keeps `-ea` on).

Output CSVs are written to an `output/` directory created relative to the **current working directory** at runtime (i.e. the project root when launched via Maven; see `io/printer/CSVPrintWriter.java`). Runs are reproducible: a `GameSession` derives each colony's `Rng` deterministically from its seed, so the same seed yields identical output.

## Architecture

### The step loop is the spine

`Settlement` is an **instance** (no longer a static singleton): a `GameSession` owns the seed and creates colonies from it (`settlement/GameSession.java`), so several independent colonies could coexist in one JVM (tests still use one). Each colony gets its **own economic `Rng`** (seeded from the session seed and the colony's index, so colonies don't interleave their random streams), but **shares** the session's `NameRegistry` and `Demography` — the shared name pool is what keeps dynasty surnames unique across every colony in the session. Colony 0 uses the bare seed, so a single-colony run is byte-identical to the old single-shared-generator design. Agents, banks and markets each hold a reference to the `Settlement` they belong to (injected through their constructors); printers receive it in `print(Settlement)`. Agent IDs are per-colony too — the `nextAvailableID` counter lives on `Settlement` (`agent.getID()` is drawn from `colony.nextAgentID()`), so two colonies have fully independent ID/account spaces. Collaborators register with the colony via `colony.addBank(...)` / `addMarket(...)` / `addAgent(...)` / `addPrinter(...)`. `colony.run(steps)` calls `colony.newDay()` repeatedly. `newDay()` first **recomputes the day's solar times** for the colony's location and current date (dawn/sunrise/sunset/dusk and the precalculated daylight duration — see *Daylight and the solar package* below), then defines the fixed phase order every tick (`settlement/Settlement.java`):

1. `agent.act()` for every alive agent — agents read last step's prices/income and **post buy/sell offers** to markets (they do not transact yet); a death is flagged via `die()`.
2. `bank.act()` for every registered bank — recomputes total loan/deposit, sets loan & deposit interest rates, pays/charges interest.
3. Remove the dead and **spawn replacements** — one successor household per dead agent, via the registered **replacement policies** (`Settlement.addReplacementPolicy`, a list tried in order until one returns non-null: a laborer policy and, in `AristocraticEconomy`, a noble policy). A dead agent that is a `Household` (laborer or noble) and gets **no** successor is an extinct dynasty, so its surname is **recycled** back into the session name pool (`names.releaseDynastyName(...)`); a successor keeps the surname, so it stays in use. In the bundled sims replacement always succeeds, so this never fires — naming stays byte-identical. Then run the registered **step actions** (`Settlement.addStepAction`, e.g. injecting external money into a bank's equity) and admit any **immigrants** (`Settlement.setImmigrationPolicy`, new households funded out of that equity). All default to no-ops.
4. `market.clear()` for every market — **this is where transactions actually settle**.
5. `printer.print()` for every printer.
6. `updateInflation()` (CPI = mean of consumer-good market prices over `INFLATION_TIME_WIN`).

The key mental model: **`act()` posts intentions against last step's state; `clear()` executes them.** Order of construction in a simulation matters because of this deferred settlement (the labor market is cleared once manually before `run()` so firms have workers in step 0).

### Markets clear by finding a price

Each `Market` (subclass of `eos/market/Market.java`) collects offers during the act phase and settles in `clear()`:
- `ConsumerGoodMarket` binary-searches for a price where demand ≈ supply (bounded to ±`zeta` of last price), then fills all buyers/sellers pro-rata and moves money via `Bank`. Buyers express willingness via a `Demand` strategy object (`market/Demand.java`), not a fixed quantity.
- `LaborMarket` shuffles employers/employees and allocates workers proportionally to each firm's wage budget, paying each worker an equal per-head wage (`wageBudget/headcount`) via `withdraw`+`credit`. The **labor each worker delivers is skill-scaled**, not 1: the firm's `Labor` good is increased by the worker's `productivity` (a skill-10 worker contributes 1, the legacy homogeneous case), so two workers at the same firm earn the same wage but supply different amounts of labor. A worker's output is then **scaled by the length of the day**: a laborer delivers 100% of its skill-scaled output at `FULL_OUTPUT_DAYLIGHT_HOURS` (8) hours of daylight, proportionally more on longer (summer) days and less on shorter (winter) ones — `productivity * (1 + sensitivity·(daylight/8 − 1))`. The `sensitivity` is itself penalized at high latitudes (`daylightSensitivity()`): full up to 45°, fading linearly to 0 at the poles, so the extreme polar daylight swings don't blow up the economy. **Only laborers** get this daylight scaling (it lives in `addEmployee(Laborer)`); nobles supplying labor to the strategic sector use the `addEmployee(int, Bank, double)` primitive and are unaffected. The coupling is calibration-sensitive: deep-winter labor *scarcity* (factor ≪ 1) is what destabilizes the colony, so the 8h reference — which keeps winter output near 100% — matters; at it, the standard colony survives up to ~58° latitude (`LatitudeSweep` finds the limit; a 10h reference drops it below London's 51.5°).
- `CapitalMarket` matches firms buying machines against `CFirm` sellers.

### Bank is the payment + credit system

`Bank` (`bank/Bank.java`) is an **instance** class (constructed with a `BankConfig`); each agent holds its accounts at a specific `Bank` (`agent.getBank()`, injected through every constructor). A bank holds one `Account` per agent keyed by the agent's ID (which doubles as account number). Every account has a `CHECKING` and `SAVINGS` balance; a **negative savings balance is a loan**. Money only moves between checking accounts. Payments are **agent-routed**: a transfer is `payerBank.withdraw(payerID, amt)` + `payeeBank.credit(payeeID, amt, purpose)`, which is correct even when the two parties bank at different `Bank` instances (there is no combined `pay` — split it). The payment-purpose codes `Bank.PRIIC`/`SECIC`/`OTHER` stay `static` (shared across banks). Each bank also carries a `CurrencyType` (`bank/CurrencyType.java`: `COPPER`/`SILVER`/`GOLD`, set via `BankConfig.currency`, default `COPPER`) — its denomination, so a colony can run banks in different currencies — the default hierarchy banks commoners in copper, nobles in silver, and the ruler in gold. The colony has a **fixed exchange rate** with copper as the base unit (prices are quoted in copper): silver = 100 copper, gold = 1200 copper (historically plausible 15th-century ratios — gold:silver ≈ 12:1, silver:copper ≈ 100:1). Because the rate is fixed and freely convertible the currencies are **fungible**, so all internal accounting stays in copper and the rate is used only to convert non-copper amounts and to **display** balances in a bank's own currency — `CurrencyType.toCopper`/`fromCopper`/`convert` (exposed as `Settlement.convert`), with `BankPrinter`/`NoblesPrinter`/`PersonsOfInterestPrinter` each carrying a `Currency` column and reporting monetary values in the holder's currency. At the fixed rate alone a run is byte-identical regardless of currency assignment (the copper-denominated figures are unchanged), so changing the *display* currency changes nothing. The dynamics only move once exchange carries **friction**: the `BankConfig.exchangeFeeRate` knob (default 0) makes a non-copper bank a **money-changer** that skims a fee whenever its client's money crosses the copper boundary. Because every price is quoted in copper, a non-copper account converts on *every* payment, so the fee is charged inside `Bank.withdraw`/`credit` exactly when `currency != COPPER` (the payer bears it on a withdrawal, the payee on a credit), retained as the bank's equity/profit; a copper bank is already in the base currency and never charges. The default silver and gold money-changers set it to `0.02` (`DEFAULT_EXCHANGE_FEE_RATE`), so a silver noble loses 2% converting each dividend in and each purchase out, and the gold-banking ruler loses 2% whenever it buys (copper-quoted) enjoyment — both banks profit from the fee they retain. This is the conversion friction a *fixed* rate previously could not provide (a genuinely *floating* rate would be a further, larger step). This tiered scheme is the **default banking system** every settlement uses, provided by `SimulationHarness`: `getCopperBank()` (the zero-profit base-currency bank for commoners — laborers and firms), `getSilverBank()` and `getGoldBank()` (the nobles' and ruler's money-changers, each charging `DEFAULT_EXCHANGE_FEE_RATE` = 2%). Each is created **lazily on first request** and shared thereafter, so a colony carries only the tiers its classes use. Because every standard settlement now has a default ruler, every standard run populates **gold** (the ruler), and the runs whose nobles bank in silver (`AristocraticEconomy`, `StrategicEconomy`, `HanseaticEconomy`) populate **silver** too; `HomogeneousEconomy`/`HeterogeneousEconomy` bank their default nobles at the single copper bank, so those run copper + gold. (Request copper before silver/gold so the banks are numbered/ordered copper, silver, gold.) The bare exceptions that skip the ruler (and the export sector) are the analytical sweeps `ScaleSweep`/`LatitudeSweep` (one copper bank) and `SmallOpenEconomy` (two *copper* banks, agents split across them as in `TwoBankEconomy` — which itself adds the ruler's gold bank on top of its two copper banks). Critically, agents read their last-step income from public `Account` fields — `priIC` (primary income: wage/sales), `secIC` (secondary: dividends), `interest` — and **reset these to 0 at the end of their own `act()`**. Loan interest rate responds to the loan/deposit gap; deposit rate redistributes collected interest to creditors. By default the bank is a **zero-profit pass-through** (`depositIR · totalDeposit == loanIR · totalLoan`). Three `BankConfig` knobs let it retain profit into a cumulative `equity` field (charted as the `Equity` column in `BankPrinter`): `spread` keeps a fraction of the interest margin instead of paying it to depositors, `feeRate` charges the payer a per-withdrawal transaction fee, and `exchangeFeeRate` charges the currency-exchange fee described above (only on a non-copper bank). All default to 0 (so default runs are byte-identical). That retained profit is **distributable to a noble owner**: the bank separately tracks `cumulativeProfit` (gross spread + fees ever retained) minus `distributedProfit` paid out, exposed as `getDistributableProfit()`, and `payDividend(amt)` skims that profit out of equity. This is deliberately *only* the profit slice of equity — not the estates-in-transit or injected external funds that also pass through equity — so a noble can draw a bank dividend without disturbing inheritance or the open-colony money buffer. Both fields are passive (and `payDividend` is only called by a noble owner), so a run with no bank-owning noble is byte-identical. **Caveat:** the zero-profit pass-through is load-bearing for the model's calibrated stability — empirically `spread ≈ 0.005` is safe and profitable, but `spread ≈ 0.02` already starves the laborer population via a price spike; making the bank meaningfully profitable needs the firm/laborer parameters retuned.

The `equity` field is also the bank's link to **inheritance** and the **open colony**. A dead account holder's estate is folded into equity (`inheritAndClose`) and immediately handed to its successor household (`openInheritedAcct`, which draws the inherited balances back out of equity) — so money stays in circulation, and in a closed run these net to zero. `injectExternalFunds(amount)` adds genuinely new money from outside the colony into equity; it is the entry point for an open colony, where accumulated equity bankrolls new immigrant households (each opened via `openInheritedAcct`, conserving every external dollar as either circulating money or equity below the immigration threshold).

### Agents

`Agent` (`agent/Agent.java`) is the abstract base: unique ID, alive flag, `act()`, and `getGood(name)`. Two families:

- **Laborers** (`agent/laborer/Laborer.java`) earn wages, decide consumption vs. savings (target-savings model sensitive to the real interest rate), buy Necessity and Enjoyment, eat one unit of Necessity per step, and `die()` if they can't eat. They steer their consumption budget toward Necessity in proportion to how far their stock sits below the **target necessity stock** (`colony.getTargetNStock()`, default 26 units ≈ a 26-day buffer), spending the surplus on Enjoyment; that target is a property of the **colony** (a colony-wide environmental constant), not a per-laborer `LaborerConfig` value. Each household also has an integer **skill** in `[0, 20]` (lifted to the `eos.agent.Household` interface, so nobles carry it too — see below) that fixes a laborer's **labor productivity** for life via the quadratic curve `Household.productivityOf` = `max(0.01, (skill/10)²)` (skill 10 → 1, the legacy homogeneous unit; skill 0 → 0.01; skill 20 → 4). Skill is drawn at birth from a Gaussian centered on the **colony's mean skill** (`colony.getMeanSkill()`, a colony-start property, default 5; sd 3, clamped) on the demographic **skill RNG** — a stream separate from both the economic and the mortality streams — and is **re-rolled for every head**, including each successor (so it is not inherited across generations). Because the default center (5) sits well below the unit-productivity skill (10), the typical worker supplies well under one unit of labor, so skill **lowers aggregate effective labor** (`E[productivity] ≈ 0.3`) versus the old "everyone = 1" world; founding a colony with a higher `meanSkill` raises it. Each laborer is a **household** represented by a `Person head` (a male given name + unique dynasty surname from `colony.getNames()`, see `name/` below); the death log reports `head.fullName()`. The head carries a `birthDate` (the source of truth for its age — the span from `birthDate` to the in-game date) and may also `die()` of old age each step against the `mortality/` life table; founding heads draw a working-age birth date from a Gaussian centered on `colony.getMeanInitAgeYears()`. Distinct from that, the household records a `foundingDate` — the in-game date it came into being, the moment its already-grown head *arrived in the colony and founded it* (a working-age adult, born years earlier, sets up the household now). So `birthDate` is the head's biological birth; `foundingDate` is the household's creation. The shared constructor draws **skill before the head**, so the head's given name is picked at a rarity that tracks skill (abler households get rarer, more distinctive given names; the surname is still the plain unique-dynasty draw — see `name/`). Reordering skill ahead of the name leaves skill values and surnames byte-identical (separate RNGs, same draw order) — only given names shift. Any death folds the estate into the bank's equity, and a **successor household** of the same dynasty inherits it (wired via `Settlement.addReplacementPolicy`) so money and the labor force stay stable. A separate **equity-funded constructor** creates a fresh new-dynasty household for open-colony immigration. Mortality is a core, always-on feature: every head ages and the demographic draw is taken on a separate salted RNG, so it never perturbs the economic random stream (see `mortality/` below).
- **Firms** (`agent/firm/`): `CFirm` produces Capital (machines); `ConsumerGoodFirm` (abstract) is subclassed by `EFirm` (Enjoyment) and `NFirm` (Necessity), which combine labor + capital via a **Cobb-Douglas production function** `A·L^β·K^(1-β)`. Consumer firms adjust output by marginal profit and decide whether to buy/replace capital based on return-on-capital vs. interest rate and capacity utilization. The **wage budget** follows one of two rules (`FirmConfig`): the legacy cash-flow-gap nudge (`laborShare == 0`), or — since the labor market pays a single wage `totalBudget/N` — a **labor-share** rule (`laborShare > 0`) that budgets a fixed fraction of revenue for wages, so total wage spending scales with the colony and wages aren't outrun by a growing labor pool. `SimulationConfig.laborShare` defaults to 0.5, so the default run uses the labor-share rule (a stable but nominally-inflating closed colony).

- **Nobles** (`agent/noble/Noble.java`) are the **owners** of firms **and banks**: rentier households that live off profit, not wages. A noble is a named household exactly like a laborer — its `Person head` is drawn with `colony.getNames().nextHead()` (unique dynasty surname) — but it never enters the labor market. Each step it *pulls* a dividend from every owned firm (`dividendRate · max(0, firm.getProfit())`) and every owned bank (`dividendRate · bank.getDistributableProfit()`), moving the surplus to the noble through the bank's **secondary-income channel** (`Bank.SECIC` / `Account.secIC`) — the dividend pathway that the model had wired but never fired until nobles existed. It then spends a fraction of its liquid wealth back into the consumer-good markets (so dividends circulate) and saves the rest. This is the "option A" ownership design: nobles influence the labor market only **indirectly**, via consumption → firm revenue → the labor-share wage budget. The dividend draws only on public firm getters and the bank's existing transfer primitives, so **no firm/bank code changed** and a run with no nobles is byte-identical to before. Like laborers, nobles **age and die** (a `birthDate` sampled the same way, old-age death against the life table), and a **successor of the same dynasty inherits the estate and the holdings** (its `firms` and `banks` lists, via a noble `addReplacementPolicy` rule) — so the aristocracy persists across generations and keeps drawing dividends. A noble is a `Household`, so it also carries a **skill** (drawn around the colony mean exactly as a laborer's, re-rolled each generation); it is unused today (nobles sell no labor) and reserved for later use. Nobles have no starvation death (they are wealthy rentiers), only old age. Parameters live in `NobleConfig` (`dividendRate`, `consumptionRate`, `necessityShare`). `AristocraticEconomy` is the worked example (two nobles split ownership of all firms); `NoblesPrinter` writes the `Nobles.csv` time-series (with an `AvgAge` column).

- **Ruler** (`agent/ruler/Ruler.java`) is the settlement's single sovereign: the owner of its **gold** bank and its sole client (the bank has no others). Unlike a `Noble` it draws no dividends — it is a **treasury that indulges**: a named `Household` that ages, dies and is succeeded by a same-dynasty heir (and is tracked as a person of interest), earns nothing, but each step spends a small fraction of its fortune (`consumptionRate`, a constructor param; `SimulationHarness.DEFAULT_RULER_CONSUMPTION_RATE` = 0.0002) on **enjoyment**, posting a buy offer to the (copper-quoted) enjoyment market exactly as a noble does. Its opening fortune (`SimulationHarness.DEFAULT_RULER_GOLD` = 10 gold, held in copper internally) is therefore drawn down gradually over the run (≈1.7 gold left after 24 years at the default rate); it never buys necessity and never starves. **Every standard settlement has a ruler**, created by `SimulationHarness.createDefaultRuler()` (which also lazily creates the gold bank) — like the export sector it is part of the default setup, skipped only by the sweeps and `SmallOpenEconomy`. Because enjoyment is priced in copper, the ruler's spending converts **gold → copper**, firing the gold bank's `exchangeFeeRate` — the only thing that ever makes the otherwise-idle gold bank transact (so the gold bank, too, turns an FX profit). It is deliberately a distinct type rather than a `Noble`: `NobleConfig`'s 5%/step consumption would drain the treasury in months, and the ruler is a tier above the nobles, so this also keeps it out of `Nobles.csv`. The two reporting touchpoints — `PersonsOfInterestPrinter` (a `Ruler` row, gold-denominated, income always 0) and the per-POI death log in `Settlement.newDay` — carry a `Ruler` case. In a sim the ruler is created **last** so its demographic draws don't perturb the commoners' or nobles'. The class hierarchy thus mirrors the currency hierarchy: commoners → copper, nobles → silver, ruler → gold.

Note: `ConsumerGoodFirm.act()` contains explicitly-labeled **"hacks"** — capital-purchase nudges that only fire after a warm-up period (`timeStep > config.warmupDays()`, default 2000) once the rolling average profit is meaningful, spurring an extra machine on a profit spike (`profit > capitalNudgeProfitFactor · avgProfit`, with the utilization bar lowered to `capitalNudgeUtilThreshold`) or one fewer on a deep loss. These knobs were lifted into `FirmConfig` (defaults preserve the original `2000`/`5`/`0.8` behaviour exactly) but the block is still model **calibration tuning stability**, not incidental logic — changing the defaults needs re-validation against the test invariants and the `ScaleSweep`.

### Goods, printers, utils

- `good/` — `Good` is a mutable quantity holder; `Capital`, `Labor`, `Necessity`, `Enjoyment` are the concrete goods.
- `io/printer/` — each `Printer` writes one CSV via `CSVPrintWriter`. Register printers with `Settlement.addPrinter(...)` (which writes the header row immediately) and finalize with `Settlement.cleanUpPrinters()` after the run. **Cadence is monthly**: the base `Printer.shouldPrint(colony)` gates every row on `getDate().getDayOfMonth() == 1` (within the printer's `[start, end]` step bounds) — there is no per-step `period`/`stepSize` knob. **Number formatting is centralized in `CSVPrintWriter`**: any `Double`/`Float` cell is rendered to two decimals (`%.2f`); ints, dates and already-formatted `String` cells pass through untouched. `BankPrinter` pre-formats its four interest-rate columns and the inflation column as truncated percents (e.g. `0.45%`) via its `formatPercent` helper, so those reach the writer as strings. `LaborersPrinter` aggregates over the **living** laborer population read from `colony.getAgents()` each step (so it tracks replacements and immigrants, not a fixed initial array), averaging over the live count, with `AvgAge` and `Count` columns. `NoblesPrinter` does the same for the noble population (dividends, income, consumption, wealth).
- `io/SimLog.java` — event logging via `java.util.logging`. `SimLog.init()` (called first thing in each `main`) installs a formatter that prefixes every record with the current **in-game date** (`Settlement.getDate()`, e.g. `1452-03-14: WARN …`), writing to **stderr** with per-record flush. Classes that emit events are annotated with Lombok `@Log` and call `log.info(...)` for events or `log.warning(...)` for anomalies (e.g. a price crossing `PRICE_SKYROCKET_FACTOR`×its initial level in `ConsumerGoodMarket`). Logged events include the **colony being founded** (`Settlement.start`) and **dying** (`updateLifecycle`, when its last laborer is gone), and a **notable person** — any household created with skill above `Household.NOTABLE_SKILL` (15) — recorded by name at creation (rare at the default `meanSkill` 5; commoner in a higher-skill colony). The colony tracks a roster of **persons of interest** — every `Noble` and every notable household — to which households register themselves at creation (`Settlement.addPersonOfInterest`, exposed via `getPersonsOfInterest()`) and from which they are removed when they die. **Death logging is confined to that roster**: only a person of interest's death is logged, exactly once (in `Settlement.newDay`'s dead-agent sweep), so ordinary laborer deaths produce no log spam. The roster's yearly *listing* is a CSV, not a log: `PersonsOfInterestPrinter` writes one row per living person of interest on the first day of each year (columns `Date, Name, Type, Skill, Age, Income, Wealth`); it is registered by the noble-bearing sims (`AristocraticEconomy`, `StrategicEconomy`). This is for discrete events — bulk time-series still goes through printers to CSV. Note `Settlement.run`'s progress counter prints the in-game date to stdout every January 1st (once per in-game year), so progress and events are on separate streams.
- The in-game date is the canonical display unit: every place a step was shown as an integer now shows `Settlement.getDate()` — the SimLog prefix, the progress counter, and the first column of every printer CSV (header `Date`, ISO `yyyy-MM-dd`). `Settlement` holds the start date (set from `cfg.startDate()` by `SimulationHarness`); `getDate()` returns `startDate + timeStep` days. The integer `timeStep` is still used internally for control flow (`getTimeStep() == 0`, the `> 2000` calibration hacks, the printers' `[start, end]` step bounds). Printers emit one row on the **first day of each in-game month** (`Printer.shouldPrint` gates on `getDate().getDayOfMonth() == 1`); there is no longer a `period`/`stepSize` cadence knob.
- `util/` — `Rng` (minimal seedable RNG over `java.util.Random`; **`GameSession` mints one per colony** — derived from the session seed and the colony's index — and the colony exposes it via `colony.getRng()`, so colonies in one session have independent, reproducible economic streams; seed the session for reproducible runs) and `Averager` (rolling window average used for inflation and long-term rates).
- `name/` — `NameRegistry` holds the **complete name sets** for a session, loaded from the tiered JSON tables in `src/main/resources` (`dynasty-human.json`, `male-human.json`, `female-human.json`) via Jackson (`jackson-databind`, the one runtime dependency beyond Lombok). Each tier owns a cumulative-weight band `(prev, percent]`, so `NameTable.pick(rng)` draws weighted toward common names. The names are stored **common→rare**, and `total` ≈ 99 acts as a percentile scale, so `NameTable.pickAtRarity(rng, percentile)` biases the draw toward a chosen slice of the rarity range (0 = commonest, 1 = rarest, jittered ±0.125 for variety) — used to give **abler households more distinctive names**. The `GameSession` owns the registry and drives it from a **separate naming RNG** (a salted copy of the seed) so naming is deterministic but never perturbs the economic random stream — economic output stays byte-identical. Dynasty surnames are drawn **without replacement** so every household *in use* across the session — in any colony — has a unique surname; given names are drawn **with replacement**. The pool is finite, so an extinct dynasty's surname is **recycled**: `releaseDynastyName(surname)` returns it (with its original weight) to the drawable pool when a household dies with no successor, so the uniqueness invariant is over *living* households, not permanent consumption. The registry tracks the `inUse` set for this and guards against releasing a surname that isn't out. `nextHead()` returns a `Person` (male given name + unique dynasty surname); `nextHeadInDynasty(surname)` draws a fresh head keeping an existing surname (a successor household, which therefore does **not** consume or release a surname). The **rarity-aware overloads** `nextHead(nameRarity)` / `nextHeadInDynasty(surname, nameRarity)` draw the given name via `pickAtRarity` instead — the household passes `skill / Household.MAX_SKILL`, so a household's given-name rarity tracks its skill (surnames are unaffected, still the unique-dynasty draw). Households are tagged with the `eos.agent.Household` interface (`getHead()`, `getSkill()`, the `MIN_SKILL`/`MAX_SKILL`/`NOTABLE_SKILL` constants, the static `productivityOf(skill)` curve, and `isNotable()`) so the step loop can recycle the surname of one that dies unreplaced and so both laborers and nobles share the skill and notability concepts.
- `mortality/` — `LifeTable` is an age-specific mortality schedule (the bundled `WEST_LEVEL_3` is Coale-Demeny West Level 3, e0 = 25; `nqx` per age group → a per-day death probability). `Demography` (owned by `GameSession`) is the per-session service for the random draws a household makes **at birth**: it draws founding household ages from a Gaussian truncated to working age and decides each step whether a head dies of old age (on its salted **mortality RNG**), and it draws a household's **skill** (`sampleSkill(meanSkill)`, Gaussian centered on the colony's mean skill, clamped to `[0, 20]`) on a **second, separate skill RNG** so skill draws don't perturb the mortality stream either. Both RNGs are salted copies of the session seed, separate from the economic one, so all these draws are deterministic per seed yet never perturb the economic random stream. Mortality is a core always-on feature (no on/off toggle).

### Daylight and the solar package

- `solar/` — a vendored sunrise/sunset calculator (the Lucky Cat Labs / `ca.rmen` algorithm, ported from `BigDecimal` to `double`). `SolarEventCalculator.computeSunriseTime/computeSunsetTime` turn a `GeoLocation` (a `(latitude, longitude)` record — the project-local replacement for the original's external `Node`), a `Zenith` (the solar declination: `OFFICIAL` 90.83°, `CIVIL` 96°, `NAUTICAL` 102°, `ASTRONOMICAL` 108°) and a `java.util.Calendar` date into a local clock time. It is **legacy `Calendar`-based**, so `Settlement` bridges from its `java.time.LocalDate`. The calculator **throws** `UnsupportedOperationException` when an event does not occur (the sun never reaches the zenith — e.g. no astronomical twilight at high latitude in midsummer); callers must catch this. (Note: the port also fixed an inverted range-normalization bug in `getSunTrueLongitude`.)
- The colony has a fixed **geographic location** — `latitude`/`longitude` in decimal degrees (north/east positive), colony-start properties threaded from `SimulationConfig` through `GameSession.newSettlement` into `Settlement` exactly like `meanSkill`/`targetNStock` (default London, 51.5074, −0.1278). From these, `Settlement` **computes and stores the day's solar times** at the top of every `newDay` (and seeds them in the constructor for step 0): `getDawn()`/`getSunrise()`/`getSunset()`/`getDusk()` (UTC `LocalTime`; **dawn/dusk are the astronomical** sunrise/sunset, sunrise/sunset the **official** ones) and `getDaylightHours()` (the precalculated sunrise→sunset span). An undefined event is stored as `null` (and `daylightHours` as `NaN`) rather than throwing — e.g. dawn/dusk are null at London around midsummer, when astronomical twilight lasts all night. These feed the daylight-scaled labor output (see `LaborMarket` above).

## Conventions

- Reproducibility hinges on the single seed set at the top of `main`; the same seed yields identical runs.
- Display names & `toString()`: `Agent.getName()` returns a display name that defaults to the class simple name; subclasses override it via `setName(...)` (firms set "Capital Firm" / "<Product> Firm"). Markets and banks (not agents) carry their own default `name` — `good + " Market"` and `"Bank " + colony.nextBankNumber()` (numbered per colony). Laborers, firms, banks and markets override `toString()` with a concise, side-effect-free debug summary (no bank lookups, so they're safe to print on dead agents).
- When adding a new market or good, register the market with `Settlement.addMarket(...)` **before** constructing agents that reference it — agent constructors look markets up by good name (`Settlement.getMarket(name)`).
- Behavioral/model parameters of an agent or bank live in an immutable `*Config` record (`LaborerConfig`, `FirmConfig`, `BankConfig`) with a `DEFAULT` constant holding the canonical values; the owner stores it as a `final` field and the constructor takes it as a required param. This makes parameters per-instance (enabling heterogeneous populations / multiple banks) and tunable per run from the simulation's `main`. Structural/non-tunable constants — capital lifetime (`CFirm.CAPITAL_LIFE`), market price-band width (`ConsumerGoodMarket.zeta`), the inflation window on `Settlement` (`INFLATION_TIME_WIN`), payment-purpose index constants (`Bank.PRIIC`/`SECIC`/`OTHER`) — stay as `static final` on the relevant class. `Settlement` keeps its constants inline (`INFLATION_TIME_WIN` stays `static final`; the per-run start date, founding-age mean, target necessity stock and **mean skill** are constructor args supplied by `GameSession` via `newSettlement`). The target necessity stock and the mean skill live on `Settlement` rather than `LaborerConfig` precisely because they are **colony-start properties**, fixed for the colony's life and uniform across its households (they cannot vary across laborers within one colony). The colony also has a clear **lifecycle**: `start()` (called at the top of `run()`) marks it founded, and it **dies** terminally the step its last laborer is gone — `newDay` detects this (`updateLifecycle`), recording `getDeathDate()` and logging it; query via `isAlive()` / `isDead()`. Mortality is always on — there is no toggle. The `ConsumerGoodFirm.act()` "hacks" calibration numbers now live in `FirmConfig` too (`warmupDays`, `capitalNudgeProfitFactor`, `capitalNudgeUtilThreshold`), so they are named and tunable per run rather than inline magic numbers, with defaults that keep the canonical run byte-identical.
- Run-level parameters live in `SimulationConfig` — an immutable record of scalars plus nested sub-records (`PriceRange`, `FirmInit`, `CFirmInit`, `LaborerInit`) with a `DEFAULT`. Besides the calendar, population counts, market price bounds and per-agent-type initial state, it carries the demographic and macro knobs: `meanInitAgeYears` (founding-age mean), `targetNStock` (the necessity stock every laborer targets, default 26), `meanSkill` (the colony's mean household skill, default 5) and the colony's `latitude`/`longitude` (default London 51.5074, −0.1278, driving the daylight-scaled labor output) — all passed into the `Settlement` constructor — `externalInflowPerStep` + `immigrationThreshold` (the open-colony money inflow and the equity level that funds each immigrant; inflow 0 = closed), and `laborShare` (the firms' wage-budget rule). The harness reads these and wires them: `enableExternalInflow(gatewayBank)` registers the inflow step-action + immigration policy (no-op when inflow is 0), and the consumer firms' `FirmConfig` is seeded with `laborShare` (overridable via `setFirmConfig`). **Each step is one in-game day**: the config holds a `startDate` (`LocalDate`, default 1444-12-11) and `durationYears` (default 25), and derives `numStep()` = days from `startDate` to `endDate()` (9131 for the default 25-year run). `dateAt(step)` maps a step back to its in-game date (step 0 == `startDate`). Each simulation's `main` binds `SimulationConfig cfg = SimulationConfig.DEFAULT` and reads values via accessors (`cfg.eFirm().checking()`); only the per-run differences stay in `main` — the **seed** and the **init logic** (`HomogeneousEconomy` fixed vs. `HeterogeneousEconomy` randomizing around the config values). The simulations are entry points with no constructor, so the config is a `main`-local bound to `DEFAULT` rather than a constructor param.
- The config records carry Lombok `@Builder(toBuilder = true)`. The hand-written `DEFAULT` constants remain the source of truth (no `@Builder.Default`); the builder exists so a run can derive a variant cheaply — `SimulationConfig.DEFAULT.toBuilder().durationYears(10).build()` — and so multi-field value records can be constructed by name instead of positionally. Builders are additive; nothing in the default run path calls them. Lombok is already a `provided` dependency (`@Getter` is used on agents); `@Builder` on records needs Lombok ≥ 1.18.20 (project is on 1.18.34).
