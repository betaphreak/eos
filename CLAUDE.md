# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`eos` is an agent-based macroeconomic simulation written in plain Java. It models a colony (closed by default, but optionally **opened** to a stream of external money that bankrolls immigration) of laborers, firms (capital, enjoyment, necessity), a bank (one by default, though the design now allows several), and markets that clear by supply/demand each time step. Each settlement also occupies a disc of a given **size** with a fixed number of build **slots** that hold its firms (and, later, housing and other buildings). Laborer households age and die on a real mortality schedule. Running a simulation produces CSV time-series files for analysis (prices, volumes, firm finances, bank rates, laborer state).

The root package is `eos`, under the standard Maven layout at `src/main/java/eos/` (e.g. `src/main/java/eos/agent/firm/CFirm.java` is `firm.agent.com.civstudio.CFirm`).

## Build & run

Maven project, Java 21 (`pom.xml`). Tests use JUnit 5 (`mvn test`).

```powershell
mvn clean compile          # compile to target/classes
mvn exec:exec              # run HomogeneousEconomy (forks a JVM with -ea)
mvn exec:exec -Dsim.main=simulation.com.civstudio.SmallOpenEconomy   # run another entry point
mvn package                # build the jar in target/
mvn test                   # run the JUnit 5 suite (each Simulation runs full as a smoke test)
```

`exec:exec` runs in a **forked JVM with assertions enabled** (`-ea`) because the code uses `assert` as real invariant checks; the main class is the `sim.main` property (default `simulation.com.civstudio.HomogeneousEconomy`). **Every standard settlement carries a default _export sector_ and a default _ruler_, is founded with one enjoyment and one necessity firm, and grows its consumer sectors by _dynamic firm provisioning_.** The export sector is a single `StrategicFirm` (`SimulationHarness.createDefaultStrategicSector`); its export earnings accrue to the equity of the bank holding its account, so that bank is not a zero-profit intermediary. **No nobles are created at founding** — the export aristocracy is **raised from the laborers by ennoblement** up to `cfg.targetNobles()` (default 5) over the first weeks, and the **ruler itself works the strategic firm** every step so it is never unstaffed during that ramp (see *Agents → Ruler* and *Markets → Ennoblement*). The ruler (`SimulationHarness.createDefaultRuler()`) banks in **gold**, spends a little on enjoyment, **taxes** the colony's banks and nobles into its treasury, works the export firm, and each month **charters or dissolves consumer-good firms** so their count tracks demand (see *Markets → Dynamic firm provisioning*); its gold transactions fire the gold bank's FX fee. `SimulationConfig.DEFAULT` now founds with `numEFirms = numNFirms = 1`; `createDefaultRuler` installs the provisioning factory automatically, so every ruler-bearing colony grows from that seed. Both are skipped only by `SmallOpenEconomy`, left bare so it keeps measuring what it was built for. The entry points (README carries the summary table; the mechanics each exercises are detailed in *Architecture* below):

- `HomogeneousEconomy` — homogeneous agents, single copper bank; the default run, founded via `SimulationHarness.foundStandardColony`.
- `SmallOpenEconomy` — opened to external inflow + immigration, growing past its starting 90 laborers (2E + 2N + 1C, two copper banks); bare, with no export sector or ruler.
- `HanseaticEconomy` — two colonies founded into **two adjacent world-map provinces** (Withacen and Hopespeak, direct neighbours at ~55°N — see *Daylight and the solar package* / `docs/geography.md`) in one `GameSession`, **run concurrently (a thread each) in lockstep** via `SessionRunner`: independent economies with disjoint surname slices (so surnames stay unique across both) and per-colony demography; 20 laborers, 5 nobles, **founded with 1E + 1N** (grown dynamically, each capped by its province's plots). Both colonies (like every ruler-bearing colony) found and replace their labor force from a peasant pool, so each **collapses** once the reserve drains (the two differ by their independent RNG streams). `run()` returns the first (Withacen). `HanseaticEconomyTest` is its concurrent smoke test (both threads run clean to dissolution) and asserts the two founding provinces are adjacent (`WorldMap.path` connects them in one step — the substrate caravan trade will route over).

**The peasant pool is the universal labor model.** Every ruler-bearing colony seeds a `Retinue` of `cfg.retinueSize()` peasants (default 900; and, so it can grow, a default `BuilderFirm` staffed from that pool); on day 0 the ruler promotes the ablest `round(promotionRatio · poolSize)` of them into laborer households (each endowed with the **sum of its 12 skill levels** in copper, funded from the ruler's treasury, and carrying its larder ration out of the pool), the rest staying as the standing reserve. Dead laborers are replaced the same way from the reserve. The reserve peasants live on a **low relief ration** the ruler funds, bought price-sensitively (so the extra mouths don't starve the workforce out of the food market — see *Agents → Peasant pool*); the reserve persists a couple of years as a genuine replacement pool, then — with no inflow yet — drains and the colony **spirals to collapse** (the default scale lasts ~5–7 years; with dynamic firm provisioning keeping food supply ahead of demand, even the tiny `HanseaticEconomy` colonies now last ~6 years rather than the months they collapsed in under a fixed minimum-firm split). Only the bare `SmallOpenEconomy` — which has no ruler/pool — keeps direct founding + same-dynasty succession and stays stable. People generated into the pool also roll a random **gender** (see *Agents → Laborers/skills*). See *Agents → Peasant pool*.

Each scenario is a `static run()` that builds and runs the colony and returns a `SimulationHarness`, plus a `main()` that calls it. Each `run()` creates a `GameSession` (which holds the seed and its own `Rng`) and from it a `Settlement` (`session.newSettlement(cfg.settlementName(), cfg.startDate(), cfg.meanInitAgeYears(), …)` — colonies are **named**, exposed via `Settlement.getName()`), then hands that colony to the harness. **`SimulationHarness` consolidates the shared construction** (markets, banks, firm/laborer loops, printers, run/cleanup) into a single `Settlement` instance; a `Simulation` supplies only what differs — which bank each agent uses (`IntFunction<Bank>`), and how each agent's initial state is drawn (`IntToDoubleFunction`). The call order in the harness mirrors the original sims and matters (deferred settlement + reproducible RNG consumption). A run is configured through `SimulationConfig.DEFAULT`.

Tests (`src/test/java`, JUnit 5) treat each simulation as a smoke test: `Simulation*Test` calls `Simulation*.run()` for a full ~10k-step run and asserts invariants on the returned harness. Because every ruler-bearing colony now founds and replaces its labor force from a finite peasant pool, those runs **collapse** (`SimulationAssertions.assertCollapsed`: the colony ran to completion without tripping an `-ea` invariant and ended with its labor force gone); `ClosedColonySmokeTest` checks the closed default colony (`HomogeneousEconomy`) reaches that collapse with the expected bank count (the banks persist past the colony's death). ## Architecture

### The step loop is the spine

`Settlement` is an **instance** (not a static singleton): a `GameSession` owns the seed and creates colonies from it (`settlement/GameSession.java`), so several independent colonies can coexist in one JVM and **run concurrently — one thread per settlement** — coordinated by `SessionRunner` (`simulation/SessionRunner.java`), which advances them in **lockstep**: each colony runs one `newDay()`, then they all rendezvous at a `Phaser` day-barrier before the next day (a colony that dies early deregisters, so uneven lifespans don't deadlock it). Each colony gets its **own economic `Rng`** (seeded from the session seed and the colony's index, so colonies don't interleave their random streams) and — so the threads share no mutable per-step state — its **own `NameRegistry` and `Demography`** too: the session owns the shared *immutable* given-name tables and a single `DynastyPool` of surnames, but hands each colony a **disjoint surname slice** (drawn/recycled lock-free within the colony on its own naming `Rng`) and its own mortality/skill generators. Disjoint slices keep dynasty surnames unique across every colony in the session without a shared registry; colony 0 reuses the bare salted seeds for its economic/mortality/skill streams, so a lone colony's economics are unchanged by the per-colony scheme (only which cosmetic surnames land where shifts, since the pool is now shuffled and sliced). The residual session-level shared state (`DynastyPool` dispensing, the `caravans` list, the lazy `TechTree`) is `synchronized`; the `SimLog` handler is process-global but its date source is **per-thread** (a `ThreadLocal` colony), so concurrent colonies log their own dates without cross-talk. Per-colony runs stay **reproducible** (byte-identical output across concurrent runs). Agents, banks and markets each hold a reference to the `Settlement` they belong to (injected through their constructors); printers receive it in `print(Settlement)`. Agent IDs are per-colony too — the `nextAvailableID` counter lives on `Settlement` (`agent.getID()` is drawn from `colony.nextAgentID()`), so two colonies have fully independent ID/account spaces. Collaborators register with the colony via `colony.addBank(...)` / `addMarket(...)` / `addAgent(...)` / `addPrinter(...)`. `colony.run(steps)` calls `colony.newDay()` repeatedly. `newDay()` first **recomputes the day's solar times** for the colony's location and current date (dawn/sunrise/sunset/dusk and the precalculated daylight duration — see *Daylight and the solar package* below), then defines the fixed phase order every tick (`settlement/Settlement.java`):

1. `agent.act()` for every alive agent — agents read last step's prices/income and **post buy/sell offers** to markets (they do not transact yet); a death is flagged via `die()`.
2. `bank.act()` for every registered bank — recomputes total loan/deposit, sets loan & deposit interest rates, pays/charges interest.
3. Remove the dead and **spawn replacements** — one successor household per dead agent, via the registered **replacement policies** (`Settlement.addReplacementPolicy`, a list tried in order until one returns non-null). The colony registers a **built-in household-succession policy** in its constructor that asks each dead `Household` for its own heir (`Household.successor(colony)`): a `Noble` returns a same-dynasty heir inheriting its firms and banks (reusing the noble's own `NobleConfig`), a `Ruler` an heir that also takes over the colony's ruler reference — so noble and ruler succession needs **no per-simulation wiring** (a new self-replacing population type is likewise succeeded automatically). A `Laborer`'s `successor` returns null (its heir needs colony-seeding constants the colony doesn't hold), so the harness registers the laborer replacement policy: for a ruler-bearing colony it **promotes the ablest peasant** out of the pool into a new laborer household (see *Peasant pool*), and for a bare pool-less colony it makes a same-dynasty heir. A dead agent that is a `Household` and gets **no** successor is an extinct dynasty, so its surname is **recycled** back into the session name pool (`names.releaseDynastyName(...)`). This **does** fire in the ruler colonies: once the pool reserve drains, promotion returns no replacement, dead laborers go unreplaced, their surnames recycle, and the colony collapses — the bare colonies' same-dynasty replacement always succeeds, so there it never fires. Then run the registered **step actions** (`Settlement.addStepAction`, e.g. injecting external money into a bank's equity) and admit any **immigrants** (`Settlement.setImmigrationPolicy`, new households funded out of that equity). All default to no-ops.
4. `market.clear()` for every market — **this is where transactions actually settle**.
5. `printer.print()` for every printer.
6. `updateInflation()` (CPI = mean of consumer-good market prices over `INFLATION_TIME_WIN`).

The key mental model: **`act()` posts intentions against last step's state; `clear()` executes them.** Order of construction in a simulation matters because of this deferred settlement (the labor market is cleared once manually before `run()` so firms have workers in step 0).

### Settlement size and build slots

A `Settlement` is a **disc of radius `size`** with a fixed number of **build slots** (a precalculated `SlotTable` geometry shared per `GameSession`): `effective` slots hold firms (and later housing), and out-of-band **special sites** hold civic buildings. A colony is **founded at `SlotTable.MIN_SIZE` (3 → 15 effective slots)** and **grows to fit its occupants** — at founding by genesis-sizing, and once live only through its `BuilderFirm` (which raises rings of `LAND`/`ROAD`/`WALL` `BuildProject`s, billed to firm/ruler sponsors). Slots take a `SlotOccupant` (the occupant interface `Agent` implements); placement moves no money and consumes no randomness. **Full detail: [`docs/settlement-slots.md`](docs/settlement-slots.md)** (the runtime-founding design that builds on it is in `docs/village-founding.md`).

### Markets clear by finding a price

Each `Market` (subclass of `eos/market/Market.java`) collects offers during the act phase and settles in `clear()`:
- `ConsumerGoodMarket` binary-searches for a price where demand ≈ supply (bounded to ±`zeta` of last price), then fills all buyers/sellers pro-rata and moves money via `Bank`. Buyers express willingness via a `Demand` strategy object (`market/Demand.java`), not a fixed quantity. It also records a **demand signal** for the ruler's dynamic firm provisioning: each clear it computes the fraction of demand left unfilled for want of supply and exposes both the short-run smoothed level (`getSmoothedUnmetFraction()`, 30-day) and the **pressure** `getUnmetPressure()` = short-run minus a 365-day baseline. Reading the *relative* pressure cancels the chronic bias of the ±`zeta` price band (which, in a nominally-inflating colony, leaves the sticky price perpetually lagging demand so the absolute unmet level is always positive), so only a genuine *rise* above the sector's own norm registers.
- `LaborMarket` shuffles employers/employees and allocates workers proportionally to each firm's wage budget, paying each worker an equal per-head wage (`wageBudget/headcount`) via `withdraw`+`credit`. The **labor each worker delivers is skill-scaled**, not 1: the firm's `Labor` good is increased by the worker's `productivity` (a skill-10 worker contributes 1, the legacy homogeneous case), so two workers at the same firm earn the same wage but supply different amounts of labor. That `productivity` is **firm-specific and computed when the market clears**: a worker delivers `productivityOf(relevantLevel)`, where `relevantLevel` is its (rounded mean) level in *that employer's* skills. Each firm declares the skills its work reads and trains (`Firm.laborSkills()` — necessity→`PLANTS` (subsistence agriculture), enjoyment→`{ARTISTIC, CRAFTING, SOCIAL}`, capital→`CRAFTING`, builder→`CONSTRUCTION`, the strategic export firm→`INTELLECTUAL`; a firm with none falls back to the worker's overall level). So **necessity output tracks a laborer's PLANTS skill, the export firm a noble's INTELLECTUAL, and so on** — each worker is as productive as it is skilled *at that firm's particular work*. Performing the labor also **trains** the worker: each step employed it gains `XP_PER_LABOR` (1, passion-scaled) experience in each of its employer's skills (which then decays daily above level 10 — see `skill/`), so workers **specialize** (better at the work they do often) and each worked skill — hence its output — settles at a train-vs-decay equilibrium. Productivity is read **live** from current skills, computed per worker per firm in `clear()` (the `relevantLevel` helper). A worker's output is then **scaled by the length of the day**: a laborer delivers 100% of its skill-scaled output at `FULL_OUTPUT_DAYLIGHT_HOURS` (8) hours of daylight, proportionally more on longer (summer) days and less on shorter (winter) ones — `productivity * (1 + sensitivity·(daylight/8 − 1))`. The `sensitivity` is itself penalized at high latitudes (`daylightSensitivity()`): full up to 45°, fading linearly to 0 at the poles, so the extreme polar daylight swings don't blow up the economy. **Only laborers** get this daylight scaling (it lives in `addEmployee(Laborer)`); nobles supplying labor to the strategic sector use the `addEmployee(int, Bank, double, SkillTracker)` primitive with a multiplier of 1 and are unaffected. The coupling is calibration-sensitive: deep-winter labor *scarcity* (factor ≪ 1) is the binding stability concern, so the 8h reference — which keeps winter output near 100% — matters. The rest-day calendar's wage-smoothing (see *The liturgical calendar*) absorbs the deep-winter shortfall, so the reference shapes winter output without setting a binding latitude ceiling in the temperate-to-subpolar range the model targets (deep-winter labor scarcity stayed survivable to ~70° when that was swept).
- `CapitalMarket` matches firms buying machines against `CFirm` sellers.
- `WeddingMarket` (`market/WeddingMarket.java`) clears **on the weekly day of rest only** (`DayType.WEEKEND`): each step an unmarried household (one with just its head — see *Marriage* below) posts itself as a seeker, and on a weekend the market marries up to `WeddingConfig.capacity()` (default **4**) of them in **priority order** — the **ruler first, then nobles, then laborers**, and within a rank the richer household first (`Household.weddingPriority()`, ruler 2 / noble 1 / laborer 0). Each seeker takes the **best available peasant of the opposite gender** to its head out of the `Retinue` (highest `overallLevel`, the younger breaking a tie) as a spouse, paying a **non-linear bride-price** (`WeddingConfig.costFor` = `baseCost·(1+skill)^skillExponent`, default `2·(1+skill)²` copper) that flows to the **ruler** (the peasants' patron); a household that cannot afford its best match waits for another weekend (the ruler weds **free**, from its own wards). The match is a deterministic skill/age sort (no RNG), so a colony with no pool or no living ruler — the bare analytical sims — clears it to nothing and is undisturbed. Reported by `WeddingPrinter` (`Weddings.csv`, one event-driven row per couple). When a weekend's wedding demand goes **unmet** — seekers were present but **no wedding happened** (the pool ran out of opposite-gender candidates, or none were affordable) — the colony **recruits one immigrant** into the pool for next time: a young, fresh adult of random gender (`Retinue.addImmigrant`), paid for in **gold that leaves the economy** (an uncredited withdrawal from the ruler's treasury, `Bank.extractExternalFunds`, the documented inverse of `injectExternalFunds`) at `WeddingConfig.immigrantCost()` = `immigrantCostFactor · baseCost` (default `100·2` = 200 copper ≈ 0.17 gold) — but **only if the ruler can afford it** from a positive treasury, else the recruitment waits. This keeps weddings from stalling permanently once the founding reserve drains (the `HanseaticEconomy` example more than doubles its weddings with it), at the cost of a slow gold outflow that the `Retinue.csv` `Imported` column tracks. Parameters live in `WeddingConfig` (override per-run via `SimulationHarness.setWeddingConfig` before `createMarkets`; `capacity 0` disables weddings — and with them the recruitment).

### Dynamic firm provisioning

Every ruler-bearing colony is **founded with one enjoyment and one necessity firm** (`SimulationConfig.DEFAULT` now sets `numEFirms = numNFirms = 1`) and grows its consumer sectors during the run. Each month (gated on the first of the month inside `Ruler.act()`) the ruler runs `reviewSectors()`, which **charters or dissolves** consumer-good firms so their count tracks demand. This is the **default**: `SimulationHarness.createDefaultRuler` installs the provisioning factory automatically (`createFirms` captures a representative firm bank for it), so every standard run grows from the 1E+1N seed; only the bare analytical sims (the sweeps, `SmallOpenEconomy`), which have no ruler, keep a fixed directly-founded firm count.

- **The signal** is read per sector from the `ConsumerGoodMarket` (the unmet-demand *pressure*, above) plus each firm's **smoothed** profit and capacity utilization (`ConsumerGoodFirm.getSmoothedProfit()`/`getSmoothedUtilization()`, trailing 30-day; utilization averaged only over operating days so the rest-day calendar doesn't bias it). The **open** rule is a hybrid: charter when the sector is **supply-constrained** (smoothed utilization ≥ 0.85 with demand still unfilled — the price-bias-free capacity signal, and the one that lets a 1-firm sector *cold-start*) **or** demand pressure surges above its baseline, in either case gated on the sector turning a (smoothed) profit. The **close** rule dissolves the weakest loss-making firm when the sector is **overbuilt** (smoothed utilization < 0.55). Necessity (inelastic) uses lower open thresholds and a firm floor, so food over-provisions.
- **Hysteresis** damps a seasonal charter/dissolve oscillation (winter daylight cuts output → utilization rises → charter; spring recovers → utilization falls → dissolve): a firm younger than `MIN_FIRM_LIFETIME_DAYS` (1y) is never dissolved, and a per-sector `SectorMemory` enforces a 1-year re-entry cooldown between opposite actions.
- **The machinery** is a `FirmFactory` (`agent/firm/FirmFactory.java`) the harness installs on the colony (`Settlement.setFirmFactory`). `charter` builds a firm with the run's standard initial params, **funds its seed capital out of the ruler's treasury** (gold→copper FX fee — so the firm's opening balance has a counterparty, conserving money), grants it to the **least-encumbered noble**, and `claimSlot`s it — which on a live colony queues a builder growth ring (the firm is economically active immediately regardless). `dissolve` detaches the owner, frees the slot (`Settlement.vacateSlot`), marks the firm dissolved and settles its account into equity (`inheritAndClose`). Both **defer** the agent add/remove to end of step via `Settlement.scheduleAddAgent`/`scheduleRemoveAgent`, so the agent set is not mutated mid-`act()`. Reported by `FirmsPrinter` (`Firms.csv`, a living-population aggregation with a firm **`Count`** column — one row per consumer sector each cycle, told apart by a `Good` column, the way `BanksPrinter` consolidates the banks). In the collapsing standard runs the open rule fires steadily (a colony ramps to dozens of firms before the pool drains) while the close rule does not (utilization stays high through each colony's life); dissolution is logically complete but **awaits a stable or contracting colony** to fire at runtime.

### Ennoblement

**The whole aristocracy is raised from the laborers** — no `Noble` is created at founding. `SimulationHarness.ennobleBestLaborer` picks the ablest laborer household (highest head `INTELLECTUAL`, the youngest breaking a tie) and **elevates it to a `Noble`** via an adopt-style `Noble(Member head, …)` constructor: the household keeps its identity (head, members, age, skills) and **re-banks in silver**, its copper balances carried over into a fresh silver account while the old copper account is closed (money conserved). It is deferred to a `Settlement.scheduleEndOfStepAction` so it runs *after* the step's market clearing — only then is the laborer's account safe to move (its buy offers have settled). Two things trigger it:

- **Staffing the export aristocracy (the main path).** A colony with an export sector runs a weekly **top-up** (`SimulationHarness.topUpAristocracy`, a step action registered by `createDefaultRuler`): while it has fewer than `cfg.targetNobles()` (default 5) living nobles and a laborer remains, it ennobles one. So the aristocracy forms gradually over the first ~5 weeks; the **ruler works the strategic firm meanwhile** so the export sector is never unstaffed (see *Agents → Ruler*). `targetNobles` is fixed (does not scale with colony size). The nobles' parameters come from `SimulationHarness.setNobleConfig` (default `NobleConfig.DEFAULT`; `HanseaticEconomy` overrides it to give them a necessity reserve).
- **A firm with no owner (a fallback).** If the dynamic provisioning charters a firm while the colony has no noble at all, it ennobles one to own it (re-checking first, so two charters the same step don't ennoble twice).

Succession is **hereditary**: a noble's heir is drawn fresh (same dynasty), so the founding meritocratic elite perpetuates itself; the top-up only refills if the count drops below target. Covered by `LaborerEnnoblementTest` (the fallback, and that an ennobled noble works the strategic firm) and exercised by every standard run (the top-up).

### Bank is the payment + credit system

`Bank` (`bank/Bank.java`) is an **instance** class (constructed with a `BankConfig`); each agent holds its accounts at a specific `Bank` (`agent.getBank()`, injected through every constructor). A bank holds one `Account` per agent keyed by the agent's ID (which doubles as account number). Every account has a `CHECKING` and `SAVINGS` balance; a **negative savings balance is a loan**. Money only moves between checking accounts. Payments are **agent-routed**: a transfer is `payerBank.withdraw(payerID, amt)` + `payeeBank.credit(payeeID, amt, purpose)`, which is correct even when the two parties bank at different `Bank` instances (there is no combined `pay` — split it). The payment-purpose codes `Bank.PRIIC`/`SECIC`/`OTHER` stay `static` (shared across banks). Each bank also carries a `CurrencyType` (`bank/CurrencyType.java`: `COPPER`/`SILVER`/`GOLD`, set via `BankConfig.currency`, default `COPPER`) — its denomination, so a colony can run banks in different currencies — the default hierarchy banks commoners in copper, nobles in silver, and the ruler in gold. The colony has a **fixed exchange rate** with copper as the base unit (prices are quoted in copper): silver = 100 copper, gold = 1200 copper (historically plausible 15th-century ratios — gold:silver ≈ 12:1, silver:copper ≈ 100:1). Because the rate is fixed and freely convertible the currencies are **fungible**, so all internal accounting stays in copper and the rate is used only to convert non-copper amounts and to **display** balances in a bank's own currency — `CurrencyType.toCopper`/`fromCopper`/`convert` (exposed as `Settlement.convert`), with `BanksPrinter`/`NoblesPrinter`/`PersonsOfInterestPrinter` each carrying a `Currency` column and reporting monetary values in the holder's currency. At the fixed rate alone, currency assignment does not change the dynamics (the copper-denominated figures are unchanged), so changing the *display* currency changes nothing. The dynamics only move once exchange carries **friction**: the `BankConfig.exchangeFeeRate` knob (default 0) makes a non-copper bank a **money-changer** that skims a fee whenever its client's money crosses the copper boundary. Because every price is quoted in copper, a non-copper account converts on *every* payment, so the fee is charged inside `Bank.withdraw`/`credit` exactly when `currency != COPPER` (the payer bears it on a withdrawal, the payee on a credit), retained as the bank's equity/profit; a copper bank is already in the base currency and never charges. The default silver and gold money-changers set it to `0.02` (`DEFAULT_EXCHANGE_FEE_RATE`), so a silver noble loses 2% converting each dividend in and each purchase out, and the gold-banking ruler loses 2% whenever it buys (copper-quoted) enjoyment — both banks profit from the fee they retain. This is the conversion friction a *fixed* rate previously could not provide (a genuinely *floating* rate would be a further, larger step). This tiered scheme is the **default banking system** every settlement uses, provided by `SimulationHarness`: `getCopperBank()` (the zero-profit base-currency bank for commoners — laborers and firms), `getSilverBank()` and `getGoldBank()` (the nobles' and ruler's money-changers, each charging `DEFAULT_EXCHANGE_FEE_RATE` = 2%). Each is created **lazily on first request** and shared thereafter, so a colony carries only the tiers its classes use. Because every standard settlement now has a default ruler **and** raises an export aristocracy by ennoblement (the ennobled nobles re-bank in **silver**), every standard run populates **all three** tiers: copper (commoners), silver (the nobles), gold (the ruler). (The export sector's `createDefaultStrategicSector` eagerly creates the silver bank up front, before the ruler's gold, so the banks stay ordered copper, silver, gold even though no noble exists at founding.) (Request copper before silver/gold so the banks are numbered/ordered copper, silver, gold.) The bare exception that skips the ruler (and the export sector) is `SmallOpenEconomy` (two *copper* banks, agents split across them). Critically, agents read their last-step income from public `Account` fields — `priIC` (primary income: wage/sales), `secIC` (secondary: dividends), `interest` — and **reset these to 0 at the end of their own `act()`**. Loan interest rate responds to the loan/deposit gap; deposit rate redistributes collected interest to creditors. By default the bank is a **zero-profit pass-through** (`depositIR · totalDeposit == loanIR · totalLoan`). Three `BankConfig` knobs let it retain profit into a cumulative `equity` field (charted as the `Equity` column in `BankPrinter`): `spread` keeps a fraction of the interest margin instead of paying it to depositors, `feeRate` charges the payer a per-withdrawal transaction fee, and `exchangeFeeRate` charges the currency-exchange fee described above (only on a non-copper bank). All default to 0. That retained profit is **distributable to a noble owner**: the bank separately tracks `cumulativeProfit` (gross spread + fees ever retained) minus `distributedProfit` paid out, exposed as `getDistributableProfit()`, and `payDividend(amt)` skims that profit out of equity. This is deliberately *only* the profit slice of equity — not the estates-in-transit or injected external funds that also pass through equity — so a noble can draw a bank dividend without disturbing inheritance or the open-colony money buffer. Both fields are passive (and `payDividend` is only called by a noble owner), so a run with no bank-owning noble is unaffected. **Caveat:** the zero-profit pass-through is load-bearing for the model's calibrated stability — empirically `spread ≈ 0.005` is safe and profitable, but `spread ≈ 0.02` already starves the laborer population via a price spike; making the bank meaningfully profitable needs the firm/laborer parameters retuned.

The `equity` field is also the bank's link to **inheritance** and the **open colony**. A dead account holder's estate is folded into equity (`inheritAndClose`) and immediately handed to its successor household (`openInheritedAcct`, which draws the inherited balances back out of equity) — so money stays in circulation, and in a closed run these net to zero. `injectExternalFunds(amount)` adds genuinely new money from outside the colony into equity; it is the entry point for an open colony, where accumulated equity bankrolls new immigrant households (each opened via `openInheritedAcct`, conserving every external dollar as either circulating money or equity below the immigration threshold).

### Agents

`Agent` (`agent/Agent.java`) is the abstract base: unique ID, alive flag, `act()`, and `getGood(name)`. The three **household** types — `Laborer`, `Noble` and `Ruler` — additionally extend `AbstractHousehold` (`agent/AbstractHousehold.java`), the base that consolidates everything a named, aging dynasty shares (each previously carried it verbatim): the identity draw (head / `birthDate` / `foundingDate` / `skill`, skill drawn **before** the head so the given name's rarity tracks it), account opening (fresh endowment vs. equity-funded inheritance/immigration), `ageDays()` / `getAgeYears()` / `getWealth()`, and death settlement — `dieAndSettleEstate()` snapshots the estate and closes the account, `checkOldAgeDeath()` runs the per-step old-age check, `resetIncomeAccumulators(acct)` clears the income fields. A subclass supplies only how it earns and spends in `act()` and which goods it holds. The `agent.com.civstudio.Household` interface they implement carries the **polymorphic contract** the step loop and printers dispatch on — `isWorkforce()` (does this household count toward the colony's labor force, whose loss ends the colony — true only for `Laborer`), `role()` (its label in logs and the persons-of-interest roster), `successor(colony)` (its own heir, see the step loop), and the reporting accessors `getIncome()` / `getWealth()` / `getBank()` — so a new population type slots in without editing `Settlement` or the printers. Two families:

- **Laborers** (`agent/laborer/Laborer.java`) earn wages, decide consumption vs. savings (target-savings model sensitive to the real interest rate), buy Necessity and Enjoyment, eat the `RationSize.FINE` ration (half a unit) of Necessity per step (`LaborerConfig.eatAmt`), and `die()` if they can't eat. They steer their consumption budget toward Necessity in proportion to how far their stock sits below the **target necessity stock** (`colony.getTargetNStock()`, default 26 units ≈ a 26-day buffer), spending the surplus on Enjoyment; that target is a property of the **colony** (a colony-wide environmental constant), not a per-laborer `LaborerConfig` value. **Skill lives on the `Person`, not the household** (`eos.skill`): every person carries a `SkillTracker` of **12 RimWorld-style skills** (`Skill` enum: Construction, Plants, Intellectual, …), each a `SkillRecord` with a `level` in `[0, 20]` and a `Passion` (`NONE`/`MINOR`/`MAJOR`, scaling its learn rate); `SkillRecord.learn(xp)` advances the level along a piecewise-linear XP curve `(0→100, 9→1000, 19→3000)`. The **single scalar** `Household.getSkill()` is now **derived** as the head's `SkillTracker.overallLevel()` (the rounded mean of the 12 levels) and drives **notability and name rarity**. A worker's **labor productivity**, though, is now **firm-specific** — its level in the firm's own skills (not the overall scalar) mapped through the **piecewise** curve `Household.productivityOf` — `max(0.01, (skill/10)²)` up to skill 10, then the steeper `(skill/10)³` above it (skill 0 → 0.01; skill 5 → 0.25, unchanged; skill 10 → 1, the legacy homogeneous unit; skill 20 → 8, so mastery is disproportionately rewarded), computed when the labor market clears (see `LaborMarket`). A person's skills are drawn at birth by `Demography.newSkillTracker(meanSkill)` on the demographic **skill RNG** (a stream separate from both the economic and the mortality streams): each skill's level around a **gender-specific colony mean skill** (`colony.getMeanSkill(gender)` — `meanSkillMale` (default 5) or the lower `meanSkillFemale` (default 2), both colony-start properties; sd 3, clamped) plus a random passion (≈60/30/10 NONE/MINOR/MAJOR), a fresh draw for every head including each successor (so skill is not inherited). Averaging 12 draws around 5 makes a typical male worker's `overallLevel ≈ 5`, so productivity ≈ `productivityOf(5) ≈ 0.25` with low spread — aggregate effective labor sits below the old "everyone = 1" world; founding a colony with a higher mean raises it. Gender is rolled (50/50) only for people generated into the `Retinue`; directly-drawn heads — nobles, the ruler, and pool-less laborers — are `MALE`, so a female reaches the labor force only by promotion. **Skills grow through work and fade through disuse**: each step a worker is employed it gains XP in its employer's skills (see `LaborMarket`), while `Settlement.newDay` ticks every living person's `SkillTracker` once a day, decaying mastery above level 10 (passion-eased; see `skill/`). Labor productivity is read **live** from the worker's current level *in the firm it works* (so a farmer's necessity output tracks its `PLANTS`, a noble's export its `INTELLECTUAL`), and `Household.getSkill()` likewise tracks current skills — each worked skill settling at a training-vs-decay equilibrium, and workers specializing by what they do most. Each laborer is a **household** holding a `List<Person>` of members (head first, exposed via `getMembers()`/`getMemberCount()`; **size 1 — just the head — for now**, the seam for households to grow later). The head is a `Person` (a given name — male for a directly-founded laborer, or the promoted peasant's own, possibly female — plus a unique dynasty surname from `colony.getNames()`, see `name/` below, and its `SkillTracker` and `Gender`); the death log reports `head.fullName()`. The head carries a `birthDate` (the source of truth for its age — the span from `birthDate` to the in-game date) and may also `die()` of old age each step against the `mortality/` life table; founding heads draw a working-age birth date from a Gaussian centered on `colony.getMeanInitAgeYears()`. Distinct from that, the household records a `foundingDate` — the in-game date it came into being, the moment its already-grown head *arrived in the colony and founded it* (a working-age adult, born years earlier, sets up the household now). So `birthDate` is the head's biological birth; `foundingDate` is the household's creation. The shared constructor builds the head's `SkillTracker` **before** naming the head, so the given name is picked at a rarity that tracks the head's overall skill (abler households get rarer, more distinctive given names; the surname is still the plain unique-dynasty draw — see `name/`). Any death folds the estate into the bank's equity; the household is then replaced via `Settlement.addReplacementPolicy` — by **promotion from the peasant pool** in a ruler-bearing colony (a fresh, skill-endowed household, *not* an inheritor — so the dead estate stays in equity and the labor force only holds while the reserve lasts; see *Peasant pool*) or by a **same-dynasty heir** in a bare pool-less colony (inheriting the estate, keeping money and the labor force stable). A separate **equity-funded constructor** creates a fresh new-dynasty household for open-colony immigration; an **adopt-a-`Member` constructor** lets a promoted peasant *become* a laborer head (keeping its name, skills and age). Mortality is a core, always-on feature: every head ages and the demographic draw is taken on a separate salted RNG, so it never perturbs the economic random stream (see `mortality/` below).
- **Firms** (`agent/firm/`): `CFirm` produces Capital (machines); `ConsumerGoodFirm` (abstract) is subclassed by `EFirm` (Enjoyment) and `NFirm` (Necessity), which combine labor + capital via a **Cobb-Douglas production function** `A·L^β·K^(1-β)`. Consumer firms adjust output by marginal profit and decide whether to buy/replace capital based on return-on-capital vs. interest rate and capacity utilization. The **wage budget** follows one of two rules (`FirmConfig`): the legacy cash-flow-gap nudge (`laborShare == 0`), or — since the labor market pays a single wage `totalBudget/N` — a **labor-share** rule (`laborShare > 0`) that budgets a fixed fraction of revenue for wages, so total wage spending scales with the colony and wages aren't outrun by a growing labor pool. `SimulationConfig.laborShare` defaults to 0.5, so the default run uses the labor-share rule (a stable but nominally-inflating closed colony).

- **Nobles** (`agent/noble/Noble.java`) are the **owners** of firms **and banks**: rentier households that live off profit, not wages. A noble is a named household exactly like a laborer — its `Person head` is drawn with `colony.getNames().nextHead()` (unique dynasty surname) — but it never enters the labor market. Each step it *pulls* a dividend from every owned firm (`dividendRate · max(0, firm.getProfit())`) and every owned bank (`dividendRate · bank.getDistributableProfit()`), moving the surplus to the noble through the bank's **secondary-income channel** (`Bank.SECIC` / `Account.secIC`). It then spends a fraction of its liquid wealth back into the consumer-good markets (so dividends circulate) and saves the rest, and **eats the `RationSize.LAVISH` ration** (a full unit/day) from a household necessity larder it keeps stocked (`necessityReserveDays`, default **30** days of its own household; capped so a noble does not hoard food — it buys necessity but never ate it before, which let an uncapped stock grow without bound and starve the workforce out of the market). This is the "option A" ownership design: nobles influence the labor market only **indirectly**, via consumption → firm revenue → the labor-share wage budget. The dividend draws only on public firm getters and the bank's existing transfer primitives, so it touches no firm/bank code, and a run with no nobles is unaffected. Like laborers, nobles **age and die** (a `birthDate` sampled the same way, old-age death against the life table), and a **successor of the same dynasty inherits the estate and the holdings** (its `firms` and `banks` lists, via `Noble.successor` and the colony's built-in household-succession policy — no per-sim rule) — so the aristocracy persists across generations and keeps drawing dividends. A noble's head carries a full `SkillTracker` exactly like any person (drawn around the colony mean, re-rolled each generation); its `overallLevel()` drives the noble's notability and name rarity, while its **`INTELLECTUAL`** skill drives its **strategic-export productivity** — the export firm declares `{INTELLECTUAL}` as its labor skill, so the labor market derives the noble's output from its INTELLECTUAL level (the general per-firm rule, see `LaborMarket`; there is no longer a noble-specific productivity method). Since the noble *trains* `INTELLECTUAL` by working the sector, this is a feedback loop: it grows more productive at exporting the more it exports (up to the train-vs-decay equilibrium). Nobles have no starvation death (they are wealthy rentiers), only old age. (Their consumption is floored at zero wealth, so a noble in debt — e.g. one just ennobled from an indebted laborer — spends nothing rather than posting a negative demand.) Parameters live in `NobleConfig` (`dividendRate`, `consumptionRate`, `necessityShare`, `necessityReserveDays`). **Nobles are no longer created at founding — the whole aristocracy is raised from the laborers by ennoblement** (see *Markets → Ennoblement*): the ruler elevates the ablest laborer (highest `INTELLECTUAL`, youngest on a tie) into a noble, an adopt-style `Noble(Member head, …)` constructor taking over the household's head and members and **re-banking it in silver** (its copper balances carried over, the old account closed — money conserved). These nobles staff the export firm and own the consumer firms the dynamic provisioning charters. A `Noble`'s same-dynasty successor is still drawn fresh (so the founding meritocratic elite becomes hereditary). `NoblesPrinter` writes the `Nobles.csv` time-series (with an `AvgAge` column).

- **Ruler** (`agent/ruler/Ruler.java`) is the settlement's single sovereign: the owner of its **gold** bank and its sole client (the bank has no others). Unlike a `Noble` it draws no dividends — it is a **treasury that indulges**: a named `Household` that ages, dies and is succeeded by a same-dynasty heir (via `Ruler.successor`, the colony's built-in succession policy — which also hands the heir the colony's ruler reference, so a builder bills the heir, not the dead sovereign; and it is tracked as a person of interest), earns no wage or dividends, but it **levies taxes**: each step, before it spends, it skims a fraction of every **public** bank's distributable profit (`SimulationConfig.bankProfitTaxRate`, default **0.05**) and of every living noble's per-step income (`nobleIncomeTaxRate`, default **0.02**) into its treasury — the first slice of the taxation feature (`Ruler.collectTaxes`, run in `act()` before spending; a no-op when both rates are 0, which leaves runs byte-identical). Its **own gold bank is exempt** — that bank is a crown holding, so its retained profit *is* the treasury rather than tax revenue (taxing it would be the crown skimming its own bank into its own account); only the public copper/silver institutions are taxed (see `docs/village-founding.md`). It also runs the colony's **dynamic firm provisioning**: once a month `Ruler.reviewSectors()` charters or dissolves consumer-good firms so their count tracks demand (funding each new firm's seed capital out of the treasury — see *Markets → Dynamic firm provisioning*). And it **works the strategic export firm itself** every step — posting its head (whose `INTELLECTUAL` drives its export output, exactly like a noble's) to the noble-only labor market — so the export sector is never unstaffed while the aristocracy is still being raised by ennoblement; the export wage credits its gold treasury (its reported income stays 0). It then spends a small fraction of its fortune (`consumptionRate`, a constructor param; `SimulationHarness.DEFAULT_RULER_CONSUMPTION_RATE` = 0.0002) on **enjoyment**, posting a buy offer to the (copper-quoted) enjoyment market exactly as a noble does. It also keeps a **lavish table**: it eats the `RationSize.GOURMET` ration (2 units/day) and restocks necessity toward a reserve — also copper-quoted, so that buying likewise fires the gold bank's FX fee (see *Goods → RationSize*). Its opening fortune (`SimulationHarness.DEFAULT_RULER_GOLD` = **50** gold, held in copper internally) is the **founding cash for the whole labor force**: on day 0 it capitalizes each promoted laborer with that laborer's skill-sum endowment (gold→copper, firing the gold bank's FX fee), then taxes and feeds the pool out of the remainder. It never starves (and, funding the pool's relief by overdrawing, may run a bounded deficit over a colony's life). **Every standard settlement has a ruler**, created by `SimulationHarness.createDefaultRuler()` (which also lazily creates the gold bank) — like the export sector it is part of the default setup, skipped only by the sweeps and `SmallOpenEconomy`. Because enjoyment is priced in copper, the ruler's spending converts **gold → copper**, firing the gold bank's `exchangeFeeRate` (its tax revenue, landing as gold, fires the same fee), so the gold bank, too, turns an FX profit. It is deliberately a distinct type rather than a `Noble`: `NobleConfig`'s 5%/step consumption would drain the treasury in months, and the ruler is a tier above the nobles, so this also keeps it out of `Nobles.csv`. The two reporting touchpoints — `PersonsOfInterestPrinter` (a gold-denominated row, income always 0) and the per-POI death log in `Settlement.newDay` — need no `Ruler` case: both read its `role()` ("Ruler") and the `Household` reporting accessors polymorphically. In a sim the ruler is created **before the labor force** (it holds the founding cash and promotes the laborers out of the pool), after the firms and any nobles. The class hierarchy thus mirrors the currency hierarchy: commoners → copper, nobles → silver, ruler → gold. **Three design notes abstract this household hierarchy and its forward direction:** the **rank ladder** — command scope, `HOUSEHOLD`/`HOLDING`/`VILLAGE` promotion & demotion (`docs/rank-ladder.md`, implemented); **`SocialClass`** — the orthogonal standing/standard-of-living axis (ration, currency tier, wedding priority; `docs/social-class.md`, proposed); and the **tech tree** — research (produced by a dedicated ruler-funded `ScienceFirm` from the aristocracy's INTELLECTUAL labor) that boosts per-sector firm productivity, unlocks content, and gates the class/rank progression (`docs/tech-tree.md`, phases 1–3 implemented, from the Renaissance slice of `src/main/resources/techs.json`). A separate, orthogonal axis is **Race** — per-person ancestry: each `Person` would carry a `Race` (default `Race.HUMAN`), varying mortality (`LifeTable`) and names per person and the colony's calendar and tech effect overlay by its **founding race** (`docs/race.md`, proposed; the first non-human race is the Harimari, whose surname table already ships in `src/main/resources/dynasty-harimari.json`).

- **Builder** (`agent/firm/BuilderFirm.java`) is the firm that **physically grows the colony** — the only thing that can enlarge a *live* settlement (see *Settlement size and build slots*). It is a labor-only firm modeled on `StrategicFirm`, and is **staffed exclusively by peasants**: it hires on a dedicated `PeasantLabor` market (`BuilderFirm.LABOR_MARKET`) whose only workers are the `Retinue`'s peasants (the corvée labor of the poor), just as the export firm draws on a noble-only market. It converts that labor into *build-units* (`A·L^β`, capped by the `BuilderConfig.scaffoldCap` **scaffold cap**; daylight scaling rides along for free, as the workers already deliver day-length-scaled labor), which it applies to the `BuildProject`s in the colony's build queue — land clearance, road building, wall raising. The work it delivers is **billed to each task's sponsor at cost**: the firm that requested a slot pays for the land it will occupy, and the **ruler** pays for the ring's roads and walls (a sponsor short on cash overdraws into a small loan via `Bank.withdraw`, so building is never stalled by tight cash; the public-works bill is charged to the colony's *current* ruler, so a succession mid-build bills the heir, not the dead sovereign's closed account). Because the peasants are the ruler's wards (it feeds the pool), **the wage the builder pays for their labor is reimbursed to the ruler**: the pool registers each peasant with the ruler's account as the wage target (`Retinue.act`), so the builder's wage budget flows to the ruler when the `PeasantLabor` market clears, and the peasants train `CONSTRUCTION` by building. The builder stays a near **zero-profit conduit** (like `StrategicFirm`): sponsors fund the wages, which the ruler — the peasants' patron — collects. It registers itself as the colony's sole builder in its constructor (`Settlement.setBuilder`), bids a wage budget only while it has work queued (and lays its workers off when idle), and is reported by `BuilderPrinter`. Parameters live in `BuilderConfig` (the build-curve `A`/`beta`, the scaffold cap, seed/working wage budgets, and per-slot work costs for land/road/wall) — all placeholders pending calibration. A colony with **no** builder simply never grows once live; a colony **with** a builder also needs a `Retinue` (its workforce). **Every pool-bearing colony now gets a default builder** — `createDefaultRetinue` creates one (at the pool's bank) before the pool, so the standard sims all carry it. It is now **active in the common case**: because a colony is founded with just 1E + 1N and the ruler's dynamic firm provisioning charters more consumer firms each month (see *Markets → Dynamic firm provisioning*), the builder continually raises growth rings to seat them — the previously-dormant growth path is now exercised on every standard run.

- **Marriage** (`market/WeddingMarket.java`, *Marriage*). A household head may take a **spouse of the opposite gender** as a second `Member`, wed off the `WeddingMarket` (above) out of the `Retinue`. The spouse adopts the household's surname and is a **full member**: it **works** (every household member is posted to the labor market with its own skills — `LaborMarket.addEmployee(Laborer)` loops the members, and `Noble.postNobleLabor` does the same for the noble export sector — so head and spouse may end up at different firms and train different skills, but all wages credit the one household account; the ruler's spouse does not work, as the ruler sells no labor), it **eats** (every household scales its ration by `getMemberCount()` — a laborer eats `FINE`, a noble `LAVISH`, the ruler `GOURMET` *per member*; the laborer's necessity stock target and minimum-buy scale with the household too), and it **ages and dies** on the same per-member schedule. Member mechanics reuse the existing seam: `AbstractHousehold.addMember` adds the spouse, and `checkOldAgeDeath` already **promotes a surviving member to head** when the head dies — so a widowed spouse simply becomes the new head (no succession/inheritance event, and a female ex-spouse thus enters the workforce as head). If a laborer household cannot feed everyone, the **non-head members starve off first** (`removeNonHeadMember`) and the household lives on; only the head going unfed dissolves it. A household seeks a spouse (`AbstractHousehold.seekSpouseIfSingle`, called from each `act()`) only while it has just its head, so each household holds at most one spouse for now (households grow no further yet).
- **Peasant pool** (`agent/Retinue.java`) is the **universal labor model** for ruler-bearing colonies (see `docs/peasant-pool.md`). It is an `Agent` holding a `List<Member>` — people with skills, a **gender** and an age but no household, no wage, no enjoyment — that the ruler **feeds** and **promotes from**. The pool is seeded with `cfg.retinueSize()` peasants (default 900; each rolling a 50/50 gender on the demographic RNG — males drawn around `meanSkillMale`, females around the lower `meanSkillFemale`, named from the matching given-name table); `SimulationHarness.foundLaborersFromRetinue` then has the **ruler promote the ablest `round(promotionRatio · poolSize)`** (highest `overallLevel`) into laborer households on day 0 — each adopting the peasant as its head (keeping its gender), endowed with the **sum of its 12 skill levels** in copper (funded from the ruler's treasury, gold→copper FX) and **carrying its larder ration out of the pool** (`drawPromotionStock`, so the food is conserved rather than created). The rest stay as the **standing reserve**. The same `promoteToLaborer` is the laborer replacement policy: a dead laborer is replaced by promoting one reserve peasant, until the reserve drains — after which deaths go unreplaced, the shrinking workforce produces less food, and the colony **collapses** (the default scale lasts ~5–7 years; the refill that would make this sustainable is a later phase). **Reserve peasants eat the `RationSize.SIMPLE` relief ration** (`RELIEF_RATION`, 0.25 units/day — the poor-relief tier; a worker eats `FINE` = 0.5, a noble `LAVISH` = 1) drawn from a **per-peasant larder** (`BUFFER_DAYS` = 15 units, the ration a promoted peasant carries out). The pool **refills that larder toward its full per-peasant size, but only up to a price-sensitive money budget** (`RELIEF_BUDGET_PER_PEASANT`, billed to the ruler who borrows): so as food grows scarce the pool buys less and **yields it to the working laborers** (who, unlike the pool, have a guaranteed minimum), rather than the extra mouths out-bidding the workforce into starvation — without this the doubled population collapses the colony in weeks. (The refill target is the *full* larder, not the ration-scaled one — refilling toward a ration-scaled target left the pool idling on its seed larder until it drained and then crashing, instead of buying daily.) The low ration plus the larder buffer let the reserve **persist a couple of years** as a working replacement pool before it drains. Pooled peasants age, die of old age, and have their skills decay daily, so the reserve also drains demographically. The one inflow today is **gold-funded immigration from the `WeddingMarket`** (see *Markets*): a weekend whose wedding demand goes unmet recruits one young immigrant into the pool, paid for in gold that leaves the colony — replenishing the marriageable reserve once founding has drained it. The whole feature is keyed on the pool's **presence** (no config flag): a colony with a pool founds/replaces from it (and gets a default builder — see *Builder*), a bare colony (the sweeps, `SmallOpenEconomy`) founds directly with same-dynasty succession. Reported by `RetinuePrinter` (`Retinue.csv`). The knobs are `retinueSize` and `promotionRatio` (`SimulationConfig`): a high ratio promotes most of the pool (small reserve, the colony lives on its laborers), a low ratio holds a large reserve.

Note: `ConsumerGoodFirm.act()` decides capital purchases from a single **return-on-capital rule** — buy a machine when the rate of return on capital clears the loan interest rate, capacity utilization clears `eUtilThreshold`, and marginal revenue clears the capital price — plus scrapped-machine replacement, floored at keeping at least one machine. (Earlier versions also carried profit-spike / deep-loss capital-purchase **"hacks"** gated on a warm-up period, with `warmupDays`/`capitalNudgeProfitFactor`/`capitalNudgeUtilThreshold`/`avgProfitWin` knobs in `FirmConfig`; these were **removed** — the colony stays stable without them, validated against the test invariants.)

### Goods, printers, utils

- `good/` — `Good` is a mutable quantity holder; `Capital`, `Labor`, `Necessity`, `Enjoyment` are the concrete goods. `RationSize` is an enum of named daily necessity rations (`GOURMET` 2.0, `LAVISH` 1.0, `FINE` 0.5, `SIMPLE` 0.25, in units/person/day): the social hierarchy of the table — the **ruler** eats `GOURMET`, **nobles** `LAVISH`, **workers** (laborers) `FINE`, **pooled peasants** `SIMPLE` (poor relief). Workers eating the modest `FINE` ration (rather than a full unit) is what frees enough food for a colony to also feed its peasant pool.
- `io/printer/` — each `Printer` declares a **typed schema** (`tableName()` + an ordered `ColumnSpec[]` of `columns()`) and writes rows through a `RowSink` (`io/sink/`), so the same printer can emit to a CSV file, a Postgres table, or both. Register printers with `Settlement.addPrinter(...)` (which binds the sink via the colony's `RowSinkFactory` — default `CsvRowSinkFactory`, preserving the old CSV output — and declares the schema / writes the header) and finalize with `Settlement.cleanUpPrinters()` after the run. The **database backend** (Spring Boot, opt-in via `mvn spring-boot:run`) is documented in [`docs/persistence.md`](docs/persistence.md); the core sim has no Spring dependency and writes CSV unless a launcher installs a DB factory. **Cadence is monthly**: the base `Printer.shouldPrint(colony)` gates every row on `getDate().getDayOfMonth() == 1` (within the printer's `[start, end]` step bounds) — there is no per-step `period`/`stepSize` knob. **Number formatting is centralized in `CSVPrintWriter`**: any `Double`/`Float` cell is rendered to two decimals (`%.2f`); ints, dates and already-formatted `String` cells pass through untouched. `BanksPrinter` writes **all of the colony's banks to a single `Banks.csv`** — one row per bank each cycle (in `colony.getBanks()` order), told apart by a `Bank` column (the bank's name, e.g. `Bank 1`) and the `Currency` column, so even two same-currency banks are distinguishable; its interest-rate and inflation columns are raw **percent-valued** numbers (a fraction of `0.0045` is reported as `0.45`, not the old `0.45%` string), so they store cleanly as typed numeric DB columns. The **consumer sectors are consolidated the same way** (one row per sector — enjoyment, necessity — per cycle, told apart by a `Good` column, iterating `colony.getConsumerGoodMarkets()`): `PricesPrinter` (`Prices.csv` — Date/Good/Price), `VolumesPrinter` (`Volumes.csv` — Date/Good/Volume/Supply) and `FirmsPrinter` (`Firms.csv` — the living-firm aggregation above), registered by `addCommonPrinters`; they replace the former per-market `EPrice`/`NPrice`, `EVol`/`NVol` and `EFirms`/`NFirms` files. `LaborersPrinter` aggregates over the **living** laborer population read from `colony.getAgents()` each step (so it tracks replacements and immigrants, not a fixed initial array), averaging over the live count, with `AvgAge` and `Count` columns. `NoblesPrinter` does the same for the noble population (dividends, income, consumption, wealth). `BuilderPrinter` charts the colony's construction — its `Size` and `EffectiveSlots` (which step up as a ring completes), the builder's `Delivered`/`TotalDelivered` build-units and finances, and the `ActiveTasks`/`RemainingWork` of the ring being built (0 when idle). `WeddingPrinter` is **event-driven, not monthly**: it writes one row per couple as weddings happen (the `WeddingMarket` records each step's weddings when it clears, just before printers run), with both parties' name/type/gender/skill/age and the bride-price — registered in `addCommonPrinters` (`Weddings.csv`), so quiet steps and pool-less colonies contribute no rows.
- `io/SimLog.java` — event logging via `java.util.logging`. `SimLog.init()` (called first thing in each `main`) installs a formatter that prefixes every record with the current **in-game date** (`Settlement.getDate()`, e.g. `1452-03-14: WARN …`), writing to **stderr** with per-record flush. Classes that emit events are annotated with Lombok `@Log` and call `log.info(...)` for events or `log.warning(...)` for anomalies (e.g. a price crossing `PRICE_SKYROCKET_FACTOR`×its initial level in `ConsumerGoodMarket`). Logged events include the **colony being founded** (`Settlement.start`) and **dying** (`updateLifecycle`, when its last laborer is gone), and a **notable person** — any household whose head's overall skill is above `Household.NOTABLE_SKILL` (15) — recorded by name at creation (rare at the default `meanSkill` 5; commoner in a higher-skill colony). The colony tracks a roster of **persons of interest** — every `Noble` and every notable household — to which households register themselves at creation (`Settlement.addPersonOfInterest`, exposed via `getPersonsOfInterest()`) and from which they are removed when they die. **Death logging is confined to that roster**: only a person of interest's death is logged, exactly once (in `Settlement.newDay`'s dead-agent sweep), so ordinary laborer deaths produce no log spam. The roster's yearly *listing* is a CSV, not a log: `PersonsOfInterestPrinter` writes one row per living person of interest on the first day of each year (columns `Date, Name, Type, Skill, Age, Income, Wealth, Currency`); it is registered by the strategic-sector sims via `SimulationHarness.addStrategicSectorPrinters`. It is **fully polymorphic** — `Type` is each household's `role()`, and income/wealth/currency come from `Household.getIncome()` / `getWealth()` / `getBank().getCurrency()` — so a new population type appears with no case added. This is for discrete events — bulk time-series still goes through printers to CSV. Note `Settlement.run`'s progress counter prints the in-game date to stdout every January 1st (once per in-game year), so progress and events are on separate streams.
- The in-game date is the canonical display unit: every place a step was shown as an integer now shows `Settlement.getDate()` — the SimLog prefix, the progress counter, and the first column of every printer CSV (header `Date`, ISO `yyyy-MM-dd`). `Settlement` holds the start date (set from `cfg.startDate()` by `SimulationHarness`); `getDate()` returns `startDate + timeStep` days. The integer `timeStep` is still used internally for control flow (`getTimeStep() == 0`, the printers' `[start, end]` step bounds). Printers emit one row on the **first day of each in-game month** (`Printer.shouldPrint` gates on `getDate().getDayOfMonth() == 1`); there is no longer a `period`/`stepSize` cadence knob.
- `util/` — `Rng` (minimal seedable RNG over `java.util.Random`; **`GameSession` mints one per colony** — derived from the session seed and the colony's index — and the colony exposes it via `colony.getRng()`, so colonies in one session have independent, reproducible economic streams; seed the session for reproducible runs) and `Averager` (rolling window average used for inflation and long-term rates).
- `name/` — `NameRegistry` holds the **complete name sets** for a session, loaded from the tiered JSON tables in `src/main/resources` (`dynasty-human.json`, `male-human.json`, `female-human.json`) via Jackson (`jackson-databind`, the one runtime dependency beyond Lombok). Each tier owns a cumulative-weight band `(prev, percent]`, so `NameTable.pick(rng)` draws weighted toward common names. The names are stored **common→rare**, and `total` ≈ 99 acts as a percentile scale, so `NameTable.pickAtRarity(rng, percentile)` biases the draw toward a chosen slice of the rarity range (0 = commonest, 1 = rarest, jittered ±0.125 for variety) — used to give **abler households more distinctive names**. The `GameSession` owns the registry and drives it from a **separate naming RNG** (a salted copy of the seed) so naming is deterministic but never perturbs the economic random stream. Dynasty surnames are drawn **without replacement** so every household *in use* across the session — in any colony — has a unique surname; given names are drawn **with replacement**. The pool is finite, so an extinct dynasty's surname is **recycled**: `releaseDynastyName(surname)` returns it (with its original weight) to the drawable pool when a household dies with no successor, so the uniqueness invariant is over *living* households, not permanent consumption. The registry tracks the `inUse` set for this and guards against releasing a surname that isn't out. `nextHead()` returns a `Person` (male given name + unique dynasty surname); `nextHeadInDynasty(surname)` draws a fresh head keeping an existing surname (a successor household, which therefore does **not** consume or release a surname). The **rarity-aware overloads** `nextHead(nameRarity)` / `nextHeadInDynasty(surname, nameRarity)` draw the given name via `pickAtRarity` instead — the household passes the head's `overallLevel() / Household.MAX_SKILL`, so a household's given-name rarity tracks its head's overall skill (surnames are unaffected, still the unique-dynasty draw). A `Person` returned by the registry is a **name only** (`skills == null`); the household attaches the head's `SkillTracker` via `Person.withSkills(...)`, and `Person` identity (`equals`/`hashCode`) is over the name, not the mutable skills. Households are tagged with the `agent.com.civstudio.Household` interface (identity members `getMembers()` with the head-first convenience `getHead()` and `getMemberCount()`, plus `getSkill()` — now derived from the head's `SkillTracker` — `getAgeYears()`, the `MIN_SKILL`/`MAX_SKILL`/`NOTABLE_SKILL` constants, the static `productivityOf(skill)` curve, and `isNotable()`; the polymorphic members `isWorkforce()`, `role()`, `successor(colony)`; and the reporting accessors `getIncome()`/`getWealth()`/`getBank()`) so the step loop can recycle the surname of one that dies unreplaced, succeed and report each household without switching on its type, and so all three household types share the skill and notability concepts. The shared machinery itself lives one level down in `AbstractHousehold` (see *Agents*), which all three extend.
- `skill/` — the **person skill system** (RimWorld-style, minus genetic aptitudes and traits). `Skill` is an enum of the 12 skills (Construction, Plants, Intellectual, Mining, Shooting, Melee, Social, Animals, Cooking, Medicine, Artistic, Crafting); `Passion` (`NONE`/`MINOR`/`MAJOR`) scales both the learn rate (0.35 / 1.0 / 1.5 via `learnRateFactor()`) and, inversely, the decay rate (1.0 / 0.5 / 0.0 via `decayRateFactor()` — a major-passion skill never fades). A `SkillRecord` holds a `level` (clamped `[0, 20]`) and a `Passion`; `learn(xp)` adds passion-scaled experience and advances the level (possibly several at once, capped at 20) along the XP curve — `xpRequiredForLevelUp(level)` reads the piecewise-linear curve through `(0→100, 9→1000, 19→3000)` via a tiny `linearInterp(points, x)` helper, clamped to the endpoints. `decay()` is the inverse: a skill **above the floor (level 10)** loses `level · 0.05 · passion.decayRateFactor()` XP/day (higher levels fade faster, carrying underflow down a level), but never below the floor — basic competence is permanent, only mastery erodes without practice. A `SkillTracker` (one per `Person`) holds one `SkillRecord` per skill, created up front, with `getSkill`/`learn`/`tick` (tick decays every record), `level(Skill)` (a single skill's level, which the labor market reads for firm-specific productivity), and `overallLevel()` (rounded mean of the 12 levels, read through `Household.getSkill()` for notability and name rarity). Skills are randomized at birth by `Demography.newSkillTracker` and then **grow through work and fade through disuse**: `LaborMarket.clear()` grants 1 XP (passion-scaled) per employed step in each skill the worker's firm trains (`Firm.laborSkills()`), while `Settlement.newDay` calls `SkillTracker.tick()` once a day for every living household member; labor productivity is read live from the worker's level *in its employer's skills* (`LaborMarket.relevantLevel`, not the overall level), so each worked skill settles at an equilibrium where training balances decay (low-passion skills plateau near competence, passionate ones can still master), keeping long-run productivity bounded.
- `mortality/` — `LifeTable` is an age-specific mortality schedule (the bundled `WEST_LEVEL_3` is Coale-Demeny West Level 3, e0 = 25; `nqx` per age group → a per-day death probability). `Demography` (owned by `GameSession`) is the per-session service for the random draws a household makes **at birth**: it draws founding household ages from a Gaussian truncated to working age and decides each step whether a head dies of old age (on its salted **mortality RNG**), and it builds a new person's skills (`newSkillTracker(meanSkill)`: one randomized `SkillRecord` per `Skill` — its level around the gender-specific colony mean skill via `sampleSkill`, clamped to `[0, 20]`, plus a random `Passion`) and rolls a person's gender (`sampleGender`, 50/50) on a **second, separate skill RNG** so these draws don't perturb the mortality stream either. Both RNGs are salted copies of the session seed, separate from the economic one, so all these draws are deterministic per seed yet never perturb the economic random stream. Mortality is a core always-on feature (no on/off toggle).

### Daylight and the solar package

The colony has a **geographic location** (`latitude`/`longitude`); `Settlement` computes and stores the day's solar times (`getDawn`/`getSunrise`/`getSunset`/`getDusk`, `getDaylightHours`) at the top of every `newDay` via a vendored `solar/` sunrise/sunset calculator, and these feed the daylight-scaled labor output (see `LaborMarket` above). An undefined event (e.g. no astronomical twilight at high-latitude midsummer) is stored as `null`/`NaN`. **Full detail: [`docs/solar.md`](docs/solar.md)**. That location now usually *comes from a map* (a separate, orthogonal axis — **geography**): a per-`GameSession` `WorldMap` of 5264 provinces (`com.civstudio.geo`) imported from the Strapi world content (exported by `ProvinceExporter` to the committed `provinces.json` resource, keyed on the game's `province_id`, keeping the core Spring-free; loaded lazily via `GameSession.getWorldMap()`), each `Province` carrying a `latitude` (and a `longitude` derived from its map bounding box), a `plots` count that **hard-caps settlement size** (build slots *are* plots, so a colony's `SlotTable` total ≤ `province.plots` — `Settlement.maxSize`/`hasRoomToExpand`, and the ruler's dynamic provisioning stops chartering at the cap rather than overrunning it), a terrain type (`LAND` settleable, `SEA`/`LAKE` water), a region `raw_key`, and a **neighbor adjacency graph** (the travel/trade network the caravan and village-founding features will route over). A colony is founded into a province via `GameSession.newSettlement(…, Province, …)` / `SimulationHarness.create(cfg, seed, provinceId)`; **the default `HomogeneousEconomy` founds into Dhenijansar** (`province_id` 4411, a small coastal LAND province in the **Rahen** super-region, lat ~23°N — capping it at size 4). Scenarios founded at explicit coordinates (the sweeps, `SmallOpenEconomy`, `HanseaticEconomy`) pass no province (`Settlement.province == null`, `maxSize == slotTable.maxSize()`) and are unchanged. Race-keyed mechanics stay (see the Race axis above); `cultures`/`countries`/`name_groups` are deliberately *not* imported (`docs/geography.md`, Phases 1–3 implemented; the adjacency-graph consumers — province-anchored caravan movement, settling, and settlement-sponsored trade that couples economies — are designed in `docs/caravan-trade.md`, not yet implemented).

### The liturgical calendar

`calendar/` classifies every in-game date as a `DayType` (`WORKDAY`/`WEEKEND`/`HOLIDAY`) on the **15th-century pre-Reformation calendar** (fixed-date feasts from `/feasts.json`; a `LiturgicalCalendar` shared per `GameSession`). **`DayType` gates labor by firm** (`Firm.operatesOn`): necessity/capital/builder run workdays only, enjoyment also on Sunday, the export firm every day — so the rest calendar idles most firms on Sundays/feasts. Three couplings make that survivable and are **calibration-sensitive**: the `LaborMarket` **fair-share cap** (a closed firm reserves but is allocated no workers, so the rest don't flood open firms), `ConsumerGoodFirm` **revenue smoothing** (`REVENUE_SMOOTH_WIN` = 30-day average revenue sizes the wage budget, so a closed day doesn't zero it), and **food's extra capacity** (`NECESSITY_TECH_FACTOR` = 2.0 lifts necessity `A` to cover the ~80 rest days). **Full detail: [`docs/calendar.md`](docs/calendar.md)**.

## Conventions

- Reproducibility hinges on the single seed set at the top of `main`; the same seed yields identical runs.
- Display names & `toString()`: `Agent.getName()` returns a display name that defaults to the class simple name; subclasses override it via `setName(...)` (firms set "Capital Firm" / "<Product> Firm"). Markets and banks (not agents) carry their own default `name` — `good + " Market"` and `"Bank " + colony.nextBankNumber()` (numbered per colony). Laborers, firms, banks and markets override `toString()` with a concise, side-effect-free debug summary (no bank lookups, so they're safe to print on dead agents).
- When adding a new market or good, register the market with `Settlement.addMarket(...)` **before** constructing agents that reference it — agent constructors look markets up by good name (`Settlement.getMarket(name)`).
- Behavioral/model parameters of an agent or bank live in an immutable `*Config` record (`LaborerConfig`, `FirmConfig`, `BankConfig`, `WeddingConfig`) with a `DEFAULT` constant holding the canonical values; the owner stores it as a `final` field and the constructor takes it as a required param. This makes parameters per-instance (enabling heterogeneous populations / multiple banks) and tunable per run from the simulation's `main`. Structural/non-tunable constants — capital lifetime (`CFirm.CAPITAL_LIFE`), market price-band width (`ConsumerGoodMarket.zeta`), the inflation window on `Settlement` (`INFLATION_TIME_WIN`), payment-purpose index constants (`Bank.PRIIC`/`SECIC`/`OTHER`) — stay as `static final` on the relevant class. `Settlement` keeps its constants inline (`INFLATION_TIME_WIN` stays `static final`; the per-run start date, founding-age mean, target necessity stock and the **gendered mean skills** (`meanSkillMale`/`meanSkillFemale`) are constructor args supplied by `GameSession` via `newSettlement`). The target necessity stock and the mean skills live on `Settlement` rather than `LaborerConfig` precisely because they are **colony-start properties**, fixed for the colony's life and uniform across its households (they cannot vary across laborers within one colony). The colony also has a clear **lifecycle**: `start()` (called at the top of `run()`) marks it founded, and it **dies** terminally the step its last laborer is gone — `newDay` detects this (`updateLifecycle`), recording `getDeathDate()` and logging it; query via `isAlive()` / `isDead()`. `run()` **stops early** once the colony is dead (a dead colony produces nothing more, so finishing the horizon only burns compute). Mortality is always on — there is no toggle.
- Run-level parameters live in `SimulationConfig` — an immutable record of scalars plus nested sub-records (`PriceRange`, `FirmInit`, `CFirmInit`, `LaborerInit`) with a `DEFAULT`. Besides the calendar, market price bounds and per-agent-type initial state, it carries the firm **founding counts** `numEFirms`/`numNFirms` (**default 1/1** — the seed the ruler's dynamic provisioning grows from; the bare `SmallOpenEconomy` overrides them with 2/2) and the demographic and macro knobs: `meanInitAgeYears` (founding-age mean), `targetNStock` (the necessity stock every laborer targets, default 26), `meanSkillMale`/`meanSkillFemale` (the centers of a person's per-skill starting-level draw by gender, default 5/2) and the colony's `latitude`/`longitude` (default London 51.5074, −0.1278, driving the daylight-scaled labor output) — all passed into the `Settlement` constructor — `externalInflowPerStep` + `immigrationThreshold` (the open-colony money inflow and the equity level that funds each immigrant; inflow 0 = closed), `laborShare` (the firms' wage-budget rule), `retinueSize` (the peasant pool the colony is seeded with, default 900), `promotionRatio` (the fraction of that pool promoted into laborer households on day 0, default 0.45), and `targetNobles` (the size of the export aristocracy the colony raises by ennoblement, default 5, fixed — see *Markets → Ennoblement*). **There is no `numLaborers` field** — the labor-force size is `round(promotionRatio · retinueSize)` for a pool colony, or the explicit count argument to `SimulationHarness.createLaborers(numLaborers, …)` for the bare pool-less `SmallOpenEconomy`. The harness reads these and wires them: `enableExternalInflow(gatewayBank)` registers the inflow step-action + immigration policy (no-op when inflow is 0), and the consumer firms' `FirmConfig` is seeded with `laborShare` (overridable via `setFirmConfig`). **Each step is one in-game day**: the config holds a `startDate` (`LocalDate`, default 1444-12-11) and `durationYears` (default 25), and derives `numStep()` = days from `startDate` to `endDate()` (9131 for the default 25-year run). `dateAt(step)` maps a step back to its in-game date (step 0 == `startDate`). Each simulation's `main` binds `SimulationConfig cfg = SimulationConfig.DEFAULT` and reads values via accessors (`cfg.eFirm().checking()`); only the per-run differences stay in `main` — the **seed** and the **init logic** (fixed in `HomogeneousEconomy`, or randomized around the config values). The simulations are entry points with no constructor, so the config is a `main`-local bound to `DEFAULT` rather than a constructor param.
- The config records carry Lombok `@Builder(toBuilder = true)`. The hand-written `DEFAULT` constants remain the source of truth (no `@Builder.Default`); the builder exists so a run can derive a variant cheaply — `SimulationConfig.DEFAULT.toBuilder().durationYears(10).build()` — and so multi-field value records can be constructed by name instead of positionally. Builders are additive; nothing in the default run path calls them. Lombok is already a `provided` dependency (`@Getter` is used on agents); `@Builder` on records needs Lombok ≥ 1.18.20 (project is on 1.18.34).
