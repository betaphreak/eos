# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`eos` is an agent-based macroeconomic simulation written in plain Java. It models an economy (closed by default, but optionally **opened** to a stream of external money that bankrolls immigration) of laborers, firms (capital, enjoyment, necessity), a bank (one by default, though the design now allows several), and markets that clear by supply/demand each time step. Laborer households age and die on a real mortality schedule. Running a simulation produces CSV time-series files for analysis (prices, volumes, firm finances, bank rates, laborer state).

The root package is `eos`, under the standard Maven layout at `src/main/java/eos/` (e.g. `src/main/java/eos/agent/firm/CFirm.java` is `eos.agent.firm.CFirm`).

## Build & run

Maven project, Java 21 (`pom.xml`). Tests use JUnit 5 (`mvn test`).

```powershell
mvn clean compile          # compile to target/classes
mvn exec:exec              # run Simulation1 (forks a JVM with -ea)
mvn exec:exec -Dsim.main=eos.simulation.Simulation2   # run another entry point
mvn package                # build the jar in target/
mvn test                   # run the JUnit 5 suite (each Simulation runs full as a smoke test)
```

`exec:exec` runs in a **forked JVM with assertions enabled** (`-ea`) because the code uses `assert` as real invariant checks; the main class is the `sim.main` property (default `eos.simulation.Simulation1`). The entry points are `Simulation1` (homogeneous agents), `Simulation2` (heterogeneous, randomized init), `Simulation3` (two banks, agents split across them ŌĆö a worked example of the multi-bank setup), `Simulation4` (a **scale sweep** that scales the consumer-firm and laborer counts down together at the default firms:laborers ratio to find the smallest economy that stays stable), and `Simulation5` (the **small open economy**: two banks, the minimum stable firm count from `Simulation4` ŌĆö 2 enjoyment + 2 necessity + 1 capital firm ŌĆö 90 laborers, mortality on, and external inflow + immigration growing the population past its starting size; bank A is the gateway and so carries equity rather than being a zero-profit intermediary), and `Simulation6` (the default economy **plus an aristocracy**: two `Noble` owner households split ownership of all firms, and the senior noble also owns the bank ŌĆö which is given a small `spread` so it profits; they draw firm and bank dividends through the secondary-income channel and spend it back into the consumer markets ŌĆö a worked example of the owner/ownership design, including noble mortality + inheritance of holdings). Each consists of a `static run()` that builds and runs the economy and returns a `SimulationHarness`, plus a `main()` that just calls `run()` ŌĆö **except `Simulation4`**, whose `main()` runs the multi-economy sweep (`Simulation4.sweep()`) and prints a stability table + the minimum stable scale to stdout (no CSVs), while its `run()` is just the convention hook returning one default-scale harness. A scale `k` there means `k` enjoyment + `k` necessity firms (+1 capital firm) and `k┬Ę45` laborers; "stable" generalizes the test invariants ŌĆö population sustained, consumer prices finite/positive **and below 1000├Ś their initial ceiling** (a runaway stays a finite double, so mere finiteness isn't enough), bank pools/rates finite, no thrown `-ea` invariant. Each `run()` creates a `GameSession` (which holds the seed and its own `Rng`) and from it an `Economy` (`session.newEconomy(cfg.startDate(), cfg.meanInitAgeYears())`), then hands that economy to the harness. **`SimulationHarness` consolidates the shared construction** (markets, banks, firm/laborer loops, printers, run/cleanup) into a single `Economy` instance; a `Simulation` supplies only what differs ŌĆö which bank each agent uses (`IntFunction<Bank>`), and how each agent's initial state is drawn (`IntToDoubleFunction`). The call order in the harness mirrors the original sims and matters (deferred settlement + reproducible RNG consumption). A run is configured through `SimulationConfig.DEFAULT`.

Tests (`src/test/java`, JUnit 5) treat each simulation as a smoke test: `Simulation*Test` calls `Simulation*.run()` for a full ~10k-step run and asserts invariants on the returned harness (most laborers survive, prices finite/positive, default banks finite & zero-profit; `Simulation3Test` also checks both banks carry loan+deposit pools). `Simulation4Test` runs only the default scale to completion (the full sweep is too heavy for CI) and checks the minimum-selection logic as a pure function over synthetic results. `Simulation5Test` checks its own invariants directly (population *grows* past its starting 90, both banks active, prices finite) rather than via `assertHealthy`, which assumes the large closed-economy population and zero bank equity. `Simulation6Test` checks its invariants directly (population sustained, prices finite) rather than via `assertHealthy`, since its bank is deliberately profitable (non-zero equity); it also asserts the firms *and* the bank paid dividends (`bank.getDistributedProfit() > 0`), the nobles grew past their seed fortunes, and a noble succession occurred. Economy state (including the agent-ID counter and the `Rng`, now owned by the `GameSession`) is per-instance, but the `SimLog` logging handler is still a process-global static, so Surefire keeps `reuseForks=false` ŌĆö a fresh JVM per test class isolates that static state (and keeps `-ea` on).

Output CSVs are written to an `output/` directory created relative to the **current working directory** at runtime (i.e. the project root when launched via Maven; see `io/printer/CSVPrintWriter.java`). Runs are reproducible: a `GameSession` seeds its `Rng` from the seed and shares it with the economy it creates, so the same seed yields identical output.

## Architecture

### The step loop is the spine

`Economy` is an **instance** (no longer a static singleton): a `GameSession` owns the seed and creates economies from it (`economy/GameSession.java`), so several independent economies could coexist in one JVM (tests still use one). Agents, banks and markets each hold a reference to the `Economy` they belong to (injected through their constructors); printers receive it in `print(Economy)`. Agent IDs are per-economy too ŌĆö the `nextAvailableID` counter lives on `Economy` (`agent.getID()` is drawn from `economy.nextAgentID()`), so two economies have fully independent ID/account spaces. Collaborators register with the economy via `economy.addBank(...)` / `addMarket(...)` / `addAgent(...)` / `addPrinter(...)`. `economy.run(steps)` calls `economy.newDay()` repeatedly, and `newDay()` defines the fixed phase order every tick (`economy/Economy.java`):

1. `agent.act()` for every alive agent ŌĆö agents read last step's prices/income and **post buy/sell offers** to markets (they do not transact yet); a death is flagged via `die()`.
2. `bank.act()` for every registered bank ŌĆö recomputes total loan/deposit, sets loan & deposit interest rates, pays/charges interest.
3. Remove the dead and **spawn replacements** ŌĆö one successor household per dead agent, via the registered **replacement policies** (`Economy.addReplacementPolicy`, a list tried in order until one returns non-null: a laborer policy and, in `Simulation6`, a noble policy); then run the registered **step actions** (`Economy.addStepAction`, e.g. injecting external money into a bank's equity) and admit any **immigrants** (`Economy.setImmigrationPolicy`, new households funded out of that equity). All default to no-ops.
4. `market.clear()` for every market ŌĆö **this is where transactions actually settle**.
5. `printer.print()` for every printer.
6. `updateInflation()` (CPI = mean of consumer-good market prices over `INFLATION_TIME_WIN`).

The key mental model: **`act()` posts intentions against last step's state; `clear()` executes them.** Order of construction in a simulation matters because of this deferred settlement (the labor market is cleared once manually before `run()` so firms have workers in step 0).

### Markets clear by finding a price

Each `Market` (subclass of `eos/market/Market.java`) collects offers during the act phase and settles in `clear()`:
- `ConsumerGoodMarket` binary-searches for a price where demand Ōēł supply (bounded to ┬▒`zeta` of last price), then fills all buyers/sellers pro-rata and moves money via `Bank`. Buyers express willingness via a `Demand` strategy object (`market/Demand.java`), not a fixed quantity.
- `LaborMarket` shuffles employers/employees and allocates workers proportionally to each firm's wage budget, paying wages through `Bank.pay(...)`.
- `CapitalMarket` matches firms buying machines against `CFirm` sellers.

### Bank is the payment + credit system

`Bank` (`bank/Bank.java`) is an **instance** class (constructed with a `BankConfig`); each agent holds its accounts at a specific `Bank` (`agent.getBank()`, injected through every constructor). A bank holds one `Account` per agent keyed by the agent's ID (which doubles as account number). Every account has a `CHECKING` and `SAVINGS` balance; a **negative savings balance is a loan**. Money only moves between checking accounts. Payments are **agent-routed**: a transfer is `payerBank.withdraw(payerID, amt)` + `payeeBank.credit(payeeID, amt, purpose)`, which is correct even when the two parties bank at different `Bank` instances (there is no combined `pay` ŌĆö split it). The payment-purpose codes `Bank.PRIIC`/`SECIC`/`OTHER` stay `static` (shared across banks). Critically, agents read their last-step income from public `Account` fields ŌĆö `priIC` (primary income: wage/sales), `secIC` (secondary: dividends), `interest` ŌĆö and **reset these to 0 at the end of their own `act()`**. Loan interest rate responds to the loan/deposit gap; deposit rate redistributes collected interest to creditors. By default the bank is a **zero-profit pass-through** (`depositIR ┬Ę totalDeposit == loanIR ┬Ę totalLoan`). Two `BankConfig` knobs let it retain profit into a cumulative `equity` field (charted as the `Equity` column in `BankPrinter`): `spread` keeps a fraction of the interest margin instead of paying it to depositors, and `feeRate` charges the payer a per-withdrawal transaction fee. Both default to 0 (so default runs are byte-identical). That retained profit is **distributable to a noble owner**: the bank separately tracks `cumulativeProfit` (gross spread + fees ever retained) minus `distributedProfit` paid out, exposed as `getDistributableProfit()`, and `payDividend(amt)` skims that profit out of equity. This is deliberately *only* the profit slice of equity ŌĆö not the estates-in-transit or injected external funds that also pass through equity ŌĆö so a noble can draw a bank dividend without disturbing inheritance or the open-economy money buffer. Both fields are passive (and `payDividend` is only called by a noble owner), so a run with no bank-owning noble is byte-identical. **Caveat:** the zero-profit pass-through is load-bearing for the model's calibrated stability ŌĆö empirically `spread Ōēł 0.005` is safe and profitable, but `spread Ōēł 0.02` already starves the laborer population via a price spike; making the bank meaningfully profitable needs the firm/laborer parameters retuned.

The `equity` field is also the bank's link to **inheritance** and the **open economy**. When mortality is on, a dead account holder's estate is folded into equity (`inheritAndClose`) and immediately handed to its successor household (`openInheritedAcct`, which draws the inherited balances back out of equity) ŌĆö so money stays in circulation, and in a closed run these net to zero. `injectExternalFunds(amount)` adds genuinely new money from outside the economy into equity; it is the entry point for an open economy, where accumulated equity bankrolls new immigrant households (each opened via `openInheritedAcct`, conserving every external dollar as either circulating money or equity below the immigration threshold).

### Agents

`Agent` (`agent/Agent.java`) is the abstract base: unique ID, alive flag, `act()`, and `getGood(name)`. Two families:

- **Laborers** (`agent/laborer/Laborer.java`) earn wages, decide consumption vs. savings (target-savings model sensitive to the real interest rate), buy Necessity and Enjoyment, eat one unit of Necessity per step, and `die()` if they can't eat. Each laborer is a **household** represented by a `Person head` (a male given name + unique dynasty surname from `economy.getNames()`, see `name/` below); the death log reports `head.fullName()`. When **mortality** is enabled (default; per-run toggle on `Economy`), the head carries a `birthDate` (the source of truth for its age ŌĆö the span from `birthDate` to the in-game date) and may also `die()` of old age each step against the `mortality/` life table; founding heads draw a working-age birth date from a Gaussian centered on `economy.getMeanInitAgeYears()`. Any death under mortality folds the estate into the bank's equity, and a **successor household** of the same dynasty inherits it (wired via `Economy.addReplacementPolicy`) so money and the labor force stay stable. A separate **equity-funded constructor** creates a fresh new-dynasty household for open-economy immigration. With mortality off, heads never age and deaths just close the account (the pre-mortality behavior) ŌĆö and the demographic RNG is never touched, so the run is economically byte-identical.
- **Firms** (`agent/firm/`): `CFirm` produces Capital (machines); `ConsumerGoodFirm` (abstract) is subclassed by `EFirm` (Enjoyment) and `NFirm` (Necessity), which combine labor + capital via a **Cobb-Douglas production function** `A┬ĘL^╬▓┬ĘK^(1-╬▓)`. Consumer firms adjust output by marginal profit and decide whether to buy/replace capital based on return-on-capital vs. interest rate and capacity utilization. The **wage budget** follows one of two rules (`FirmConfig`): the legacy cash-flow-gap nudge (`laborShare == 0`), or ŌĆö since the labor market pays a single wage `totalBudget/N` ŌĆö a **labor-share** rule (`laborShare > 0`) that budgets a fixed fraction of revenue for wages, so total wage spending scales with the economy and wages aren't outrun by a growing labor pool. `SimulationConfig.laborShare` defaults to 0.5, so the default run uses the labor-share rule (a stable but nominally-inflating closed economy).

- **Nobles** (`agent/noble/Noble.java`) are the **owners** of firms **and banks**: rentier households that live off profit, not wages. A noble is a named household exactly like a laborer ŌĆö its `Person head` is drawn with `economy.getNames().nextHead()` (unique dynasty surname) ŌĆö but it never enters the labor market. Each step it *pulls* a dividend from every owned firm (`dividendRate ┬Ę max(0, firm.getProfit())`) and every owned bank (`dividendRate ┬Ę bank.getDistributableProfit()`), moving the surplus to the noble through the bank's **secondary-income channel** (`Bank.SECIC` / `Account.secIC`) ŌĆö the dividend pathway that the model had wired but never fired until nobles existed. It then spends a fraction of its liquid wealth back into the consumer-good markets (so dividends circulate) and saves the rest. This is the "option A" ownership design: nobles influence the labor market only **indirectly**, via consumption ŌåÆ firm revenue ŌåÆ the labor-share wage budget. The dividend draws only on public firm getters and the bank's existing transfer primitives, so **no firm/bank code changed** and a run with no nobles is byte-identical to before. Like laborers, nobles **age and die** under mortality (a `birthDate` sampled the same way, old-age death against the life table), and a **successor of the same dynasty inherits the estate and the holdings** (its `firms` and `banks` lists, via a noble `addReplacementPolicy` rule) ŌĆö so the aristocracy persists across generations and keeps drawing dividends. Nobles have no starvation death (they are wealthy rentiers), only old age. Parameters live in `NobleConfig` (`dividendRate`, `consumptionRate`, `necessityShare`). `Simulation6` is the worked example (two nobles split ownership of all firms); `NoblesPrinter` writes the `Nobles.csv` time-series (with an `AvgAge` column).

Note: `ConsumerGoodFirm.act()` contains explicitly-labeled **"hacks"** (capital-purchase nudges gated on `timeStep > 2000`) tuning the model's stability ŌĆö treat that block as model calibration, not incidental logic.

### Goods, printers, utils

- `good/` ŌĆö `Good` is a mutable quantity holder; `Capital`, `Labor`, `Necessity`, `Enjoyment` are the concrete goods.
- `io/printer/` ŌĆö each `Printer` writes one CSV via `CSVPrintWriter`. Register printers with `Economy.addPrinter(...)` (which writes the header row immediately) and finalize with `Economy.cleanUpPrinters()` after the run. `LaborersPrinter` aggregates over the **living** laborer population read from `economy.getAgents()` each step (so it tracks replacements and immigrants, not a fixed initial array), averaging over the live count, with `AvgAge` and `Count` columns. `NoblesPrinter` does the same for the noble population (dividends, income, consumption, wealth).
- `io/SimLog.java` ŌĆö event logging via `java.util.logging`. `SimLog.init()` (called first thing in each `main`) installs a formatter that prefixes every record with the current **in-game date** (`Economy.getDate()`, e.g. `1452-03-14: WARN ŌĆ”`), writing to **stderr** with per-record flush. Classes that emit events are annotated with Lombok `@Log` and call `log.info(...)` for events (e.g. a laborer dying) or `log.warning(...)` for anomalies (e.g. a price crossing `PRICE_SKYROCKET_FACTOR`├Śits initial level in `ConsumerGoodMarket`). This is for discrete events ŌĆö bulk time-series still goes through printers to CSV. Note `Economy.run`'s progress counter prints the in-game date to stdout every January 1st (once per in-game year), so progress and events are on separate streams.
- The in-game date is the canonical display unit: every place a step was shown as an integer now shows `Economy.getDate()` ŌĆö the SimLog prefix, the progress counter, and the first column of every printer CSV (header `Date`, ISO `yyyy-MM-dd`). `Economy` holds the start date (set from `cfg.startDate()` by `SimulationHarness`); `getDate()` returns `startDate + timeStep` days. The integer `timeStep` is still used internally for control flow (`getTimeStep() == 0`, the `> 2000` calibration hacks, printer period gating).
- `util/` ŌĆö `Rng` (minimal seedable RNG over `java.util.Random`; **an instance owned by `GameSession` and shared with its `Economy` via `economy.getRng()`** ŌĆö seed it at construction for reproducible runs) and `Averager` (rolling window average used for inflation and long-term rates).
- `name/` ŌĆö `NameRegistry` holds the **complete name sets** for a session, loaded from the tiered JSON tables in `src/main/resources` (`dynasty-human.json`, `male-human.json`, `female-human.json`) via Jackson (`jackson-databind`, the one runtime dependency beyond Lombok). Each tier owns a cumulative-weight band `(prev, percent]`, so `NameTable.pick(rng)` draws weighted toward common names. The `GameSession` owns the registry and drives it from a **separate naming RNG** (a salted copy of the seed) so naming is deterministic but never perturbs the economic random stream ŌĆö economic output stays byte-identical. Dynasty surnames are drawn **without replacement** (unique per session, shared across the session's economies); given names are drawn **with replacement**. `nextHead()` returns a `Person` (male given name + unique dynasty surname); `nextHeadInDynasty(surname)` draws a fresh head keeping an existing surname (for a successor household).
- `mortality/` ŌĆö `LifeTable` is an age-specific mortality schedule (the bundled `WEST_LEVEL_3` is Coale-Demeny West Level 3, e0 = 25; `nqx` per age group ŌåÆ a per-day death probability). `Demography` (owned by `GameSession`, drawing from its own salted **mortality RNG** so it never perturbs the economic stream) draws founding household ages from a Gaussian truncated to working age and decides each step whether a head dies of old age. Like naming, mortality is deterministic per seed but separate from economic randomness; it is only consumed when mortality is enabled, so a no-mortality run stays economically byte-identical.

## Conventions

- Reproducibility hinges on the single seed set at the top of `main`; the same seed yields identical runs.
- Display names & `toString()`: `Agent.getName()` returns a display name that defaults to the class simple name; subclasses override it via `setName(...)` (firms set "Capital Firm" / "<Product> Firm"). Markets and banks (not agents) carry their own default `name` ŌĆö `good + " Market"` and `"Bank " + economy.nextBankNumber()` (numbered per economy). Laborers, firms, banks and markets override `toString()` with a concise, side-effect-free debug summary (no bank lookups, so they're safe to print on dead agents).
- When adding a new market or good, register the market with `Economy.addMarket(...)` **before** constructing agents that reference it ŌĆö agent constructors look markets up by good name (`Economy.getMarket(name)`).
- Behavioral/model parameters of an agent or bank live in an immutable `*Config` record (`LaborerConfig`, `FirmConfig`, `BankConfig`) with a `DEFAULT` constant holding the canonical values; the owner stores it as a `final` field and the constructor takes it as a required param. This makes parameters per-instance (enabling heterogeneous populations / multiple banks) and tunable per run from the simulation's `main`. Structural/non-tunable constants ŌĆö capital lifetime (`CFirm.CAPITAL_LIFE`), market price-band width (`ConsumerGoodMarket.zeta`), the inflation window on `Economy` (`INFLATION_TIME_WIN`), payment-purpose index constants (`Bank.PRIIC`/`SECIC`/`OTHER`) ŌĆö stay as `static final` on the relevant class. `Economy` keeps its constants inline (`INFLATION_TIME_WIN` stays `static final`; the per-run start date and founding-age mean are constructor args supplied by `GameSession`, and the mortality toggle is a setter the harness calls before constructing agents). Inline calibration magic numbers (e.g. the `ConsumerGoodFirm.act()` "hacks") are deliberately left un-extracted.
- Run-level parameters live in `SimulationConfig` ŌĆö an immutable record of scalars plus nested sub-records (`PriceRange`, `FirmInit`, `CFirmInit`, `LaborerInit`) with a `DEFAULT`. Besides the calendar, population counts, market price bounds and per-agent-type initial state, it carries the demographic and macro knobs: `mortalityEnabled`, `meanInitAgeYears` (founding-age mean, passed into the `Economy` constructor), `externalInflowPerStep` + `immigrationThreshold` (the open-economy money inflow and the equity level that funds each immigrant; inflow 0 = closed), and `laborShare` (the firms' wage-budget rule). The harness reads these and wires them: `enableExternalInflow(gatewayBank)` registers the inflow step-action + immigration policy (no-op when inflow is 0), and the consumer firms' `FirmConfig` is seeded with `laborShare` (overridable via `setFirmConfig`). **Each step is one in-game day**: the config holds a `startDate` (`LocalDate`, default 1444-12-11) and `durationYears` (default 25), and derives `numStep()` = days from `startDate` to `endDate()` (9131 for the default 25-year run). `dateAt(step)` maps a step back to its in-game date (step 0 == `startDate`). Each simulation's `main` binds `SimulationConfig cfg = SimulationConfig.DEFAULT` and reads values via accessors (`cfg.eFirm().checking()`); only the per-run differences stay in `main` ŌĆö the **seed** and the **init logic** (`Simulation1` fixed vs. `Simulation2` randomizing around the config values). The simulations are entry points with no constructor, so the config is a `main`-local bound to `DEFAULT` rather than a constructor param.
- The config records carry Lombok `@Builder(toBuilder = true)`. The hand-written `DEFAULT` constants remain the source of truth (no `@Builder.Default`); the builder exists so a run can derive a variant cheaply ŌĆö `SimulationConfig.DEFAULT.toBuilder().durationYears(10).build()` ŌĆö and so multi-field value records can be constructed by name instead of positionally. Builders are additive; nothing in the default run path calls them. Lombok is already a `provided` dependency (`@Getter` is used on agents); `@Builder` on records needs Lombok Ōēź 1.18.20 (project is on 1.18.34).
