# Design: serving per-province plots from the server (on-demand, sim-paused generation)

**Status:** ✅ **Implemented 2026-07-11** (engine + server + browser; verified live — a deep-zoom
map renders per-plot terrain fetched from `/api/plots/{id}` with no static pack). The **lobby
generation-progress surface** and the **persistent-volume deploy** wiring are the remaining follow-
ups. Companion to `docs/province-plots.md` (the plot generation algorithm) and `docs/client-server.md`.

**Simplification adopted (ship-all).** Of 5264 provinces only ~176 (deep-ocean SEA) lack a shelf
grid, so rather than track *which* provinces have plots, the bundle **ships all provinces** and the
browser fetches `/api/plots/{id}` for whatever it zooms into — a deep-ocean province just yields an
**empty grid** (open sea). This deleted the whole plot-presence apparatus: no committed plot index,
no `plotIndex`/`hasPlots` in the bundle, no `build.mjs` `packPlots`. The `WorldBundleGoldenTest`
fixture was regenerated for the +176 provinces (a regen hook is in the test).

## Today — static / CDN (what we're replacing)

- `web/build.mjs` (`packPlots`) bakes every displayed province's canonical plot grid
  (`map/provinces/<id>.json.gz`, each a standalone gzip member) into **one committed 16 MB
  `web/assets/plots.pack`**, plus a `plotIndex = {id: [byteOffset, len]}`.
- `plots.pack` ships to **Azure Static Web Apps** (a CDN); the browser (`web/js/plots.mjs`)
  **Range-fetches** a province's slice and gunzips it in-page. The server only merges `plotIndex`
  into `GET /api/bundle`; it never serves plot bytes.
- The source grids (`map/provinces/*.json.gz`, ~32 MB, **gitignored**) are the bake input;
  `plots.pack` is committed and must be regenerated by hand (`node web/build.mjs`, which needs the
  grids) and kept in sync with `plotIndex`.

## Target — the server generates on demand, pauses the sim, caches to a volume

Retire the static plot path entirely; **the server becomes the sole plot source.**

### Request flow — `GET /api/plots/{id}`

1. **In-memory LRU** hit → serve.
2. **Volume cache** hit (`PLOT_CACHE_DIR/<id>.json.gz`) → load → LRU → serve.
3. **Miss → generate** (under a per-province lock that coalesces concurrent requests for the same id):
   1. **Pause the sim** — `SessionHost` pauses all `HostedSession`s (refcounted, so batched/overlapping
      generations pause once and resume once). Generation then runs **uncontended**; the lockstep tick
      loops don't jitter. **This is the performance guarantee.**
   2. Generate the field — `ProvincePlotPool.generate(province, registry, raster, rng)` with the
      seed-independent `RngSeed.forProvinceCanonical` stream, serialized to the `ProvincePlotStore`
      gzip-JSON format (the exact bytes the browser already gunzips).
   3. Write to the volume + LRU.
   4. **Resume the sim.**
   5. Serve.
- Response: the province's `.json.gz` bytes verbatim, `Content-Encoding: gzip`,
  `Cache-Control: public, max-age=31536000, immutable`.

### Why this doesn't tank performance

1. **Sim-paused generation** (the key insight) — generation never competes with the tick loops for
   CPU; the sim takes a brief, clean pause instead of sustained jitter, and resumes deterministically.
2. **Volume cache ⇒ generate once per province, ever** — the cache is a persistent mount, so a
   province is generated a single time across all clients *and* restarts. After the world is explored
   (or a one-time warm, below), generation never recurs → the sim is never paused → **zero
   steady-state cost.**
3. **Immutable HTTP caching** — the browser caches each province permanently; a client fetches each
   province at most once.
4. **In-memory LRU** for the hottest provinces; serving is trivial I/O (~3 KB gz → socket).

### Lobby progress surface

Because generation pauses the sim, the lobby *shows* it rather than hiding a stutter: the server
publishes a generation status — the province currently generating and a world warm-up counter
`N / <total> cached` — over SSE (or a small status endpoint the lobby polls). The lobby renders
"Generating terrain…" during a pause and a **warm-up progress bar**; the same bar visualizes the
optional full-world warm.

### Optional: one-time full-world warm

A background/offline job (or a first-deploy step) generates all displayed provinces into the volume
once, so runtime generation — and therefore sim pauses — effectively never happen in production. The
lobby progress bar is its natural readout.

## The retired-CDN caveat (infra requirement)

Retiring the SWA CDN makes the **single-region ACA container the direct origin** for plot fetches.
Immutable browser caching covers per-client repeats, but each client's *first* fetch of a province
hits the container. For a cache shared across clients — the role the SWA CDN played — put a **CDN /
Azure Front Door in front of `/api/plots/*`**. There is none today; the interim is browser-cache-only
(acceptable given small payloads + the volume cache, but a shared CDN or the full-world warm is the
robust answer). Manual-deploy / guest-identity constraints apply (see `spectator-server-deployment`).

## Changes required

- **Engine** — `ProvincePlotStore`: expose *serialize a plot field to gz bytes* and accept a
  **configurable cache dir** (today `save`/`load` are fixed to the source-tree/classpath). Reuse
  `ProvincePlotPool.generate` for the field itself.
- **Server** — `PlotService` (LRU + volume cache + per-province coalescing lock + sim-pause
  coordination) and `PlotController` (`GET /api/plots/{id}`); `SessionHost.pauseAll()/resumeAll()`
  (refcounted). Generation status published for the lobby.
- **Config** — `PLOT_CACHE_DIR` (writable volume, mirrors `ANBENNAR_CACHE_DIR`).
- **Bundle** — replace the `plotIndex` byte-offset map with a `hasPlots` set in `WorldBundle`
  (the browser only needs to know *which* provinces have plots, not offsets).
- **Browser** — `web/js/plots.mjs` fetches `/api/plots/{id}` (no Range); `web/js/core.mjs`
  `PLOT_INDEX` → a `hasPlots` set. Lobby renders the progress surface.
- **Retire** — remove the committed `plots.pack`, `build.mjs`'s `packPlots`, and the `plotIndex`
  from the bundle; stop serving `plots.pack` from SWA (`deploy-web.yml` note).
- **Deploy** — `PLOT_CACHE_DIR` **must be a persistent mount**; on an ephemeral dir the
  once-per-province guarantee is lost on every restart (cold world after each deploy).
