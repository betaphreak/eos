# Design & plan: de-vendoring the Civ4 / C2C files (on-demand fetch)

**Status:** proposed ÔÇö **plan only, not implemented** (2026-07-11). The companion to
[[anbennar-files]] (which de-vendored the Anbennar EU4 mod files): do the same for the Caveman2Cosmos
(Civ4) art + XML now committed under `data/civ4/`.

**The one big difference from Anbennar, up front:** `data/civ4/` is **`.dockerignore`-excluded and
never read by the running server** ÔÇö it is used **only at dev time** (the `geo/export/*` exporters
that bake the committed `map/*.json` / resource JSON, and the node web bakers that bake the committed
`web/assets`). So this is *simpler* than the Anbennar cut: **no runtime fetch, no mounted volume, no
Spring/server config, no `AnbennarSourceConfigurer` analogue, no Docker/CI/deploy changes.** Purely a
dev-time build-input concern.

**Canonical source:** the **Caveman2Cosmos GitHub repo**, fetched with authenticated **`gh api`**
(raw media type) ÔÇö `raw.githubusercontent.com` rate-limits and writes HTML error pages as files. See
the `gh-api-for-civ4-files` note. Pin a commit/tag (a committed `civ4-source.lock`, like Anbennar's).

---

## Goal

Stop vendoring the ~32 MB / 58 files under `data/civ4/` (the C2C art + XML). Fetch what the build
needs, on demand, from the pinned C2C GitHub source, into a gitignored cache ÔÇö with **no
`data/civ4` in the repo**.

## What's under `data/civ4` (32 MB, 58 tracked files)

- **C2C info/schema XML** (top level): `CIV4{Bonus,BonusClass,Feature,Improvement,Route,RouteModel,
  Terrain}Infos.xml`, `CIV4ArtDefines_{Bonus,Terrain}.xml`, `C2C_CIV4TerrainSchema.xml`,
  `Caveman2Cosmos.xsd`, `C2C_Planet_Generator_0_68.py`.
- **Terrain art** (`assets/`, ~24 MB): land/water texture `.dds`, feature `.dds` (trees, palms,
  icepackÔÇª), river/wave `.dds`, and `res/Fonts/GameFont.tga` + `GameFont_120.tga` (the resource-icon
  atlases baked into `web/assets`).
- **Tech XML** (`assets/XML/ÔÇª`): `Technologies/CIV4TechInfos.xml`, `GameText/Tech_CIV4GameText.xml`.
- **CivStudio-derived splits** (see below): `Regular_CIV4BuildingInfos.xml` (4.7 MB),
  `SpecialBuildings_CIV4BuildingInfos.xml`, `zProviders_CIV4BuildingInfos.xml`,
  `Manufactured_CIV4BonusInfos.xml`.

## Who reads it ÔÇö all dev-time

- **Java exporters** (`civstudio-engine/.../geo/export/`): `BonusExporter`, `FeatureExporter`,
  `ImprovementExporter`, `RouteModelExporter`, `TerrainArtExporter`, `HousingExporter`,
  `RecipeExporter`, `ManufacturedBonusExporter` (+ `Civ4Xml` for the schema). Each reads a
  `data/civ4/*.xml` and emits a **committed** resource JSON (`terrains.json`, `features.json`,
  `improvements.json`, `bonuses.json`, `recipes.json`, `map/terrain-art.json`, ÔÇª). Run rarely, by
  hand, to regenerate those resources.
- **Node web bakers**: `web/build.mjs` (resolves terrain `.dds` under `data/civ4/assets` and reads
  `CIV4BonusInfos.xml` + `CIV4ArtDefines_Bonus.xml` for the bonus icons via the GameFont atlas ÔåÆ
  bakes `web/assets`), `tools/build-techs.mjs` (tech icons ÔÇö note the `build-techs.mjs`-art memory),
  and `gamefont.mjs` (reads `res/Fonts/GameFont*.tga`).
- **Nothing at runtime**: `.dockerignore` excludes `data/civ4`, and the committed resource JSON /
  web assets are what ship. Confirmed: no `data/civ4` read in the server runtime path.

## The crucial distinction: verbatim-C2C vs CivStudio-derived

The analogue of Anbennar's runtime-vs-devtime split. Before de-vendoring, each file must be
classified ÔÇö this is **step 1** of any implementation:

- **Verbatim from C2C** (fetchable 1:1 from a path in the C2C repo at the pinned commit): the
  standard `CIV4*Infos.xml`, `CIV4ArtDefines_*.xml`, `C2C_CIV4TerrainSchema.xml`, `Caveman2Cosmos.xsd`,
  the `.dds` terrain art, `GameFont*.tga`, `CIV4TechInfos.xml`, `Tech_CIV4GameText.xml`, the planet
  generator `.py`. These can be dropped and fetched on demand.
- **CivStudio-derived / curated** (NOT a verbatim C2C filename ÔÇö a split or subset produced here):
  the `Regular_` / `SpecialBuildings_` / `zProviders_CIV4BuildingInfos.xml` and
  `Manufactured_CIV4BonusInfos.xml` look like **splits of C2C's monolithic** `CIV4BuildingInfos.xml`
  / `CIV4BonusInfos.xml` (the manufactured-goods / recipe port). These can't be fetched directly ÔÇö
  they must either be **regenerated by a split step** from the verbatim C2C monolith, or **kept
  committed** as CivStudio-authored inputs. Determining this per-file is the first task.

## The design (mirrors `AnbennarFiles`, GitHub-sourced, dev-time only)

- **A `com.civstudio.data.Civ4Files` provider** (engine, plain Java) ÔÇö `get("CIV4BonusInfos.xml")` /
  `get("assets/terrain/textures/land/GrassDetail.dds")` ÔåÆ a local cached `Path`, downloading on a
  miss. Fetches from the C2C GitHub repo at the pinned commit via the GitHub **contents/raw API**
  with a token (`GITHUB_TOKEN` / the `gh` auth), not `raw.githubusercontent.com`. Cache keyed by the
  locked ref (`<cacheDir>/<ref>/<path>`), atomic temp-then-move ÔÇö same shape as `AnbennarFiles`.
  **No server bridge / Spring config** (the server never reads Civ4), so the engine defaults + an
  env token are the whole configuration.
- **A node fetch helper** for `web/build.mjs` / `build-techs.mjs` (they can't call the Java provider):
  a small `web/civ4.mjs` that fetches the same paths at the same ref via `gh api` (or a token'd
  fetch), caching alongside. Shares the ref via the committed lock.
- **Pinning**: a committed `civ4-source.lock` (the C2C commit the exporters/bakers were last run
  against), read by both the Java provider and the node helper. Bumped deliberately when pulling a
  newer C2C ÔÇö the same "dependency lock, not live tail" model as Anbennar, but even lower-stakes here
  since the outputs (resource JSON, web assets) are committed and regenerated only on demand.

## Consumer repointing

- Each exporter's `INPUT`/`ART` constant (`"data/civ4/CIV4*.xml"`) ÔåÆ `Civ4Files.get("CIV4*.xml")`.
- `web/build.mjs`'s `resolveUnder(data/civ4/assets, ÔÇª)` + the `CIV4BonusInfos.xml`/`ArtDefines`
  reads ÔåÆ the node helper; likewise `build-techs.mjs` and `gamefont.mjs`.
- The **derived** building/bonus XMLs: either point their consumers at a regenerated split (a new
  dev step that fetches the C2C monolith and splits it) or keep them committed and leave those reads
  as-is.

## Removal / cutover steps (when implemented)

1. Classify every `data/civ4` file as verbatim-C2C (with its C2C repo path) or CivStudio-derived.
2. Add `Civ4Files` (fetch + cache, reading the committed `civ4-source.lock`) + the node helper.
3. Repoint the exporters + node bakers to fetch verbatim files on demand.
4. Decide the derived split files: regenerate-from-monolith step, or keep committed.
5. `git rm` the verbatim files; add the cache dir to `.gitignore`; the `data/civ4` line in
   `.dockerignore` becomes moot (nothing to exclude) ÔÇö drop it.
6. Verify: the exporters still regenerate byte-identical resource JSON; `node web/build.mjs` +
   `build-techs.mjs` still bake identical `web/assets` (compare against the committed baseline).
7. No Docker/CI/deploy changes, and no history rewrite is *required* ÔÇö but see Deferred for the 24 MB
   of art blobs in history.

## Deferred / open questions

- **Does the C2C GitHub repo carry all the art at matching paths?** The `.dds`/`.tga` need confirming
  against the repo layout (C2C's `Assets/` tree) ÔÇö some may live at different paths or be packed.
- **The GameFont atlases** (`res/Fonts/GameFont*.tga`) ÔÇö the `build-techs.mjs`-art memory notes tech
  art is already missing/fragile on this machine; confirm the atlas is fetchable before relying on it.
- **The derived building/bonus splits** ÔÇö provenance + whether a clean regenerate-from-C2C step
  exists, or they stay committed (they may be the smallest-friction option).
- **History rewrite** ÔÇö like Anbennar, `git rm` won't reclaim the ~24 MB of art blobs already in
  history; a `git filter-repo --path data/civ4` purge + force-push is the optional size win.
- **Scope**: worth it? The win is ~32 MB out of the repo + always-fresh C2C inputs; the cost is a
  GitHub-fetch dependency for anyone regenerating resources/assets. Lower-value than the Anbennar cut
  (which also removed a *runtime* dependency); decide before committing effort.
