# Plan: Anbennar wiki lore import — `WikiExporter` → studio lore collections → in-game

**Status:** **PLAN** (2026-07-21). Scrape the Anbennar Fandom wiki, structure its lore into a `wiki-*`
namespace of Strapi collection types, **correlate** each entity to the canonical EU4-derived types we
already store (`country`/`culture`/`religion`/`region`/`province`), and ride it into the engine over the
existing `WorldSource` bundle so lore is available in-game. Net-new subsystem, but every mechanism it
needs already exists here. Companion to
[`docs/studio-datamodel-rebuild-plan.md`](studio-datamodel-rebuild-plan.md) (the `WorldSource` seam),
[`docs/studio-control-plane-plan.md`](studio-control-plane-plan.md), and the fetch/exporter precedents
in [`docs/civ4-files.md`](civ4-files.md) / [`docs/plot-place-naming.md`](plot-place-naming.md).

## The organising idea

The wiki is not a parallel lore island — **about a third of its typed content already has a canonical
home in our model**, and the value of this plan is in *attaching* lore to those entities rather than
duplicating them. The wiki keys on **display names**; our types key on **stable keys** (`tag`/`key`/
`provinceId`). So the whole design reduces to one move repeated: **a name→key join, materialized as a
Strapi relation** from a new lore row to the canonical row — never a merge into the generated tables.

Three rules follow:

1. **Relate, don't invade.** `country`/`culture`/`religion`/`region`/`province` are generated by
   `studio/scripts/gen-schemas.mjs` from the spec mirroring engine enums; hand-adding a `lore` field
   there fights the generator. Every `wiki-*` type is **hand-written** (guarded out of the generator
   like `province`/`tech`/`recipe`) and owns the relation. In-game, a canonical entity gains its lore by
   following the inverse relation.
2. **Reproducibility is `seed + content-version`, same as all authored content.** The wiki snapshot is
   pinned by a lock file; a re-scrape that changes content bumps `contentVersion`.
3. **Nothing bespoke.** Fetch/cache/lock clones `AnbennarFiles`; export writes `target/generated/` via
   `Exports.outFile`; ingest is `seed.js` `buildSpecs()`; delivery is a `world-bundle.ts` projection.

## Current state (verified 2026-07-21)

### The wiki (MediaWiki 1.43.9, `anbennar.fandom.com`)

- **5,860 pages / 2,511 real articles / 2,029 images**; `writeapi` on, `maxarticlesize` 2 MB.
- Content shape is **structured infobox + sectioned prose + a `[[wikilink]]` graph**:
  ```wikitext
  {{Country
  |Capital=[[Anbenncóst]]  |Region=[[Cannor]]
  |Government=Elective Magocratic Monarchy
  |Established=1221 ([[Grand Summit of Aranthíl]])
  |Preceded by=[[Damerian Republic]], [[Castanor]], [[Jexisian Empire]]
  }}
  '''Anbennar''' … ==History== === 1221-1444 === …
  ```
- **Infobox templates are the schema, already designed by the wiki:** `Country`, `Character`, `Location`,
  `Race`, `Culture`, `Religion`/`Deity`, `Dynasty`, `River`, plus `Infobox event`/`military conflict`.
- **174 categories** form a clean taxonomy — geography (Cannor 544, Bulwar 220, Aelantir 340, Haless 180,
  Sarhal 135, Escann 154, Serpentspine 114…), races (Human 393, Ruinborn 298, Elven 285, Dwarven 154,
  Orcish 67, Goblin 50, Halfling 46, Fey 42…), and concepts (Countries 532, People 432, Religion 134,
  Culture 133, Companies 105, History 226, Magic 71, Deity 77, Dynasty 36, Wars and Conflicts 73…).
- **Redirects** (`#REDIRECT [[…]]`) and **Stub/Empty** pages exist and must be filtered/resolved.

### Two acquisition paths (we use both)

- **Bulk XML dump** — `Special:Statistics` → *Database download*, a single `.7z` XML of the whole corpus,
  CC BY-SA-compliant, atomic. **Excludes images.** This is the seed/source-of-record.
- **MediaWiki API** (`/api.php`) — `prop=imageinfo`/`list=allimages` for the 2,029 images, and
  incremental refresh. Batch ≤50 titles, honour `continue`/`maxlag`, descriptive `User-Agent`, ~1 req/s.

### The seams this rides (all present)

- **Fetch/cache/lock:** `com.civstudio.data.AnbennarFiles` — double-checked per-`(ref,path)`
  `ConcurrentHashMap` lock, atomic `.part`→`ATOMIC_MOVE` writes, JDK `HttpClient`, 429/`Retry-After`
  backoff, `ref` pinned from a classpath `*.lock`. **The template for `WikiFiles`.**
- **Exporter pattern:** `public final class *Exporter` dev-tool with `static void main`, input via
  `AnbennarFiles.get`, output Jackson 3 (`tools.jackson`) JSON via `Exports.outFile(...)` into
  `target/generated/…`. `generated/` is no longer committed — exporters feed studio.
- **Name→key precedent:** `CountryExporter` already parses EU4 loc into a **`tag → name`** map and uses a
  hand-maintained override for the ambiguous tail — exactly the join + curation this needs.
- **Ingest → deliver:** `studio/scripts/seed.js` `buildSpecs()` descriptors (two-phase: scalars by
  natural key, then relink relations) → `world-bundle.ts` projection under a `resources["/wiki/…"]` key →
  `StrapiWorldSource` (`BundleWorldSource`) → engine, gated by `contentVersion`.

## The design

### New collection types (`wiki-*`, all hand-written)

House conventions throughout: `"draftAndPublish": false`; unique **non-localized** natural `key`
(normalized wiki title, with `wikiPageId` as the stable tiebreak); human display text **i18n-localized**.

- **`wiki-article`** — base + fallback + shared shape:
  `key`, `title` (i18n), `summary` (i18n, lead paragraph), `body` (i18n, **cleaned markdown**),
  `wikiPageId`, `wikiUrl`, `revisionId` (reproducibility stamp), `isStub`,
  `infobox` (JSON — raw parsed params, so nothing is lost even when a field isn't promoted to a column),
  `categories` (relation → `wiki-category`), `links` (self many-to-many — the `[[wikilink]]` graph).
- **Typed subtypes** = article fields + a correlation relation + a few infobox-derived columns:
  - `wiki-country` → `country`; +`capital`, `government`, `established`
  - `wiki-character` → `country`/`culture`/`religion`/`wiki-dynasty`; +`birth`, `death`, `titles`
  - `wiki-location` → `province`; +`region`, `settlementType`
  - `wiki-culture` → `culture`; `wiki-religion` → `religion`; `wiki-deity` → `wiki-religion`
  - `wiki-race` — `raceKey` (engine enum string, no row to relate to); +`homeland`
  - `wiki-dynasty`, `wiki-organisation`, `wiki-event`, `wiki-river`
- **`wiki-category`** — the 174 categories as a taxonomy: `key`, `name` (i18n), self-relation
  (parent/child), so geography/race/concept facets are queryable in-game.

### `WikiExporter` (engine, `com.civstudio.wiki.export`)

- **`WikiFiles`** — clone of `AnbennarFiles`: `.wiki-cache/<revision>`, pinned by
  `civstudio-engine/src/main/resources/wiki-source.lock` (the dump's snapshot date/revision). Atomic
  writes, per-key lock, `Retry-After` backoff. Offline + reproducible bakes (the locked-junction model).
- **`WikitextParser`** — the one real engineering chunk:
  1. brace-matched `{{Infobox|param=…}}` extraction → typed columns + the `infobox` JSON blob;
  2. `[[link]]` harvest → relation graph + **redirect resolution** to canonical titles;
  3. wikitext → **cleaned markdown** (strip templates/`'''`, resolve links to entity refs, keep
     `==sections==`) for `body`, plus lead-paragraph `summary`.
- **`WikiExporter`** (`static void main`) — decompress the `.7z` dump, stream the MediaWiki XML once
  (~2,511 articles), drop redirects/stubs, classify by template+category, **correlate** by loading
  `countries.json`/`cultures.json`/`religions.json`/`provinces.json` and normalized-name-joining to
  stamp the correlation key, then emit one Jackson JSON per type into
  `civstudio-engine/target/generated/wiki/*.json`. Images fetched from the API into the wiki cache.
- **Unmatched report + overrides** — the exporter writes an ambiguous/unmatched list and reads an
  optional hand-maintained `wiki-overrides.json` (title→key), mirroring `CountryExporter`'s override tail.

### End-to-end pipeline

```
Special:Statistics .7z dump ─► WikiExporter (parse + correlate) ─► target/generated/wiki/*.json
        + API (images)                                                     │
studio: wiki-* schema.json ◄── seed.js buildSpecs() descriptors ──► Strapi Postgres
                                                                           │
world-bundle.ts projections ─► GET /api/world-bundle ─► StrapiWorldSource ─► engine / in-game
```

Shipping = the same three studio touch-points as any dataset: **(a)** the `wiki-*` `schema.json` files,
**(b)** `buildSpecs()` entries in `seed.js` (scalars by key, then relink relations to
`country`/`culture`/… via `R.one(...)`), **(c)** projections in `world-bundle.ts` under `resources["/wiki/…"]`.
Bump `contentVersion` so `StrapiWorldSource` re-fetches.

## Phases

- **P0 — vertical slice (de-risk). ✅ SHIPPED** (`2baea6e`). `WikiFiles` (API fetch + `.wiki-cache`,
  lock/atomic/backoff), `WikitextParser` (brace-matched infobox + `[[link]]` graph + wikitext→cleaned
  markdown), `WikiExporter` over a stratified `Category:Countries` sample → `wiki-country.json` +
  `_unmatched` report. *Result:* **90%** auto-match (27/30; misses are genuinely tag-less lore), 0
  leftover markup, no encoding bug (`�` was console-only). No dump/Strapi/bundle wiring yet.
- **P1 — full corpus, base type. ✅ exporter + studio seed SHIPPED.** `WikiArticleExporter` walks the whole
  mainspace (2509 articles, 99.96% clean) → gzipped `wiki-article.json.gz` (11 MB→3.5 MB); the
  `wiki-article` Strapi type + a **loose seeder** (`seedWikiArticles` in `seed.js`, reads the gz from disk,
  skips loudly if absent) load it. **Wiki lore does NOT ride the engine `world-bundle`** — the Java sim
  never reads lore; it's for the web viewer + chatbot, which read Strapi directly. So no `world-bundle.ts`
  change and no `contentVersion` coupling. The gz is a **committed exporter output** — written straight to
  `civstudio-engine/src/main/resources/wiki/` (like the GeoNames subset), so a clean checkout / CI seeds it
  without re-scraping. *Remaining:* swap the API source for the `.7z` dump.
- **P2 — classification + correlation. ✅ exporter SHIPPED.** Rather than 11 parallel collections, each
  article is classified into an **`entityType`** (13 types, from the infobox template normalized —
  `Country`/`Infobox country`/`Infobox_country` all fold — falling back to categories for the
  template-less religions/regions) and **correlated** to a canonical entity by the shared `WikiNames`
  name→key join: COUNTRY 75%, REGION 53%, RELIGION 44%, CULTURE 37%, LOCATION 30% (`entityRef` +
  `entityKey` per row). The unmatched are **genuinely lore-only** — 0 of them have an exact fixture name
  (verified) — so the rates reflect that the wiki has far more lore entities than the game has tags/keys.
  `_unmatched-correlation.json` report emitted for the curated-override tail. **Studio SHIPPED:**
  `entityType`/`entityRef`/`entityKey` + one-way m2o **relations** (`country`/`culture`/`religion`/
  `province`/`region`/`superRegion`) on the `wiki-article` type, seeded and verified (2509 typed, 553
  relation links; `Empire of Anbennar → Z01`). Fixing these surfaced a **pre-existing core `seed.js` bug**:
  it set relations with a bare `documentId`, which this Strapi rejects for i18n→i18n relations, so every
  relation to an i18n type silently failed (`province linked 27/5268`). Fixed with a `localizeRels` helper
  (locale-qualified `{set:[{documentId, locale}]}` for i18n targets) — seed "row errors" **16,262 → 943**
  (the rest are genuine missing targets). *Remaining:* curated `wiki-overrides.json` for the tail.
- **P3 — images.** API image fetch → Strapi media (Azure Blob, per `studio/CLAUDE.md`) → infobox art
  in-game.
- **P4 — in-game surfacing.** Web/engine reads lore off the canonical entity via the inverse relation
  (province/country/culture panels gain a lore tab). Separate follow-up doc.
- **P5 — lore chatbot (RAG) on anbennar.civstudio.com.** A grounded Q&A assistant answering from the
  wiki lore **and** the structured Strapi facts. See *Forward* below — the reason P1/P2 store sectioned
  markdown with per-entity provenance is to make this cheap when we get here.

## Data source & licence

Anbennar Fandom wiki (`anbennar.fandom.com`), pinned by `wiki-source.lock` (the `Special:Statistics`
dump snapshot) — a *dev-only* dependency, needed only to re-run the exporter. Fandom user content is
licensed **CC BY-SA** (the same share-alike family as our other imports). Credited in the README, and
the ingested lore is a derived work under the same licence — **attribution must ship with any in-game
display of it** (a source/credit line on the lore panel, and the entry's `wikiUrl` retained on every
`wiki-*` row). Images carry their own per-file licences on the wiki; P3 must retain each file's source
URL/attribution rather than assuming the corpus licence. This mirrors how GeoNames (**CC BY 4.0**,
[`docs/plot-place-naming.md`](plot-place-naming.md)) and the C2C/Anbennar imports are credited.

## Forward: the lore chatbot (RAG) — what to get right now

The end goal is a chatbot on **anbennar.civstudio.com** that answers lore questions from this data. We
are **not building it in this plan**, but a few near-free choices in P1/P2 decide whether it's easy or a
rewrite later. The retrieval substrate a RAG assistant needs is *chunked text + embeddings + provenance*,
and this import already produces most of it:

- **Chunking is free from the structure we already keep.** The cleaned markdown preserves `==section==`
  headers, so a chunker splits on them — each `History`/`Culture`/`Government` section is a natural
  passage. *Implication for P1:* keep the section headings in `body` (we do); don't flatten to one blob.
- **Provenance is already required by the licence.** Every `wiki-*` row keeps `wikiUrl` + the entity
  relation, so a retrieved chunk can **cite its source** — which is exactly the CC BY-SA attribution
  obligation. RAG citations and the licence credit are the same field; get it once.
- **Ground on structured facts too, not just prose.** The correlation relations (`wiki-country → country`,
  `wiki-location → province`, …) let the bot answer *"who owns Anbenncóst / what culture is Escann"* from
  the **canonical Strapi rows** and blend that with the prose lore. Retrieval should span both the `wiki-*`
  lore and the EU4-derived types — the join we're building here is what makes that possible.
- **Where it will live (all already in the stack).** Postgres is the store, so **pgvector** is the natural
  embedding home — a `wiki-chunk` type (or a vector column) of `(text, embedding, sourceEntity, wikiUrl)`,
  populated in a later phase from the same seeded lore. The serving side has a home too: `civstudio-server`
  already runs **Spring AI** (the MCP server, `docs/mcp-server.md`) — Spring AI's vector-store + RAG/advisor
  abstractions make the chatbot a sibling of the existing tool surface, not a new stack. Reproducibility
  still rides `contentVersion` (embeddings are rebuilt on a lore-version bump).

*Net:* nothing extra to build now; just don't destroy section structure or drop `wikiUrl`, and prefer
pgvector-in-Postgres + Spring AI when P5 lands. The full architecture (pgvector schema + HNSW index,
hybrid retrieval, TEI embeddings, the Spring AI + Claude citation loop) is spec'd in
[`docs/lore-chatbot-plan.md`](lore-chatbot-plan.md).
